## Per-item verification

### 1. toolkit `repair_card` Ms1 arm

- **Citation:** "was ~:784-818 pre-Cycle-E" → **DRIFTED-by-(+328)**. At v0.80.0 the `Ms1` arm of `repair_card` is `crates/mnemonic-toolkit/src/repair.rs:1112-1150` (function itself starts `:1081`).
- **How it corrects ms1 today:** loops per supplied chunk (ms1 is single-chunk by spec, but the function stays chunk-count-agnostic), pre-gates via `parse_chunk` (precise `HrpMismatch`/`ReservedInvalidLength`/long-code rejection at `:1125-1132`), then delegates to `repair_via_ms_codec` (`:1183-1219`), which calls `ms_codec::decode_with_correction(chunk)` and translates its `Vec<CorrectionDetail>` into the toolkit's `RepairDetail.corrected_positions` via `apply_ms_corrections` (`:1226-1246`).
- **Correction-COUNT signal:** YES, already available. `ms_codec::decode_with_correction` returns `(Tag, Payload, Vec<CorrectionDetail>)` (verified in ms-codec, see item 3) — an empty vec means "no correction," non-empty means "≥1 correction," and `repair_via_ms_codec` already branches on `corrections.is_empty()` (`:1190`). The toolkit does NOT need any new signal from ms-codec.
- **Current `set_verify`:** hardcoded `SetVerify::Blessed` at `:1148`, with an explicit comment (`:1145-1147`) stating *"ms1 delegates entirely to ms_codec::decode_with_correction, which only ever returns `Ok` on full decode success — no behavior change from Cycle E."* This is precisely the line that needs to flip.
- **Can it reuse the shipped `SetVerify{Blessed,Unverified}`?** YES, zero new type needed. `SetVerify::Unverified { reason: String }` (`:459-465`) already exists and is a free-text field — the Ms1 arm just needs to return `Unverified { reason: <ms1-specific text> }` whenever `!repairs.is_empty()`, mirroring the shape (not the semantics) of the Mk1 arm's `verify_mk1_set` (`:967-1080`) result. No `chunk_set_id`/reassembly concept applies to ms1 (single string, no grouping) — the ms1 rule is simpler: touched ⇒ always Unverified (never Blessed, never a distinct Reject-vs-Candidate split the way mk1 has, since there is no second oracle to distinguish "complete-and-consistent-but-wrong" from "just incomplete").

- **`mnemonic repair` caller mapping** — `crates/mnemonic-toolkit/src/cmd/repair.rs::run` (`:101-249`). ### 4. ms-cli `ms repair`

- **Citation:** "`crates/ms-cli/src/cmd/repair.rs`, was ~:89" → **ACCURATE, exact.** Line 89 is exactly `let (_tag, _payload, corrections) = ms_codec::decode_with_correction(&original)?;`.
- **Current exit-code logic:** binary, no tri-state at all. `:123-124`: `let any_correction = reports.iter().any(|r| !r.corrected_positions.is_empty()); Ok(if any_correction { 5 } else { 0 })`. Uncorrectable input already propagates via `?` at `:89` → `ms_codec::Error::TooManyErrors` → `CliError::FormatViolation` → exit 2 (module doc `:16-22` confirms this mapping is already wired). So ms-cli's exit-2 (Reject/uncorrectable) path is ALREADY correct under Option B; only the "any correction ⇒ exit 5" line needs to demote.
- **Mirror-fix scope, compared to `mk repair`'s Cycle E fix:** `mnemonic-key/crates/mk-cli/src/cmd/repair.rs` added a `classify_mk1_set`/`GroupVerdict`{Bless,Reject,Candidate}/`SetVerify`{Blessed,Unverified} classifier (`:168-380`ish) that re-groups corrected chunks by `chunk_set_id` and reassembles via `mk_codec::decode`. **ms1 has no analogous grouping concept** (single string, no `chunk_set_id`, no cross-chunk hash) — so the ms-cli mirror-fix is structurally SIMPLER than mk-cli's: there is no classifier to write, just a straight demotion of the existing binary branch to a 3-way `{0, 4-or-whatever-code, 2}` (see the open design question below).
- **OPEN DESIGN QUESTION (flag, do not resolve in recon):** mk-cli's Cycle E fix deliberately KEPT exit 5 for its `Unverified`/Candidate state (with an added stderr advisory), explicitly asymmetric from `mnemonic repair`'s exit 4 for the identical state — see `mnemonic-key/crates/mk-cli/src/cmd/repair.rs:141-148` (`SetVerify::Unverified => { unverified_advisory = Some("correction UNVERIFIED — reassemble the full card ...") }`) and `:165` (`Ok(if any_correction { 5 } else { 0 })` — unconditionally 5 whenever any correction fired, Blessed or Unverified alike). The toolkit's own manual (`docs/manual/src/40-cli-reference/44-mk-cli.md:234`) documents this precedent explicitly. **This directly conflicts with this cycle's stated principle "Exit 5 becomes unreachable for ms1 EXCEPT via a real oracle"**, because `ms repair` (standalone) has **no oracle at all** (unlike `mk repair`, which at least has the chunk-set reassembly check as a de-facto partial oracle even in the fully-standalone case). If `ms repair` mirrors `mk-cli`'s asymmetry (stay-at-5-plus-advisory), then exit 5 would remain reachable for ms1 with literally zero verification — precisely the funds-unsafe behavior Option B exists to close. If instead `ms repair` breaks CLI parity and adopts exit 4 (mirroring `mnemonic repair` instead of `mk repair`), the 4-manual-chapter "exit-5 REPAIR_APPLIED is consistent across all four CLIs" sentence (item 6) becomes doubly false (it is already arguably stale post-Cycle-E for the mk1-Candidate case, since `mnemonic repair`=4 vs `mk repair`=5 for the same underlying state). **This is a genuine architectural fork the brainstorm must resolve explicitly — recon flags it, does not decide it.**

### 5. verify-bundle auto-oracle — integration point + hard parts

- **Where mk1 is held during intake:** `SuppliedCards<'a> { pub ms1: &'a [String], pub mk1: &'a [String], pub md1: &'a [String] }` (`verify_bundle.rs:1990-1994`) is constructed once per top-level dispatch (`run_full`, `run_watch_only`, `run_multisig`, `descriptor_mode_verify_run`) and threaded into the shared check-emitters `emit_verify_checks` (single-sig, `:2007-2160`+) and `emit_multisig_checks` (multisig). Raw mk1 strings are available in scope wherever ms1 auto-repair fires; the DECODED `mk_codec::KeyCard` is available at different points depending on path:
  - **Single-sig (`emit_verify_checks`):** ms1 auto-repair fires at `:2079` (inside the `ms_codec::decode(supplied_ms1)` `Err` arm, `:2076-2102`), but mk1 is not decoded until `:2107-2108` (`mk_codec::decode(&mk1_strs)`) — i.e. AFTER the ms1 block, in current code order. **Integration requires restructuring:** either decode mk1 speculatively earlier (cheap, harmless to do twice) or move the ms1 block after the mk1 decode — either way the emitted `checks` Vec push ORDER (which is schema-relevant, ms1_decode/ms1_entropy_match before mk1_decode/...) must stay unchanged even though the underlying decode COMPUTATION order changes. This is a real but modest restructuring, not a redesign.
  - **Multisig (`emit_multisig_checks`):** notably EASIER — `card_for_cosigner: Vec<Result<&mk_codec::KeyCard, MappingFailure>>` (`:2377-2444`) is fully computed BEFORE the per-cosigner ms1 loop (`:2448+`, ms1 auto-repair at `:2503`). So for the multisig path, the corresponding cosigner's already-decoded mk1 `KeyCard` (if `Ok`) is available in scope with NO reordering needed.
- **Can the corrected ms1 seed become an xpub/fingerprint for comparison? Which crate/fn?** YES, and there is DIRECT PRIOR ART in this exact file: `emit_full_path_parent_fingerprint_check` (`verify_bundle.rs:3239-3429`+) already does this precise operation for a *different* purpose (v0.25.0 §2.D depth-≥2 parent-fingerprint cross-check): it decodes a supplied ms1 (`ms_codec::decode`, handling both `Payload::Entr` and `Payload::Mnem` — wire-language-wins logic at `:3356-3393`), builds a BIP-39 `Mnemonic` + seed (`bip39::Mnemonic::from_entropy_in` + `.to_seed(passphrase)`, `:3398-3402`), derives `Xpriv::new_master` (`:3403-3406`), then derives along a **derivation path** and compares fingerprints/xpubs. **The derivation primitive used is `bitcoin::bip32::{Xpriv, Xpub}` directly (already a toolkit dependency) — NOT `mk_codec`.** `mk_codec` is a wire-format en/decoder, not a derivation engine.
- **Derivation-path/account assumption — SIMPLER than expected.** `emit_full_path_parent_fingerprint_check`'s existing code derives the path from **md1's `OriginPath`** (`:3265-3275`, `:3412-3424`), because ITS purpose is a parent-fingerprint check that needs md1's declared path. But `mk_codec::KeyCard` (the decoded mk1 struct, `mnemonic-key/crates/mk-codec/src/key_card.rs:24-56`) **already carries its own `origin_path: DerivationPath` field** (verbatim, reconstructed at decode time) — independent of md1. So the auto-oracle does NOT need md1 at all, and does NOT need to guess an "account"/default path: it can derive `master.derive_priv(&secp, &mk_card.origin_path)` directly from the corrected-ms1-derived master xpriv, and compare the FULL resulting `Xpub` (depth, chain code, public key) against `mk_card.xpub` — a strictly stronger oracle than a bare fingerprint compare. `KeyCard.origin_fingerprint: Option<Fingerprint>` is also available as a cheaper first-pass check but is optional (privacy-preserving mode may omit it, per `key_card.rs:36-40`), so the full-xpub compare should be the primary oracle, not the fingerprint.
- **Hard parts to flag for the brainstorm (not resolved here):**
  1. **Passphrase.** ms1 encodes ONLY entropy, never the BIP-39 passphrase (by design, constellation-wide). The oracle derivation needs `args.passphrase` threaded in exactly as `emit_full_path_parent_fingerprint_check` already does — fine, but must not be forgotten, and the CLI has no way to know if the "corrected" seed even used the right passphrase (a wrong passphrase silently derives a DIFFERENT, still-plausible-looking xpub — the oracle can only ever prove match/mismatch against the passphrase actually supplied, not detect "right seed wrong passphrase" as a distinct outcome).
  2. **What if mk1 ALSO needs repair simultaneously?** The oracle's precondition is "mk1 decodes/derives cleanly to serve as ground truth." If mk1 is ALSO corrupted in the same invocation, its own repair (already gated by the SAME generic mechanism, item 2) may itself only be a Candidate — chaining two unverified corrections together (derive from an unverified ms1 candidate, compare against an unverified mk1 candidate) is a materially weaker guarantee than the "one bearer secret, one external card" case the design assumes. Needs an explicit rule (likely: oracle only engages if mk1 decodes CLEANLY with zero repair; if mk1 itself needed repair, fall through to ms1-Candidate/exit-4 as today).
  3. **Multisig watch-only / partial-supply interactions.** `mk1` is `required_unless_present_any = ["bundle_json", "extra_strings"]` (`:183`) — so in virtually every real single-sig invocation mk1 IS present, satisfying the oracle's precondition; but multisig has legitimate NotSupplied/partial-cosigner states (`MappingFailure::NotSupplied`, `:2378`) where the corresponding cosigner's mk1 simply isn't there — the oracle degrades gracefully to "no oracle available, stay Candidate/exit-4" in that case, which must be an explicit branch, not an accidental panic/unwrap.
  4. **Reject typed-error plumbing.** A NEW `ToolkitError`/`RepairError` variant is likely needed for "oracle ran, xpub mismatched" (distinct from the existing decode-failure checks) if the design wants a hard, named Reject rather than just an `ms1_decode: false` check row with a stderr note. Per the repo's alphabetical-ordering convention for new `ToolkitError` variants (CLAUDE.md Conventions), this needs to land alphabetically in the (still only partially sorted, per the tracked `error-rs-retroactive-alphabetical-sort` backlog) match blocks.
  5. **`no_auto_repair` / `--no-auto-repair` interaction** is already plumbed generically into every site (`no_auto_repair: bool` param) — the oracle path must respect it identically (skip entirely), no new flag needed for this.

## Cross-cutting observations

1. **The single biggest scoping finding: 8 of 9 auto-repair call sites need ZERO code changes.** Cycle E's generic `SetVerify::Blessed`-vs-else gate in `try_repair_and_short_circuit` and the generic `Unverified`-vs-else check in `cmd/repair.rs::run` were both written kind-agnostically even though only mk1 exercised the `Unverified` branch at the time. Flipping the Ms1 arm's hardcoded `Blessed` to a conditional `Unverified` is sufficient, by itself, to satisfy "never a silent exit-5 bless" + "exit-4 VERIFY-ME for the standalone `mnemonic repair`" for every intake path EXCEPT the 2 verify-bundle ms1 sites, which need NEW code specifically to implement the auto-oracle (not just to avoid a silent bless — they already avoid it for free).
2. **The verify-bundle auto-oracle needs no md1 and no account-path guess** — `mk_codec::KeyCard.origin_path` (already public) is the exact derivation path, and `bitcoin::bip32` (already a toolkit dep, already used for an analogous purpose in `emit_full_path_parent_fingerprint_check`) is the exact derivation primitive. This makes the auto-oracle considerably more tractable than "derive at a guessed default path" would have been.
3. **A genuine architectural fork is open and unresolved: does `ms repair` (standalone, no oracle) get exit 4 (mirroring `mnemonic repair`) or stay exit 5 + advisory (mirroring `mk-cli`'s Cycle E precedent)?** The task's framing ("Exit 5 becomes unreachable for ms1 EXCEPT via a real oracle") argues strongly for the former, but that breaks the existing 4-manual-chapter "exit 5 consistent across all four CLIs" invariant a second time (it's arguably already broken once, by mk1). The brainstorm must decide this explicitly and update the manual invariant sentence accordingly — this is a spec decision, not a recon finding to silently resolve.
4. **The manual's own Cycle-E-era prose (`41-mnemonic.md:3056-3059`) currently argues ms1 has NO analogous risk** ("no partial-set concept applies") — this sentence will read as WRONG once Cycle F ships and needs a rewrite (not an addition) explaining that ms1's risk is actually a worse, fully-undetectable variant of the same class, closed only by the new verify-bundle oracle (and left open for the fully-standalone `ms repair`/`mnemonic repair --ms1` case, where no oracle exists).
5. **Sync state is clean across all three repos** — no drift risk from stale local branches; all citations were checked against actual current file content (not memory/inference).
6. **The "11 callers" figure has no traceable source** in either repo's `design/` tree — treat it as approximate/incorrect; the spec should cite 9.
