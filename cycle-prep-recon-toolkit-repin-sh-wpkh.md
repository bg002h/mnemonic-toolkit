## Cross-cutting observations

1. **COMPILE-BLOCKING, not in either FOLLOWUP's scope: md-codec 0.41.0 adds `Error::MalformedPayloadPadding` (F-A8), and the toolkit has two hand-exhaustive matches over `md_codec::Error` that will fail to compile without a new arm.** Per md-codec's own `CHANGELOG.md` (`## md-codec [0.41.0] — 2026-07-10`), the release is "**SemVer-minor — BIP-alignment cycle: decode-behavior changes + new error variant**" and bundles **F-A1 (sh(wpkh), the one FOLLOWUP-cited change) + F-A2 + F-A8**, not F-A1 alone:
   - `md_codec_exit_code()` — `crates/mnemonic-toolkit/src/error.rs:505-569` — self-documents at line 494: `` /// SPEC §6.4.5 routing. md_codec::Error is NOT `#[non_exhaustive]`; match is exhaustive. `` — confirmed no wildcard arm (closes at line 568-569 on `WireVersionMismatch => 3`). `md_codec::Error` genuinely has no `#[non_exhaustive]` (confirmed at `descriptor-mnemonic/crates/md-codec/src/error.rs:20`).
   - `friendly_md_codec()` — `crates/mnemonic-toolkit/src/friendly.rs:216-406` — self-documents at line 217: `// md_codec::Error is NOT #[non_exhaustive]; exhaustive match required.` — also genuinely exhaustive.
   - Both **MUST** gain a `md_codec::Error::MalformedPayloadPadding { bits }` arm before the re-pin will compile. This is a hard requirement, not a discretionary nice-to-have — the SPEC needs to pick an exit code (almost certainly `2`, the decode-reject class every sibling TLV/chunk-shape variant already uses) and a friendly message, and decide alphabetical-vs-grouped placement (the existing blocks are grouped by exit-code-class/chronology, not alphabetical, despite CLAUDE.md's alphabetical-ordering convention for *new* exhaustive blocks — a call for R0, since this is an addition to a pre-existing non-alphabetical block, not a from-scratch one).
   - A precedent test pattern already exists for exactly this kind of "N new variants land in one bump" cell: `md_codec_v0_30_new_variant_routing` (`error.rs:1222-1269`) — the SPEC should add an analogous `md_codec_v0_41_new_variant_routing` (or fold into it) pinning `MalformedPayloadPadding`'s exit code + `kind()`.
   - `repair.rs`'s own translation of `md_codec::decode_with_correction`'s `Err` (used by BCH auto-repair, `repair.rs:1684-1713`) is **NOT** exhaustive (has an `Err(other) => Err(RepairError::PostCorrectionDecodeFailed { ... detail: other.to_string() })` wildcard at line 1710) — it will compile fine unmodified, but will silently degrade a `MalformedPayloadPadding` surfaced mid-repair into a generic message. Given `reject_non_zero_pad` runs inside the shared TLV parser hit by every decode path (confirmed: `descriptor-mnemonic/crates/md-codec/src/tlv.rs:290-334`, called from both the plain-decode and chunk-reassembly/`decode_with_correction` routes), a BCH-corrected candidate that happens to land on non-zero padding bits would newly reject post-repin where it previously silently passed — worth a funds-adjacent test cell in the SPEC (not compile-mandatory, but a real corrected-outcome-flip risk in `repair.rs`).

2. **`decode_md1_string`'s F-A2 auto-dispatch fix (also in the same 0.41.0 release) appears NOT to change toolkit-observable behavior — but should be confirmed by full-suite run, not assumed.** md-codec's CHANGELOG: "`decode_md1_string` auto-dispatches a chunked single string (was `WireVersionMismatch { got: 9 }`)." The toolkit's only direct call to `decode_md1_string` in its own `src/` is a test fixture (`prop_repair_never_wrong.rs:483`, a known-non-chunked constant, unaffected). The toolkit's production decode paths (`restore.rs:316`, `verify_bundle.rs:388`) call `md_codec::chunk::reassemble` directly, not `decode_md1_string`. `decode_with_correction`'s own internal single-string bypass (`descriptor-mnemonic/crates/md-codec/src/chunk.rs:606-634`) ALREADY branches on the chunked-flag itself before deciding whether to call `decode_md1_string` (flag==0) or fall through to `reassemble` (flag==1) — so it was never exposed to the bug F-A2 fixes. **Conclusion: likely a no-op for the toolkit, but the SPEC should run the full suite (not just targeted tests) to confirm, per the standing convention that CLI/decode-adjacent re-pins ripple outside any one phase's targets.**

3. **F-A3 (`--force-long-code` hard error) and F-A4 (P2SH-multisig stderr advisory) are `md-cli`-only changes, not `md-codec` — irrelevant to this re-pin, and correctly walled off by the frozen sibling pin.** Confirmed via `descriptor-mnemonic/CHANGELOG.md`: `## md-cli [0.12.0] — 2026-07-10` (separate crate/entry from `md-codec [0.41.0]`) carries F-A3/F-A4.
