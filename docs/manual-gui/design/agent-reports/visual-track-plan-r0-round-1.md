### I1 — P1's "reuse the whole-form render incl. positionals/action-bar" names helpers that are NOT reusable as they stand; the promotion refactor is missing from the file list.

`tests/ui_harness/mod.rs::render_whole_form` (`mod.rs:418-453`) **deliberately omits** positionals, the action bar, and the bespoke sub-surfaces — its own doc says so (`mod.rs:410-417`: "Deliberately omits the bespoke SUB-SURFACES (SlotEditor, the archetype param sub-form, the tree builder, positional widgets)"). The extended path the plan actually needs exists — but as **private `fn`s inside `tests/gui_render_faithfulness.rs`**: `render_one_positional` (:74), `render_positionals` (:109), `render_action_bar` (:121), `render_extended_form_harness` (:127), with the file's header stating the situation exactly ("`render_whole_form` (PR #24) renders only the flag grid — no positionals, no action bar. This file adds the missing render surface", :39-44). Integration-test binaries share code only via `mod ui_harness;` (`gui_render_faithfulness.rs:65`, `ui_harness_i1_roundtrip.rs:35`); a new `tests/gui_form_snapshots.rs` cannot call private fns in another test binary. **Fold:** P1's file list gains (a) promote the four extended-render fns from `gui_render_faithfulness.rs` into `tests/ui_harness/mod.rs` (pub), (b) re-point `gui_render_faithfulness.rs` at the shared copies (behavior-identical move — but it edits a shipped GREEN test file, so it must be planned and covered by the P1 R0, not improvised). Without this, the implementer either duplicates ~80 lines of render-path code (drift seed) or makes the unplanned edit anyway.

### I2 — `workflow_dispatch` mechanics: the P0 trigger half is inert on a branch, and P1's red-corpus mitigation is mechanically impossible as written.

GitHub docs (events-that-trigger-workflows, `workflow_dispatch`): **"This event will only trigger a workflow run if the workflow file exists on the default branch."** — fetched and quoted verbatim; UI, API, and `gh workflow run` all refuse a branch-only workflow. Consequences for the plan:

- **P0** (`workflow_dispatch`+PR-triggered throwaway on the feature branch): workable, but ONLY via the `pull_request` half — workflow files added on a same-repo PR head branch do run for that PR's `pull_request` events. The `workflow_dispatch` half is dead weight until merged. The plan's "(iii) … re-run the job" must be stated as **"Re-run all jobs" on the PR-triggered run (fresh runner instance each time) or empty-commit pushes** — not dispatch. See explicit ruling (1) below.
- **P1 Risk bullet** — "if the PR's snapshots job REDs on the local corpus, regenerate via the job's own environment (the I2 ladder's `workflow_dispatch` path) BEFORE merge" — is broken twice: no such workflow is in P1's deliverable list, and even if added on the branch it **cannot be dispatched** (default-branch rule). **Fold:** pre-merge in-CI regeneration = push a temporary commit flipping the PR's own snapshots job to `UPDATE_SNAPSHOTS=1` + upload the regenerated corpus as an artifact, commit it, revert the flip (or an equivalent guarded step keyed on a label). Separately, state the disposition of the spec-§5 ladder's `workflow_dispatch` regen path explicitly: **deferred-until-systemic-drift (spec rung 4), post-merge-only by construction** — deferred-documented, not a P1 deliverable. Right now the plan cites it as if it exists.

### I3 — "Required check" has no enforcement substrate: mnemonic-gui master is UNPROTECTED. The plan needs an explicit repo-settings ship-step.

Verified live: `gh api repos/bg002h/mnemonic-gui/branches/master/protection` → **404 "Branch not protected"**; `rulesets?includes_parents=true` → **`[]`**. Adding the `snapshots` job to `build.yml` makes it a check that *runs*; nothing on this repo makes any check *required* — today all 11 PR checks (clippy, headless, msrv, 7 build-matrix, schema-mirror) are green-before-merge by convention only. The spec (GREEN) mandates "as a REQUIRED check", so the plan must carry what the workflow file cannot: **a one-time admin ship-step in P1's gate list — create a branch-protection rule (or repository ruleset) on `master` naming the `snapshots` status-check context as required** — flagged as an out-of-diff repo-settings action. Scope note to surface, not decide silently: requiring ONLY `snapshots` is the minimal footprint that satisfies the spec; requiring all 12 contexts changes merge discipline repo-wide (user call, one line in the P1 R0 report). Pin the context name (job id `snapshots` or its `name:` — whichever the job declares) so the protection rule matches.

### I4 — P0's MSRV live-check is vacuous as written: `cargo check --locked` never compiles dev-deps.

Plan P0 step 1: "Confirm `cargo check --locked` at MSRV 1.88 still passes (dify has no declared MSRV — **this is the live check**)". It is not: `cargo check` builds lib+bins only; `egui_kittest` and its `snapshot`-feature graph (dify/colored/getopts/rayon) are dev-dependencies compiled only for test targets. The CI msrv job has the same shape (`build.yml:84-85`, plain `cargo check --locked`) — so nothing anywhere compiles dify at 1.88 under the plan as written, and the spec-§5 obligation ("verify dify compiles at 1.88") is silently unmet. **Fold:** the live check is `cargo +1.88.0 check --locked --all-targets` (or `cargo +1.88.0 test --no-run`). Budget note: `--all-targets` at 1.88 compiles the whole existing test tree for the first time at the MSRV floor — if a pre-existing dev-dep floor violation surfaces, it is a P0 spike *finding* (report + decide), not an implicit fix obligation. Leave the CI msrv job's lib/bins scope unchanged (its charter is the shipped-binary floor).

### I5 — P2 image mechanics: "fails loudly" is empirically FALSE for the HTML artifact, the 5-consumer constraint cannot be met by a bare path form alone, and the named `\graphicspath` value is wrong.

Three verified facts the phase must absorb:

1. **pandoc `--self-contained` on a missing image = WARNING + exit 0** (empirically confirmed, pandoc 3.x: "[WARNING] Could not fetch resource …", exit 0, zero `data:` URIs in the output). `make html` goes GREEN with 61 broken images — and HTML is the gh-pages + release-asset artifact (`manual-gui.yml:303-314,367-426`). The plan's Risk line "fails loudly, but budget the iteration" is true only for xelatex (`-halt-on-error`, Makefile:173-182). **Fold:** the P2 gate gains a positive embed census on the built HTML — `grep -c 'img src="data:' build/m-format-gui-manual.html` == 61 (or build HTML with `--fail-if-warnings`) — as a scripted check, not an eyeball.
2. **lychee `--offline` DOES check markdown image links, and resolves them relative to the CONTAINING FILE** (empirically confirmed: `![](figures/gui/x.png)` in `sub/doc.md` resolves `sub/figures/gui/x.png` and errors; `../figures/…` passes). pandoc resolves relative to CWD/`--resource-path` (make runs in MANUAL_DIR). Therefore **no bare path form satisfies all five consumers**: manual-dir-relative (`figures/gui/x.png`) fails lychee (`lint.sh:86` runs over `$SRC_DIR`, files in `src/75-gui-forms/`); file-relative (`../../figures/gui/x.png`) fails pandoc-HTML inlining (silently, per fact 1) and xelatex-from-`build/`. "Pick ONE path form" must become "one path form + a named consumer-config alignment". The repo's own precedent points at the cleanest fix: the mermaid cache filter injects images by **absolute path via an exported env var** (`Makefile:98`, `MERMAID_CACHE_DIR := $(abspath …)`); a minimal Lua image-path rewrite (or `--resource-path` for HTML + `\graphicspath`/copy for PDF) reconciles the pandoc consumers while source md keeps the lychee/markdownlint-friendly relative form. Decide-in-phase stands — but with this corrected constraint map, so the phase iterates inside a solvable space instead of hunting for a path form that does not exist.
3. **The example value is wrong:** `\graphicspath{{../figures/gui/}}` prepends the directory to the path *as written* — with `figures/gui/<x>.png` image paths it searches `../figures/gui/figures/gui/<x>.png`. The value that matches that path form is `\graphicspath{{../}}`. (Placement in `pandoc/preamble.tex` is right — it exists and is `-H`-included, Makefile:166.)

Plan P0 renders ≥3 forms and measures "wall-time (cold + rust-cached) + per-PNG sizes" — an extrapolation, not the mandated measurement, at the one phase whose entire purpose is measurement and which cannot be cheaply re-run after ratification. The fix is two lines: the spike already runs the enumerator to compute the tallest-form ranking, so **render all 61** (write + size + time them; keep the cross-runner threshold comparison on the 3 anchor forms, or extend to all 61 for free — which also strengthens the (iii) fleet evidence). Fold into P0 step 2/3 and the persisted-report contents.

## Explicit rulings requested by the charter

**(1) Spike-trigger-mechanics ruling:** a branch-only `workflow_dispatch` workflow is **NOT dispatchable** — GitHub docs, verbatim: *"This event will only trigger a workflow run if the workflow file exists on the default branch."* The plan's hybrid trigger survives **only** because of its `pull_request` half: same-repo PR head branches run their own added workflow files on `pull_request` events. Ruling: the spike runs via the PR event; the ≥3-distinct-runner-instances evidence comes from **"Re-run all jobs"** (fresh runner each re-run) or empty-commit pushes; `workflow_dispatch` may remain in the file as a post-merge convenience but must carry no pre-merge weight. The same rule invalidates P1's pre-merge "regenerate via the workflow_dispatch path" mitigation (I2) — pre-merge in-CI regeneration must ride the PR's own job (temporary `UPDATE_SNAPSHOTS=1` flip + artifact). The ladder's dispatch-regen workflow is post-merge-only by construction; the plan should say "deferred until systemic drift, per spec rung 4" rather than citing it as existing.

**(2) Required-check-wiring ruling:** adding a job to `build.yml` creates a check that RUNS on every PR/master/tag push (triggers verified, `build.yml:3-8`) — it does **not** make it required. Verified live: **no branch protection, no rulesets** on `bg002h/mnemonic-gui` master. "Required" is achievable only via a repo-settings action (branch protection rule or ruleset naming the `snapshots` context). The plan MUST carry this as an explicit, honest ship-step in P1 (out-of-diff, admin, one-time), plus the scope question (require only `snapshots` vs all contexts) surfaced to the user. Until that step runs, the spec's "REQUIRED check" is satisfied only by the repo's existing green-before-merge convention — acceptable interim, but the plan may not imply the workflow file alone delivers it.
