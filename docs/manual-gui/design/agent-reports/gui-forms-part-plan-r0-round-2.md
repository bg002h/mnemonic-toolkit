## New Minor / Nit (non-blocking — apply during P2)

### M-6 — Harden the removal script against the 2 shared-stem TOUR collisions.

Two of the 61 stems are **also** tour includes with the identical fence:
`30-tour/31-first-launch.md:21` includes `gui/mnemonic-bundle.gui` and `:82` includes `gui/mk-inspect.gui` — and `mnemonic-bundle` / `mk-inspect` are both stems in `transcripts/gui/`. A removal script that keys on the **include path** (`include="gui/<stem>.gui"`) rather than on the **byte-uniform lead-in paragraph** would match the tour's two fences as well.

- **Wrongful-ADD case** (strip tour render *and* insert a cross-link there): the offending stem gets 2 `](#gui-form-S)` links (tour + subcommand chapter) → `gui-form-xref` RED. **Caught.**
- **Wrongful-REMOVE-only case** (strip the tour render, add no cross-link): the stem keeps 1 anchor + 1 link → gate **GREEN**; `verify-examples-gui` census counts distinct `.gui` files (61, Makefile:277) so it's unchanged; `make html` still builds. The tour silently loses a pedagogical inline render, leaving a dangling "…structural render … **is:**" with nothing after it. **Gate-invisible** — only the (placed, mandatory) post-impl whole-diff review catches it, as an unexpected 62nd changed file.

The plan-as-written is already safe (it names the *byte-uniform lead-in paragraph* — which is tour-absent — as the removal anchor, and explicitly states *"The tour … UNCHANGED (keeps its 2 inline renders)"*). This is why it is **Minor, not Important**: a faithful reading does not regress. But the adjacent instruction *"generated from the SAME stem list — script the edit"* could tempt an include-path-keyed implementation that trips the collision. Add one sentence to P2: **anchor the removal on the byte-uniform lead-in paragraph** (equivalently: restrict the edit set to the 61 files containing that lead-in, OR explicitly exclude `src/30-tour/`) so the 2 tour fences sharing stems `mnemonic-bundle` / `mk-inspect` are never matched.

### M-7 — `TRANSCRIPTS_GUI` wording: reuse the existing var, don't redefine it.

P1 phrases the M-4 fold as *"`TRANSCRIPTS_GUI := $(abspath $(SRC_DIR)/../transcripts/gui)` passed to lint.sh."* The variable **already exists** at Makefile:276 as `TRANSCRIPTS_GUI := $(TRANSCRIPTS)/gui` (round-1 M-4 located it there). The two expressions resolve to the same path, but the plan's phrasing reads as a fresh — and redundant — redefinition. The correct, minimal action is: add `TRANSCRIPTS_GUI="$(TRANSCRIPTS_GUI)"` to the `lint:` arg block (Makefile:296-305) referencing the **existing** line-276 var; do not introduce a second `:=` assignment. Functionally inert either way (same path), so cosmetic only.
