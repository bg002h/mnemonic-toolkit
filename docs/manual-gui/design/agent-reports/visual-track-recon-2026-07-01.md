All crate citations from the versions resolved in mnemonic-gui's lockfile: egui_kittest-0.31.1, egui-wgpu-0.31.1, wgpu-24.0.5, egui-0.31.1, epaint-0.31.1, dify-0.7.4; registry paths abbreviated `REG`. ## Q1 — The snapshot API (VERIFIED, source)

- **Features** (`REG/egui_kittest-0.31.1/Cargo.toml:125-141`):
  - `snapshot = ["dep:dify", "dep:image", "image/png"]`
  - `wgpu = ["dep:egui-wgpu", "dep:pollster", "dep:image", "dep:wgpu", "eframe?/wgpu"]`
  - There is **no `default` feature list** → today's featureless `egui_kittest = "0.31"` dev-dep (mnemonic-gui `Cargo.toml:108`) is AccessKit-only, as assumed.
- **API**: `Harness::snapshot(name)` / `try_snapshot` / `snapshot_options` / `try_snapshot_options` (`src/snapshot.rs:359-450`); they call `Harness::render() -> Result<image::RgbaImage, String>` (`src/lib.rs:431-433`) which drives the `TestRenderer` (`src/renderer.rs:3-21`). Default renderer is `LazyRenderer` → `WgpuTestRenderer` when the `wgpu` feature is on (`src/renderer.rs:36-45`). The renderer tessellates the last frame's shapes and draws into a fresh offscreen texture (`LoadOp::Clear(TRANSPARENT)`), then reads it back (`src/wgpu.rs:131-212`, `src/texture_to_image.rs:8-62`). No window, no surface, no compositor. Free functions `try_image_snapshot[_options]` accept any `RgbaImage` (`src/snapshot.rs:190,311`) — usable from a gated bin, not just tests. The 0.30-era `wgpu_snapshot*` names are deprecated aliases (`src/snapshot.rs:452-492`).
- **File layout** (`src/snapshot.rs:17-18, 179-185, 208-218`): default `output_path = "tests/snapshots"` (a `PathBuf`, overridable via `SnapshotOptions::output_path`); committed file `{name}.png`; run artifacts `{name}.new.png`, `{name}.diff.png`, `{name}.old.png` (README says gitignore `.new`/`.diff`, `README.md:53-57`).
- **Comparison**: NOT byte-equal. `dify::diff::get_results(previous, new, threshold, /*detect_anti_aliased_pixels=*/true, None, &None, &None)` (`src/snapshot.rs:274-275`). Default `threshold = 0.6` with the comment "enough for most egui tests to pass **across different wgpu backends**" (`src/snapshot.rs:10-14, 22-27`). Semantics (dify source): per-pixel squared YIQ distance `0.5053·ΔY² + 0.299·ΔI² + 0.1957·ΔQ²` (`REG/dify-0.7.4/src/yiq.rs:55-59`) compared raw against the threshold (`diff.rs:69-71`; max possible delta 35215, `diff.rs:7` — so 0.6 is a near-exact, ±1-LSB-class tolerance). Pixels classified anti-aliased are excluded (`diff.rs:113-115`); **any single remaining differing pixel fails** (`diff.rs:109-112, 120-124` → `snapshot.rs:277-292`). There is no pixel-count budget (tracked upstream as egui #5683, cited in `README.md:79`).
- **Update flow**: env var `UPDATE_SNAPSHOTS` (anything except `false|0|no|off` enables), updates files and **succeeds** the test (`src/snapshot.rs:106-107, 161-169`).

## Q2 — Headless / GPU-less rendering (VERIFIED, source + upstream CI + local experiment)

- **Adapter selection**: `default_wgpu_setup()` removes WebGPU and installs a `native_adapter_selector` whose comment is literally "Prefer software rasterizers": sort key `DeviceType::Cpu => 0` ("CPU is the best for our purposes!"), backend order Metal < Vulkan < Dx12 < Gl (`src/wgpu.rs:11-50`). So with lavapipe/llvmpipe present, kittest picks the CPU adapter automatically.
- **Backends compiled on Linux**: kittest's own `wgpu` dep only lists `metal`+`dx12` features (`egui_kittest Cargo.toml:106-113`), but wgpu 24 enables **Vulkan and GLES unconditionally on non-Apple unix via target-cfg deps**, independent of features: `[target.'cfg(any(windows, all(unix, ...)))'.dependencies.wgc] features = ["vulkan"]` and `[target.'cfg(all(unix, not(target_vendor="apple")))'...] features = ["gles"]` (`REG/wgpu-24.0.5/Cargo.toml:161-178`). So the Linux build always has Vulkan+GL backends.
- **Env knobs**: `WGPU_BACKEND` / `WGPU_POWER_PREF` are honored — kittest's setup starts from `WgpuSetupCreateNew::default()` which reads `wgpu::Backends::from_env()` (`REG/egui-wgpu-0.31.1/src/setup.rs:10-18, 167-173`).
- **egui's OWN CI does NOT run snapshots on Linux**: the main test job (`cargo test --all-features`) runs on **`macos-latest`** with `lfs: true`, master workflow comment: "We run the tests on macOS because it will run with an actual GPU" (fetched `raw.githubusercontent.com/emilk/egui/{0.31.1,master}/.github/workflows/rust.yml`). Snapshots are uploaded as artifacts on failure.
- **wgpu's OWN CI is the software-rasterizer precedent**: runs its GPU test suite on `ubuntu-24.04` with a **pinned Mesa 25.2.7** tarball (from `gfx-rs/ci-build` releases), setting `VK_DRIVER_FILES` to a generated lavapipe ICD json, `LD_LIBRARY_PATH`, `LIBGL_DRIVERS_PATH`; Windows uses the same Mesa (llvmpipe) + WARP (fetched `raw.githubusercontent.com/gfx-rs/wgpu/trunk/.github/{workflows/ci.yml,actions/install-mesa/action.yml}`).
- **Ubuntu package path**: `mesa-vulkan-drivers` on noble (24.04) ships lavapipe: `/usr/lib/x86_64-linux-gnu/libvulkan_lvp.so` + `/usr/share/vulkan/icd.d/lvp_icd.x86_64.json` (fetched packages.ubuntu.com filelist).
- **Local empirical proof**: spike crate built with `egui_kittest = { version = "=0.31.1", features = ["wgpu","snapshot"] }`; with `DISPLAY`/`WAYLAND_DISPLAY` **unset**, `harness.render()` succeeded both on Intel Vulkan and on `llvmpipe (LLVM 22.1.6, 256 bits)` `device_type=Cpu` via `WGPU_BACKEND=gl LIBGL_ALWAYS_SOFTWARE=1`. Headless-without-display: demonstrated. (Lavapipe-Vulkan specifically was not testable on this machine — no lvp ICD installed; the lavapipe fact rests on wgpu CI + the apt filelist. Marked as the one residual to confirm in the P0 CI spike.)

## Q3 — Determinism (VERIFIED, mixed source + experiment)

- **Same machine, same backend/driver**: two consecutive runs produced **byte-identical** raw RGBA and PNG (sha256 `e2b5e6af…` both runs). PNG encoding via `image` is deterministic given identical pixels.
- **Same machine, different backend**: Intel-Vulkan vs llvmpipe-GL renders of the identical UI **differ at the byte level** (raw sha `c7632f…` vs `697cec…`). At dify threshold 0.1 → 34 differing pixels; at kittest's default **0.6 → 0 differing pixels (MATCH)**. So the default threshold absorbed a *full backend swap* for a typical form UI.
- **Maintainer documentation**: kittest's README has a whole section "What do do when CI / another computer produces a different image?" enumerating drift sources — MSAA sample placement, texture filtering, float rounding per driver/backend/driver-version (`README.md:71-116`); the default-threshold comment cites cross-backend variance (`snapshot.rs:11-13`); egui stores snapshots in **git LFS** (`README.md:66`; `snapshot.rs:135` error text tells users to set up git-lfs).
- **Pinned-container byte-identity**: a container pins mesa+LLVM versions, but llvmpipe/lavapipe JIT-compile with CPU-feature-dependent codegen (the adapter string itself advertises "LLVM 22.1.6, **256 bits**" — vector width varies by host CPU). Whether pixels are bit-identical across AVX2 vs AVX-512 hosts is **UNVERIFIED** — no authoritative statement found either way. Treat byte-diff-in-container as plausible but unproven.
- **CONCLUSION**: a regenerate-anywhere **byte-diff gate is not supportable** (disproved empirically and by upstream docs). The evidence supports (b) **threshold/perceptual diff** as the gate primitive — specifically kittest's own dify comparison, whose default 0.6 is simultaneously near-exact (sub-LSB YIQ tolerance, any 1 pixel over fails) and empirically robust to backend/driver swap. Option (a) pinned-container byte-diff is a possible *tightening* but rests on the unverified LLVM-JIT-identity assumption.

## Q5 — Runtime determinism knobs (VERIFIED)

- **Cursor blink**: the Harness constructor itself disables it — `ctx.all_styles_mut(|style| style.visuals.text_cursor.blink = false); // Disable cursor blinking so it doesn't interfere with snapshots` (`src/lib.rs:94-95`). Construction runs to quiescence via `run_ok()` (`lib.rs:132-134`); `run()` steps until no immediate repaint is requested, bounded by `max_steps` (default 4, `step_dt` 0.25s — `builder.rs:19-24, 46-64`), so egui animations settle before a snapshot. `run_steps(n)` exists for still-animating UIs (`lib.rs:342-346`).
- **Size/scale**: `HarnessBuilder::with_size` (default 800×600) and `with_pixels_per_point` (default 1.0) (`builder.rs:19-44`); `fit_contents()` shrink-to-fit (`lib.rs:257-263`).
- **Theme**: egui defaults to **dark** — `theme_preference: ThemePreference::System` with `fallback_theme: Theme::Dark` (`egui-0.31.1/src/memory/mod.rs:321-322`); kittest supplies no system theme. mnemonic-gui sets no custom visuals (grep for `set_visuals|Visuals::|ThemePreference` in `src/` — no hits).
- **Fonts**: fully embedded — egui/epaint default features include `default_fonts` → `epaint_default_fonts` crate embedding `Hack-Regular.ttf`, `Ubuntu-Light.ttf`, `NotoEmoji-Regular.ttf`, `emoji-icon-font.ttf` via `include_bytes!` (`egui-0.31.1/Cargo.toml:118-119`, `epaint-0.31.1/Cargo.toml:131-132`, `epaint-0.31.1/src/text/fonts.rs:229`). mnemonic-gui's eframe features include `default_fonts` (`Cargo.toml:47`). No system fonts touch the render; glyph atlases are CPU-rasterized by epaint, so residual drift comes only from GPU blending/AA.
- **Backend pinning in CI**: `WGPU_BACKEND=vulkan` + lavapipe ICD (Q2) makes the adapter choice explicit rather than relying on the CPU-preference sort.

## Q6 — Sizing (measured; extrapolation is an estimate)

- Spike form-like UI: **800×600 @ ppp 1.0 → 18,197 bytes**; **1600×1200 @ ppp 2.0 → 56,165 bytes** (measured PNGs).
- 61 forms → **~1.1 MB at 1x, ~3.4 MB at 2x (ESTIMATE)**; denser real forms somewhat higher. Well under git-lfs-necessity; egui uses LFS for its (much larger) corpus, but LFS complicates CI checkout + plain-clone UX. Plain committed PNGs with a size budget look sufficient; a spec decision, not a constraint.

## Feasibility verdict

**YES — with a software Vulkan rasterizer installed in CI.** Plain `ubuntu-latest` + `apt-get install mesa-vulkan-drivers` (lavapipe, package verified) — or, more reproducibly, the wgpu-CI pattern of a **pinned Mesa (25.2.7) tarball + `VK_DRIVER_FILES`** — gives kittest exactly the CPU adapter its selector is designed to prefer. Compile path (Vulkan always built on Linux), adapter-selection source, wgpu's own ubuntu-24.04 CI, and a local headless render all corroborate. Residual to confirm in the P0 spike: one live run of lavapipe-Vulkan on an actual GitHub runner — llvmpipe-GL headless is proven locally, lavapipe itself UNVERIFIED-by-recon (verified only via wgpu CI precedent).

## Gate-architecture recommendation

**Threshold-diff (option b), implemented with kittest's own dify mechanism, against committed PNGs regenerated in CI on a version-pinned Mesa.** Concretely:

1. Commit `{name}.png` per form; gate = regenerate + `try_image_snapshot_options` at the default **0.6** threshold (near-exact: any single pixel beyond ~1-LSB YIQ distance fails, AA-detected pixels excluded) — empirically robust across a full backend swap while still catching visible change.
2. **Pin the CI rasterizer** (wgpu-style Mesa tarball or at minimum the apt package on a pinned runner image) + `WGPU_BACKEND=vulkan` + fixed `with_size`/`pixels_per_point`/dark-theme/masked fixtures — drift-window small, regeneration reproducible in practice; within one pinned environment the output was byte-stable run-to-run.
3. Do **not** promise regenerate-anywhere byte-identity (disproved), and don't build the gate on pinned-container byte-diff unless the P0 spike also proves cross-CPU (AVX2 vs AVX-512) bit-identity of llvmpipe/lavapipe output — that fact remains UNVERIFIED. `UPDATE_SNAPSHOTS=1` is the regeneration UX; `.new.png`/`.diff.png`/`.old.png` must be gitignored.

This differs from the structural track's byte-diff gate by necessity; the threshold gate is the visual-domain analogue with an explicitly sourced tolerance rather than an accidental one.
