- The real re-pin is the **structural render**: `render_emit::render_form` enumerates the dropdown's `opts` **inline** (`render_emit.rs:678` `format!("dropdown[{}]", join_opts(opts))`; `:698-702` maps `""`→`"(none)"`). Export-wallet's `--template` currently renders `dropdown[bip44,…,tr-sortedmulti-a]` with **no `(none)`** and is an **exact-ASCII `assert_eq!` pin** (`gui_render_emit.rs:95`, inside `exact_render_...export-wallet`, `:93-115`).

So the honest census is **per artifact-surface**, not "N of 61 forms":

| `figures/gui` gallery PNG (closed form) | **inert (0)** |

The "≤2 of 61" **bound** is fine as a scope-smell tripwire; the plan's **encoding** of *which* artifacts move is wrong (it points at the gallery; the movement is in the structural render + its toolkit copy + the exact-ASCII source). Fold into I1.

### (e) The uncounted ~+16 MiB committed `docs/gui_example.pdf` — **RULED: FLAG-TO-USER, NOT an autonomous call. (→ I2)**
Weighing the repo-growth reality honestly, the plan's "PDF-committing is spec-locked (Examples.pdf precedent)" (line 154) **does not hold up**:
- **The precedent is not weight-comparable.** `docs/Examples.pdf` is **215,270 bytes (215 KB)** — a text-heavy CLI book with little/no raster. `gui_example.pdf` embeds 51 whole-window PNGs at ~330 KB each ≈ **~16 MiB**, re-committed to git history on *every* corpus change, forever. "Examples.pdf is the committed-PDF precedent" is a precedent for committing a *small text* PDF, not a multi-MiB raster one.
- **The stack this book joins argues the other way.** `gui_example.pdf` lives inside `docs/manual-gui/` and shares its Makefile/lint/workflow. The existing manual-gui book PDF (`m-format-gui-manual.pdf`) is **not committed** anywhere (`git ls-files docs/**/*.pdf` shows only `Examples.pdf` + the unrelated `m-format-*.pdf` books + figure-cache PDFs) — it is **built in CI and release-attached** (`manual-gui.yml:380-382`, `gh release upload … m-format-gui-manual.pdf`). So the in-stack convention is **release-attach, not commit**. Under the plan, `gui_example.pdf` would be **both** committed (16 MiB in history) **and** release-attached (P4) — redundant.
- **The user did not explicitly lock "commit."** §1.1's lock is "gui_example.pdf **sitting beside** docs/Examples.pdf" — a *location* statement plus "generated + gated." Whether the built PDF is git-committed vs release-attached-only is not something the user weighed; the spec §3.2c over-read "beside" into "commit." The ~16 MiB permanent cost is real.

**This is a user decision, plainly.** The plan should NOT autonomously commit; it should carry an explicit checkpoint presenting: **(A)** commit `docs/gui_example.pdf` (Examples.pdf-beside literal; ~16 MiB permanent, re-committed per corpus change), vs **(B)** release-attach only (manual-gui-stack convention; keeps git lean; "beside Examples.pdf" satisfied by the release asset + a committed README pointer). If the user picks (B), spec §3.2c is amended. It bites at P3 (far downstream), so the question can be resolved before P3 without blocking P1 mechanically — but the plan-text must stop asserting it as settled. (Note: this is exactly the class the house STOP/escalation discipline reserves for the user, and the R0 brief explicitly asked whether this was spec-locked or an assumption — it is an assumption.)
