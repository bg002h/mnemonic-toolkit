## 1. Goal

A standalone, **generated + gated** GUI tutorial book — `gui_example.pdf`, sitting
beside `docs/Examples.pdf` — that walks the same wallet journeys as Examples.pdf but
as **GUI walkthroughs**: each tutorial step shows the real mnemonic-gui application
window with the form filled in, then the same window after clicking **Run**, with the
output panel genuinely populated by a real pinned-CLI execution. Every screenshot is
machine-generated by the house snapshot machinery and CI-gated; every full CLI output
is machine-captured and byte-gated (standing rule: all CLI output in all docs is
binary-generated).

### 1.1 USER-LOCKED design decisions (2026-07-02 — NOT re-openable in R0)

1. **Scope: ALL of Examples.pdf's journeys, GUI-style** — single-sig card set →
   conventional 2-of-3 multisig → the pathological 4-tier degrading wsh vault → its
   taproot twin → watch-only export (source `.examples-build/Examples.md`; built book
   `docs/Examples.pdf`).
2. **Visuals: filled screenshots, generated + threshold/byte-gated** (house pattern),
   and **WHOLE-WINDOW** — tab bar + subcommand selector + the filled form + the output
   panel, not the current form-only crops. No OS titlebar — kittest renders egui
   content only; documented caveat (see §10 non-goals and the reader-facing note).
3. **Post-Run shots show the output panel GENUINELY POPULATED by a real pinned-CLI
   execution in the harness** — two shots per step (filled-form, post-Run), the I4 +
   snapshot machineries combined.

Recon fidelity note on decision 2: in the current shell the "subcommand selector" is a
`ComboBox` dropdown inside the CentralPanel (`mnemonic-gui src/main.rs:525`), not a
persistent side list. Whole-window shots therefore show: top tab strip
(`main.rs:424-453`), central panel with pinned-version line + subcommand combo +
`?` help + the scrollable form (`main.rs:512-889`) + action bar with `Run`
(`main.rs:975-1026`), bottom output panel (`main.rs:456-509`). That IS the whole
window as a user sees it; the spec does not invent a side list.

## 2. Recon summary — the four load-bearing answers

**(A) App-shell extraction — MODERATE, well-mapped.** `MnemonicGuiApp` is bin-private
in `src/main.rs` (struct `:98-151`, `impl eframe::App` `:383`); the `update()` body
**never touches `eframe::Frame`** (`_frame` unused, verified) and contains no
viewport/repaint/file-dialog/clipboard entanglement. All OS side-effects live in
`main()` + `new(cc, …)`: window-capture protection (`:166-174`), 1 Hz keepalive thread
(`:195-202`), SIGINT/SIGTERM + ctrlc handlers (`:217-256`), and `AppState::detect_all()`
$PATH probing (`:263`). egui_kittest 0.31.1 natively steps Context-closures with real
panels (`Harness::build_state`, `egui_kittest-0.31.1/src/builder.rs:113`). Minimal
extraction: lift the app into a `#[cfg(feature="gui")]` lib module, split the
constructor (pure init + injectable `AppState`, side-effects stay in the `cc` wrapper),
extract `fn ui(&mut self, ctx)` from `update()`. No egui_kittest `eframe` feature
needed.

**(B) Populated pane — FEASIBLE, and simpler than feared.** The runner is **fully
synchronous**: `spawn_and_capture` (`main.rs:1220`) calls
`runner::run_with_stdin` inline **inside `update()`**; completion lands in
`app.last_run` the *same frame* as the Run/confirm click. No channels, no polling, no
spinner, no timestamps/durations/PIDs anywhere in the pane — the pane is timing-free
(recon B §2). The production argv echo uses bare `schema.cli_name` as argv[0]
(`form/invocation.rs:115-116`) → deterministic. The one conditional between click and
spawn is the "Confirm secret-bearing run" modal (`main.rs:1081-1124`), which a driver
clicks through (two-click drive for secret forms) — itself a tutorial teaching moment.
CI already knows how to put the four pinned CLIs on `$PATH` under bare names
(`schema-mirror.yml:59-133`).

**(C) Fixtures + flows — everything is GUI-expressible; secrets are demo-only.**
Every journey subcommand has a GUI form (schema cites in recon C §3). The J3 vault
feeds `--spec policy.json` through the form's `Path` field (`schema/mnemonic.rs:3921`,
`stdio_sentinel:true`) — the guided-builder **refusal** teaching moment survives; the
vault itself rides `--descriptor-file` Path fields, exactly like the CLI journey. The
only secret-bearing inputs anywhere are the **three world-known BIP-39 test phrases**
(S0/S1/S2), all printed verbatim in Examples.md; J3/J4/J5 are entirely watch-only.
Masking for the exact journey entry paths (SlotEditor `@N.phrase` rows; `convert
--from phrase=` composite) is complete — via the `SECRET_SLOT_SUBKEYS` /
`SECRET_NODE_TYPES_ARGV` taxonomies, **not** the I3 64-flag census (precision matters:
see §7). Shell-only glue (python hashlock helpers, jq/sed md1 hops, `$(cat …)`)
all has enumerated adaptations (recon C §5).

**(D) Sizing — fixed window is new territory; scrolling has a guaranteed fallback.**
The existing corpus is auto-sized (`fit_contents()`); a fixed whole-window shot must
budget for the tab strip + output panel and the tallest journey forms
(`mnemonic-restore` 1169 px, `mnemonic-bundle` 912 px physical @ ppp 2.0), so
**in-viewport scrolling is unavoidable for some steps**. kittest scroll control has
zero in-repo precedent (UNVERIFIED → spike), but a guaranteed fallback exists: a
scroll-offset seam (`ScrollArea::vertical_scroll_offset`) on the extracted app.
Conventions to inherit unchanged: ppp 2.0, dark theme, embedded fonts, dify 0.6
threshold, lavapipe CI recipe, CI-compares-never-regenerates.

**(E) Book — follow the manual-gui gated model, not the un-CI'd Examples model.**
`docs/Examples.pdf` is generated by `.examples-build/gen.sh` (real-binary capture,
hard-pinned `mnemonic 0.55.3`, `gen.sh:22`) but has **no CI gate at all**. The
manual-gui stack (pandoc→xelatex Makefile, 9-phase lint, verify-figures byte-census,
pinned-clone plumbing, release job) is fully reusable and fully gated. The tutorial
book therefore lives inside `docs/manual-gui/` as a **sibling book** sharing that
stack (decision + rationale in §5.2).

**Version-skew consequence (spec decision):** Examples.pdf is branded `v0.55.3`
throughout; the current tier is toolkit v0.75.0. The GUI tutorial pins its **own**
tier (the GUI repo's pinned toolkit tag at capture time) and **explicitly disclaims
byte-parity with Examples.pdf's printed output**. Journey *shape* (commands, flags,
teaching moments) mirrors Examples.pdf; output *bytes* are regenerated at the
tutorial's pin. A P1 task re-runs each journey's commands against the current tier and
flags any behavioral divergence (e.g. changed refusal wording) for the chapter prose.

### 3.1 GUI leg (mnemonic-gui)

**(a) App-shell extraction (the enabling refactor — zero behavior change).**
Per recon A §5:
1. Lift `MnemonicGuiApp` (struct + impls + `render_exit_badge` + `spawn_and_capture`
   + `AUTOSAVE_INTERVAL`; `main.rs:98-1260`) into a new `#[cfg(feature = "gui")]`
   library module (working name `src/app_window.rs`), re-exported; `main.rs` shrinks
   to `fn main()` + the eframe bootstrap.
2. Split the constructor: `MnemonicGuiApp::new_headless(app_state: AppState, loaded:
   Option<PersistedState>, state_path: Option<PathBuf>)` = pure field init only
   (today's `main.rs:258-335` minus side-effects). The existing `new(cc, …)` becomes a
   thin wrapper: run the three `cc`-dependent OS effects (window-capture, keepalive
   thread, signal handlers), call `AppState::detect_all()`, then `new_headless`.
3. Extract `fn ui(&mut self, ctx: &egui::Context)` = today's `update()` body verbatim
   (already `Frame`-free); `impl eframe::App::update` becomes `self.ui(ctx)`.

Invariants: the production binary's behavior is byte-identical (pure relocation);
`cargo build --no-default-features` still builds (the lifted module is gui-gated —
this class of edge bit the form-renders cycle twice pre-code, so it is an explicit
per-phase check); `gui-render` bin and the existing 61-form snapshot suite are
untouched; **no clap-surface change anywhere in the cycle → no toolkit schema-mirror
update needed** (the paired-PR rule is satisfied vacuously — state this in the PR).

**(b) Tutorial capture harness** — new env-gated integration test
`tests/gui_tutorial_snapshots.rs` (`GUI_TUTORIAL_SNAPSHOTS=1`, early-return-skip like
`gui_form_snapshots.rs:109-116`):
- Builds the real app: `Harness::builder().with_size(WINDOW_SIZE)
  .with_pixels_per_point(2.0).build_state(|ctx, app| app.ui(ctx), app)` where `app =
  MnemonicGuiApp::new_headless(fixed_appstate_all_found(), None, None)`. **Fixed
  window size** (spike-ratified, §4) — no `fit_contents()`.
- Executes the **step manifest** (one `#[test]` fn, steps strictly sequential — this
  permits a single `set_current_dir` to the fixture dir so Path fields hold short
  display names like `policy.desc`, and permits intra-journey chaining):
  per step: select tab + subcommand (real combo drive), fill fields via the existing
  drive-dispatch primitives (`tests/ui_harness/mod.rs:282-334`, extended to drive
  widgets *inside the real window* by AccessKit node lookup), set scroll position,
  `harness.run()` to quiescence, snapshot `<stem>-form.png`; click `Run` (+ modal
  `Run` for secret steps, snapshotting `<stem>-modal.png` where the manifest says so);
  the subprocess completes **in-frame** (recon B §1); `harness.run()`; snapshot
  `<stem>-run.png`.
- After each Run: persist the **full RunResult text** as corpus files
  `tests/snapshots/tutorial/<stem>.stdout.txt` (+ `.stderr.txt` when non-empty, +
  exit code in a one-line `.exit.txt`), byte-gated exactly like the PNGs
  (regenerate-and-diff in the same job). This guarantees pane-pixels ↔ transcript
  identity from a single execution, and satisfies the standing binary-generated-output
  rule without re-running commands in toolkit CI.
- Intra-journey chaining replaces the jq/sed glue: e.g. J5's restore step parses the
  md1 chunks out of the *previous step's* captured `RunResult.stdout` (`--json` run)
  and types them into the repeating `--md1` rows — deterministic, no derived-fixture
  drift, and mirrors what a real user does (copy chunks from the output pane).
- Assertions per step (beyond snapshots): `exit_code` matches the manifest's expected
  value (refusal steps expect non-zero — J3 build-descriptor, J4 depth-2 + BSMS
  steps); chained values parse; secret-hygiene assertions (§7).

**(c) Fixtures** — `tests/tutorial/fixtures/`: byte-copies of the four tracked
primitives from toolkit `.examples-build/` under their tutorial display names
(`policy.json` gets the same `opensessame`-digest rewrite `gen.sh:33-37` applied

dir's README noting source repo + SHA + transform). Seed phrases live as the S0/S1/S2
literals in the manifest (already published in Examples.md — §7). Everything else is
produced by chaining.

**(d) CI job** `tutorial-snapshots` in `mnemonic-gui/.github/workflows/build.yml`:
the `snapshots` job recipe (lavapipe, `WGPU_BACKEND=vulkan`, CPU-adapter guard,
`.new.png` census, `.diff.png` artifacts; `build.yml:87-147`) **plus** the
schema-mirror job's pinned-CLI install recipe (`schema-mirror.yml:59-88`) so the
pinned `mnemonic` (and if any step needs them, `md`/`ms`/`mk`) are on `$PATH` as bare
names. Census: `.new.png` count == manifest shot count AND `.stdout.txt` regen count
== manifest run count. Runs on every PR/master/tag push (no path filter — same

**(e) Pin bump + tag.** Bump `mnemonic-gui/pinned-upstream.toml` toolkit tier

## 5. The book: placement, structure, naming

### 5.1 Naming scheme

Step stem: `tut-<j>-<nn>-<slug>` (e.g. `tut-j2-06-bundle-all-seeds`). Artifacts per
step: `<stem>-form.png` (one or more: `-form`, `-form2` for multi-scroll steps),
optional `<stem>-modal.png`, `<stem>-run.png`, `<stem>.stdout.txt` (+ `.stderr.txt`,
`.exit.txt`). Same stem across GUI corpus, toolkit figures/, transcripts/, and the
chapter anchor `{#tut-<j>-<nn>-<slug>}` — the xref gate keys on it (§8). The manifest
(in the GUI repo) is the single source of truth for the stem list; both censuses
derive their expected counts from it (a committed `manifest-stems.txt` emitted by the
harness, so the toolkit gate needs no Rust).

### 5.2 Placement decision

**Inside `docs/manual-gui/` as a sibling book** (chapters `tutorial/`, figures
`figures/tutorial/`, transcripts `transcripts/tutorial/`, shared Makefile + lint.sh +
workflow + **one** `pinned-upstream.toml`).

Rationale: (i) reuses the 9-phase lint + pinned-clone plumbing + release job wholesale;
(ii) one pin bump moves the form gallery, `.gui` renders, and tutorial corpus in
lockstep — two books pinning the same upstream at different tags would be a standing
incoherence generator; (iii) avoids duplicating the Dockerfile.build/pandoc stack.
Rejected alternative: standalone `docs/gui-examples/` dir (duplicate pin + lint + CI;
the `.examples-build` model additionally has **no CI gating at all** — recon E5 — and
inheriting that gap would violate the standing generated-output rule).

### 5.4 Scrolling / clipping contract (explicit, per the locked no-clipping question)

- Tutorial shots are **viewport-faithful**: fixed window, the form scrolls inside the
  CentralPanel `ScrollArea` exactly as production (`main.rs:593`). A visible scrollbar
  is correct, not a defect.
- Each `-form` shot is captured at a manifest-recorded scroll offset chosen so **every
  field the step fills is visible**; when driven fields span more than one viewport,
  the step emits additional `-form2…` shots at recorded offsets. No step may ship a
  shot where a driven field is clipped out of every form shot (machine-checkable: the
  driver asserts each driven widget's rect intersects the viewport in at least one
  captured offset).
- The **no-clipping canonical reference for full form geometry remains the existing
  form-only gallery** (`figures/gui/<tab>-<sub>.png`); every tutorial step cross-links
  its gallery figure. The tutorial's contract is "what the user sees at this moment,"
  not "the whole form at once."
- Post-Run shots: the pane's stdout `ScrollArea` clips at `max_height(180.0)`
  (`main.rs:490-495`) — deterministic top-slice. The **full** output always accompanies
  the shot as the gated transcript block. Chapter 0 explains both conventions to the
  reader, plus the no-OS-titlebar caveat.

## 6. Determinism contract

The corpus is deterministic given:
1. **Pinned renderer env** (unchanged conventions): egui_kittest 0.31.1 wgpu+snapshot,
   ppp 2.0, dark theme, embedded fonts, lavapipe CPU adapter with backend guard, dify
   0.6 threshold, CI compares/never regenerates, remediation ladder per
   `SPEC_gui_visual_screenshot_track.md:31`.
2. **Fixed window size** (S4-ratified), fixed manifest scroll offsets, single
   sequential `#[test]` (stable frame sequence; no parallel-test cwd races).
3. **Injected `AppState`** (all four CLIs `Found`) — no `$PATH`-dependent tab
   grey-out; `state_path = None` — no persisted-toggle interference (fresh-app
   defaults show argv+exit+stdout+stderr, `main.rs:271`).
4. **Pinned CLI tier** on `$PATH` as bare names (tutorial-snapshots job installs the
   `pinned-upstream.toml` tags). Pane argv echo is bare-`cli_name` argv[0] by
   production construction (`invocation.rs:115-116`) — no path normalization needed on
   the click path. (The I4-style argv[0]-overwrite pattern is NOT used for shots.)
5. **Deterministic commands only**: every journey command is a pure function of its
   fixed inputs (bundle/convert/export-wallet/restore/build-descriptor/compare-cost on
   the demo fixtures — no RNG, no timestamps; recon B §2 verified the pane itself is
   timing-free; refusal steps pin exact non-zero exits). Runner env is inherited +
   `MNEMONIC_FORCE_TTY=1` (`runner.rs:199`) — output is captured in TTY mode,
   consistently for pane and transcript since both come from the same RunResult.
6. **cwd pinned** to the fixture dir for the whole test (short relative display names
   in Path fields and argv echoes; runner inherits cwd, sets none itself — recon B §6).

## 7. Secret hygiene (first-class bar)
