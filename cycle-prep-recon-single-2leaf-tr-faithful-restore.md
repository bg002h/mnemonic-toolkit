**Findings:**
1. **No hidden blocker.** md-codec renders general single-leaf AND depth-1 two-leaf `tr(NUMS,‚Ä¶)` correctly; Display on `95fdd1c` is faithful for both; `derive_receive_addresses` consumes the reparsed string (multipath split + `derive_at_index` are tr-safe). The feature is genuinely doable NOW with zero md-codec changes.
2. **The `95fdd1c` Display bug is LEFT-BRANCH-specific, not depth-specific:** any `TapTree` appearing as a LEFT child mis-Displays (close-brace swallowed); pure right-spine trees of any depth Display correctly. The "depth‚â•2 = unsafe" framing in the FOLLOWUP is a conservative superset of the actual bug. Recommendation: KEEP the conservative ‚â§2-leaf (= no `Tag::TapTree` child of a `Tag::TapTree`) gate anyway ‚Äî a contract keyed to "which shapes a known upstream bug happens to spare" is fragile and unexplainable to users; the whole carve-out dissolves on the first miniscript release containing #953.
3. **Built-in backstop, but not a substitute for the gate:** ¬ß5 (`restore.rs:1152-1159`) `from_str`-parses the reconstructed string BEFORE ¬ß8 composes stdout ‚Äî a malformed Display aborts with exit 2 ("--md1 descriptor parse: taptree branch must have 2 children‚Ä¶") and never prints. So the funds-adjacent "user trusts a malformed printed descriptor" scenario does NOT materialize in the current code shape for left-heavy trees ‚Äî BUT right-spine depth‚â•2 would silently SUCCEED (correct Display), creating an inconsistent accepted-shape set; and relying on a downstream parse error for the refusal yields a cryptic message with no slug citation. The structural depth gate stays mandatory.
4. **The literal-`NUMS`-token asymmetry:** bundle's `.descriptor` echo preserves the literal `NUMS` token (T1 pins this); the restored descriptor renders the H-point hex (`50929b74‚Ä¶803ac0`). Flipped tests must NOT assert string equality against the bundle input ‚Äî use golden reconstructed-descriptor strings + golden `bc1p` addresses (v0.49.1 precedent), and note the hex-vs-token rendering in the manual prose.

## Design decisions / risks (R0-worthy)

1. **Classifier restructure (the route).** `taproot_template_and_internal_key` currently returns `Result<(CliTemplate, TaprootInternalKey)>` ‚Äî total: template or refuse. T3 needs a 3-way outcome: `Template(CliTemplate)` (single-leaf `multi_a`/`sortedmulti_a`, byte-identical legacy path) | `GeneralFaithful` (is_nums:true, ‚â§2 leaves) | refuse (is_nums:false unchanged; tree:None unchanged; **NEW** depth‚â•2 with a message citing `upstream-miniscript-taptree-depth2-display-asymmetry`). ¬ß3 then sets `(template_opt=None, tap_internal_key=Some(Nums))` for the general arm so ¬ß4/¬ß5/¬ß7/¬ß8 fall through the EXISTING general-policy machinery (labels, `threshold_field=None`, string-path address derivation) untouched. Risk: the `:710` `_ =>` arm's message disappears/changes ‚Äî T1 deliberately pins it; flipping is the point, but the depth‚â•2 and sortedmulti_a refusal messages must be designed, not inherited.
2. **`ReconstructTranslator` Single pass-through (REQUIRED, probe-proven).** Strict variant recommended: `DescriptorPublicKey::Single` passes through unchanged iff x-only == NUMS H-point; anything else errors (no origin/path/network content to fix on the NUMS const; never promote it to multipath). R0 question: strict-NUMS vs any-Single.
3. **Depth gate shape.** `fn taptree_depth_exceeds_one(inner) -> bool` = "any `Tag::TapTree` child of a `Tag::TapTree`" (md-codec taptrees are strictly binary, so ‚â§2 leaves ‚ü∫ depth ‚â§1 ‚ü∫ no nested TapTree ‚Äî single clean predicate; the `count_tap_leaves` test helper at `parse_descriptor.rs:2621` is the pattern). R0 question: conservative ‚â§2-leaf gate (recommended) vs the tight "no left-branch TapTree" gate the probe would permit.
4. **`sortedmulti_a` inside a 2-leaf tree.** Routes to the faithful arm and fails in md-codec (`:423-428`) ‚Üí surfaced via `faithful_multisig_descriptor`'s `bad(‚Ä¶)` wrap. Decide: accept the wrapped converter message, or pre-gate structurally (walk for `Tag::SortedMultiA` under a `TapTree`) with a message citing `md-codec-sortedmulti-a-to-miniscript-rendering-gap`. Lean: pre-gate ‚Äî the converter message ("must be a tap-leaf root child") is engineer-speak.
5. **`is_nums:false` ‚Äî stay refused this cycle (recommended).** A keyed internal key is reconstructable in principle (the key is in the wallet-policy key list) but needs leaf-membership analysis for the cross-check document + a decision on whether the IK position counts as a "cosigner"; separate concern, already tracked by the `:689` arm + T1 cell. Lean confirmed: NUMS-only.
6. **Belt-and-braces Display-fidelity check.** Optionally assert in `faithful_multisig_descriptor` (or ¬ß5) that `from_str(rendered)` re-Displays equal ‚Äî catches ANY future Display infidelity, not just the known one. Cheap; ¬ß5 already pays the parse. R0 question: add the equality leg or rely on parse-ability alone.
7. **`--format` emission for general tr.** `build_multisig_import_payload` general arm (`template=None`, `restore.rs:744-751`) calls `script_type_from_descriptor` ‚Äî its `Tr(_)` arm (`wallet_export/mod.rs:229+`, multi_a-substring heuristic) classifies general tr as taproot-singlesig vs multisig by substring; verify at spec time which `WalletScriptType` a general tr maps to and that descriptor-driven formats emit sanely (template-requiring formats already refuse via existing branches). `tap_internal_key=Some(Nums)` threads through as for the template arm.
8. **Keypath-only `tree:None` arm (T3 item ii).** Needs a hand-built md-codec wire fixture (bundle can't produce it); independent of the reconstruction route. Can ride along or split ‚Äî recommend split (keep this cycle's blast radius minimal); same for the `:689` "multisig"-wording diagnostics nit (T3 item iii) ‚Äî rewording `:689` flips the T1 pin cell, so do it deliberately if at all.
