## Per-finding verification

### M8 ¬∑ `build-descriptor` accepts a key with an extra derivation suffix ‚Üí silently derives a deeper subtree (genuine wrong-address)
- **WHAT (from bughunt report `design/agent-reports/constellation-bughunt-2026-06-20.md:705-718`):** The renderer unconditionally appends BIP-388 multipath `/<0;1>/*` to every key, but nothing enforces the account-level-xpub contract. A key like `[fp/48h/0h/0h/2h]xpub.../5` is accepted and rendered `xpub.../5/<0;1>/*`, deriving from the `‚Ä¶/5` subtree, not the account level ‚Üí a different (wrong) set of addresses than the user intended. (A trailing `*` is caught; a fixed extra index is not.) class **A-wrong-address**, toolkit, MED (genuine funds-loss-on-legitimate-input).

- **Citations (report `:707-708` cites `descriptor_builder/ir.rs:218-227`, `:22-23`; `gate.rs:347-360`):**
  - `crates/mnemonic-toolkit/src/descriptor_builder/ir.rs:23` ‚Äî `MULTIPATH_SUFFIX = "/<0;1>/*"` const ‚Äî **ACCURATE.** (`:21-23` doc says "appended to every key by the renderer ‚Ä¶ Inputs are account-level `[fp/path]xpub` strings" ‚Äî the documented-but-unenforced contract the report names.)
  - `ir.rs:218-227` ‚Äî the suffix-append site (`fn with_multipath` `:218-220` ‚Üí `format!("{key}{MULTIPATH_SUFFIX}")`; `fn render_keys` `:222-228` maps every key through it) ‚Äî **ACCURATE.** Unconditional literal concat; no normalization/validation.
  - `ir.rs:22-23` account-level contract comment ‚Äî **ACCURATE** (comment lives at `:21-23`, off by ‚â§1; content exact).
  - `gate.rs:347-360` "only an xprv screen exists" ‚Äî **DRIFTED-by-~1, content ACCURATE.** `fn check_secret_key` is at `gate.rs:347`; doc-comment `:338-346`; body `:347-361`. It strips an optional `[origin]` via `key.rsplit(']').next().unwrap_or(key)` (`:348`) then screens ONLY for an extended-private prefix (`:349` `as_bytes().get(1..4) == Some(b"prv")`). Called from `validate_fields` `:235` (Pk/Pkh) and `:240` (Multi/Sortedmulti keys). **No trailing-segment / path-suffix check exists anywhere in `descriptor_builder/`** (grep-verified across `ir/gate/archetype/mod/schema.rs`).

- **STILL-REPRODUCES verdict: YES ‚Äî confirmed structurally end-to-end (not just from the report).**
  1. **Intake (both paths) takes the key verbatim.** Preset path: `archetype.rs` `one_key()` (`:432`) returns `params.keys.first()` unmodified ‚Üí placed into `PolicyNode::Pk(one_key(...).to_string())` (e.g. `:510`) with no stripping. Spec path: the JSON `Pk`/`Multi.keys` string flows straight into the IR; the only step-1 screen (`check_secret_key`) is xprv-only.
  2. **Render appends blindly:** `with_multipath("[fp/...]xpub.../5") = "[fp/...]xpub.../5/<0;1>/*"` (literal concat, `ir.rs:219`).
  3. **Gate step-2 `Descriptor::from_str` ACCEPTS it ‚Äî VERIFIED against the pinned miniscript parser** (`~/.cargo/git/checkouts/rust-miniscript-ce5fa57e8900265e/95fdd1c/src/descriptor/key.rs::parse_xkey_deriv` `:1146`+): the parser `split('/')`s the path and processes segments in order ‚Äî a plain `5` is "just a regular derivation index" (`:1213-1221`), the `<0;1>` token is a one-shot BIP-389 multipath step, and the trailing `*` is the wildcard. `5` BEFORE `<0;1>` is fully legal ‚Üí parses as a valid `MultiXPub`. So the descriptor type-checks and passes `sanity_check` (step-3) ‚Äî **the wrong subtree is silent.**
  4. **"trailing `*` is caught" ‚Äî VERIFIED.** Same parser: any segment AFTER a wildcard returns `InvalidWildcardInDerivationPath` ("'*' may only appear as last element", err variant `:471`/`:490`). So a key already ending in `*` ‚Üí `xpub.../*/<0;1>/*` ‚Üí REJECTED; a fixed index `xpub.../5/<0;1>/*` ‚Üí ACCEPTED. The report's asymmetry claim is correct.
  5. **No existing test** in `tests/cli_build_descriptor.rs` or the `gate.rs`/`ir.rs` `#[cfg(test)]` modules passes a key with a trailing index suffix (all fixtures/constants are origin-only or bare xpubs). Clean baseline for a new RED test.

- **Exact fix-site:** `descriptor_builder/gate.rs::check_secret_key` (`:347`) ‚Äî or a sibling step-1 check called from the same `validate_fields` arms (`:235`, `:240`). After `let key_part = key.rsplit(']').next()‚Ä¶` (which already strips the `[origin]` prefix), reject any `key_part` whose xpub body is followed by extra `/`-segments (i.e. the key carries its own derivation path). The renderer (`ir.rs::with_multipath`) **owns** the `/<0;1>/*` suffix; an account-level key must be bare-`[origin]xpub` with no trailing path. New gate diagnostic kind (e.g. `KeyPathSuffix`), node-addressed, never echoing the key (match the `SecretKey` no-leak discipline). Add the variant to `enum DiagnosticKind` (`gate.rs:90-119`) + its `as_str` arm (`:122-134`) ‚Äî **this enum is ordered by gate-step, NOT alphabetically** (the CLAUDE.md alphabetical rule is `ToolkitError`-only; append as a step-1 check next to `SecretKey`).

- **Decision ‚Äî reject-the-suffix vs honor-it-explicitly:** **REJECT (fail-closed).** Honoring `‚Ä¶/5` would mean the user's account-level contract is silently overridden by an in-key path the builder never documented; the only funds-safe behavior is to refuse and tell the user to supply a bare account-level `[origin]xpub` (the renderer owns the receive/change/index suffix). This matches the report's recommended fix and the program's "emitters/intake must refuse rather than silently transform" discipline.

### L23 ¬∑ `ecies_decrypt_message` panics (not `InvalidScalar`) on a zero privkey (latent public-API robustness)
- **WHAT (report `:814-820`):** `Scalar::from_be_bytes` accepts 0 (‚â§ n‚àí1), so the `InvalidScalar` guard passes; then `mul_tweak` returns `Err(InvalidTweak)` for a zero tweak and hits `.expect(...)` ‚Üí panic instead of a typed `EciesDecryptError::InvalidScalar`. Not CLI-reachable; latent public-API misuse. class **E-panic-dos (latent)**, toolkit, LOW.

- **Citations (report `:816` cites `electrum_crypto.rs:345-351`; `:309-311` safe caller):**
  - `crates/mnemonic-toolkit/src/electrum_crypto.rs:345` ‚Äî `Scalar::from_be_bytes(*privkey).map_err(|_| EciesDecryptError::InvalidScalar)?` ‚Äî **ACCURATE.** `Scalar::from_be_bytes` accepts 0 (it only range-checks `< n`); the zero scalar passes this guard.
  - `electrum_crypto.rs:350-351` ‚Äî `.mul_tweak(&secp, &scalar).expect("valid point √ó nonzero scalar in [1,n-1] is never the identity")` ‚Äî **ACCURATE (report's `:345-351` window covers it; the `.expect` is precisely `:351`).** A zero tweak ‚Üí `mul_tweak` `Err(InvalidTweak)` ‚Üí `.expect` panics rather than returning a typed error.
  - `:309-311` "sole in-tree caller is safe" ‚Äî **ACCURATE.** `derive_storage_eckey` (`:298`) reduces mod n then guards `if scalar.iter().all(|&b| b == 0) { return Err(InvalidScalar) }` at `:309-310`, so it never hands a zero scalar to `ecies_decrypt_message`.
  - **CLI-reachability ‚Äî VERIFIED latent.** The only CLI path is `cmd/import_wallet.rs:468` ‚Üí `ecies_decrypt_storage` (`:384`/`:392-393`) ‚Üí `derive_storage_eckey(password)` (rejects zero) ‚Üí `ecies_decrypt_message(blob, &privkey)`. The privkey is always reduced-and-nonzero ‚áí the panic is unreachable from the CLI today. It is a robustness item only if a future caller (or a downstream lib consumer) passes a raw zero privkey.

- **Exact fix-site:** `electrum_crypto.rs` `ecies_decrypt_message` (`pub fn` at `:321`), at the scalar check (`:345`). Either (a) add a zero-scalar reject before `mul_tweak` (`if privkey.iter().all(|&b| b==0) { return Err(EciesDecryptError::InvalidScalar) }`), or (b) `map_err` the `mul_tweak` result to `EciesDecryptError::InvalidScalar` instead of `.expect(...)`. (a) is cleaner / more explicit; (b) is the minimal diff. Either way: no new error variant (`InvalidScalar` already exists), pure internal robustness.

- **Decision ‚Äî folds in:** **YES, fold L23 into the cycle-7 branch.** Same file zone as M8's review (`descriptor_builder/` + `electrum_crypto.rs` = WS-DESCBUILD per the program), trivial (‚â§5-line typed-error swap + one unit test), no wire/CLI change. The program explicitly groups it (`design/PLAN_constellation_bughunt_fix_program.md:429-430`: "WS-DESCBUILD ‚Ä¶ M8 (formal), L23 (trivial)").
