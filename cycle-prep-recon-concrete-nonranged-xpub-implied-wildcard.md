## Per-slug verification
### `concrete-nonranged-xpub-implied-wildcard`

- **WHAT (from FOLLOWUPS.md, lines 39-42):** A concrete non-ranged descriptor `wpkh([fp/84'/0'/0']xpub)` (no `/*`) substitutes to bare `@0[fp/..]`, then md-codec's `wildcard_for` always emits a wildcard → the card silently becomes ranged `xpub/*` (a different, non-standard wallet). Claimed indistinguishable from a legitimate keyless template at `lex_placeholders`, so must be fixed upstream at the substitution layer.

- **Citations:**

  - `md-codec crates/md-codec/src/to_miniscript.rs:133` `fn wildcard_for` — **ACCURATE.** Current source (`origin/main@ef1f3e71`):
    ```rust
    /// `Wildcard::Hardened` for a `/*h` use-site, else `Wildcard::Unhardened`.
    /// Hardened wildcards are pre-refused at the derivation boundary
    /// ([`has_hardened_use_site`]); this only governs the rendered text.
    fn wildcard_for(use_site: &UseSitePath) -> Wildcard {
        if use_site.wildcard_hardened {
            Wildcard::Hardened
        } else {
            Wildcard::Unhardened
        }
    }
    ```
    Confirmed: this function has exactly two possible return values, both wildcard-bearing. There is no third arm. It cannot return `Wildcard::None` given ANY `UseSitePath` value, because...

  - `md-codec crates/md-codec/src/use_site_path.rs:49-54` `struct UseSitePath` — **STRUCTURAL ROOT CAUSE, confirmed by reading (not cited by the FOLLOWUP but load-bearing for the "cannot be fixed downstream" claim):**
    ```rust
    pub struct UseSitePath {
        pub multipath: Option<Vec<Alternative>>,
        pub wildcard_hardened: bool,
    }
    ```
    There is **no field anywhere in `UseSitePath` capable of representing "no wildcard at all."** `multipath: None` means "single-path," not "no wildcard" — a single-path *wildcard* descriptor (bare `/*`) and a genuinely fixed non-ranged key both would collapse to `multipath: None`. This is a **wire-format-level** limitation, not merely a bug in `wildcard_for`. Cross-checked against upstream `rust-miniscript`'s own `Wildcard` enum (`vendor/miniscript/src/descriptor/key.rs:195-202`, present in **both** repos' vendored copies), which DOES have a `None` variant — so the gap is m-format's own choice not to plumb it through `UseSitePath`, not a rust-miniscript limitation. `design/SPEC_cycleA_descriptor_use_site_collapse.md:226` (toolkit, primary source, R0-reviewed) makes this an explicit, already-ruled-on design decision: *"First-class fixed-step support in md1/`UseSitePath` — explicitly NOT pursued (md1 is multipath-centric)."* This confirms the fix must NOT touch md-codec's wire format — it must reject upstream in the toolkit before a fixed-key descriptor is ever encoded.

  - toolkit `crates/mnemonic-toolkit/src/wallet_import/pipeline.rs:330-400` `fn concrete_keys_to_placeholders` — **ACCURATE, with one framing refinement.** Read in full. `key_regex()` (`pipeline.rs:38-42`) matches only the `[fp/path]xpub…` span (`m.end()` lands immediately after the xpub base58 string) — it does **not** consume any trailing `/*`, `/<a;b>/*`, or absence thereof. The substitution (`pipeline.rs:376-381`) replaces `[fp/path]xpub` with `@N[fp/path]` and otherwise copies the surrounding descriptor text **verbatim**, including whatever wildcard suffix (or lack of one) followed the original xpub in the source string. So the wildcard-presence **text** is not, strictly, destroyed at this function — a following `/*` (if any) survives unchanged into `placeholder_form` and IS visible to `lex_placeholders`'s `wild` capture group downstream.
    - **What actually is lost** (a more precise statement than the FOLLOWUP's phrasing, confirmed by reading `lex_placeholders`/`make_use_site_path` below): the signal that gets discarded is not "was there a `/*` in the text" (that survives) but **"was this `@N[fp/path]` slot produced by substituting a REAL concrete xpub, or was it hand-typed directly by the caller as a keyless-template annotation"** — because `bundle --descriptor` also accepts literal `@N[fp/path]` syntax typed directly (the "AtN direct-lex" entry point cited in `design/FOLLOWUPS.md`'s `bip388-double-star-shorthand-support` entry: `bundle:1395`), and once `concrete_keys_to_placeholders`'s output text is handed to the shared `lex_placeholders`, both origins are byte-identical and genuinely indistinguishable. This is a MINOR wording nuance, not a structural error — the FOLLOWUP's conclusion (fix must live in `concrete_keys_to_placeholders`, before the ambiguity is created) is exactly right, because at THAT point the caller still knows for certain "this bracket came from a real xpub, not from hand-typed text," which is precisely the fact `lex_placeholders` can no longer recover.
    - Bonus scope-relevant finding: `descriptor_concrete_to_resolved_slots` (`pipeline.rs:406+`, the `verify-bundle --descriptor` concrete-descriptor fork) calls the **same** `concrete_keys_to_placeholders` (`pipeline.rs:411`). A single fix at the substitution layer therefore closes **both** the `bundle`-encode silent-acceptance path and the `verify-bundle` false-pass path in one choke point.

  - toolkit `crates/mnemonic-toolkit/src/parse_descriptor.rs:61-232` `fn lex_placeholders` (regex `:98-100`, residue-floor check `:206-217`, `fn make_use_site_path` `:325-338`) — **ACCURATE.** Confirmed a bare `@N[fp/path]` (no `wild`, no `mpath` captured) lex-passes cleanly — the residue-floor check added in Cycle A only rejects *non-terminator* residue after the optional groups; an immediate terminator (`)`, `,`, `}`, EOF, whitespace) is accepted by design, because that is also the shape of the canonical bare-`@N` keyless-template form. `make_use_site_path` (`:325-338`) builds `UseSitePath { multipath: alts.is_empty() ? None : Some(alts), wildcard_hardened: occ.wildcard_hardened }` — reading `occ.wildcard_hardened` (a bool, default `false` when no `wild` matched at all) with **no separate field for "wild present."** So `@0[fp/path]` (no wild) and `@0[fp/path]/*` (unhardened wild) build **byte-identical** `UseSitePath` values. Test `lex_residue_floor_accepts_bare_at_n_d1_deferred` (`parse_descriptor.rs:1905-1914`) explicitly pins bare `@0` as required-to-lex-pass, with an inline comment naming this exact FOLLOWUP slug as the deferred item. `design/SPEC_cycleA_descriptor_use_site_collapse.md:79-85` (D1, R0-confirmed-sound) is the primary-source design record backing the "cannot fix at lexer" ruling — quoted: *"A blanket bare-`@N` reject would break the canonical multisig keyless-template form… The genuine sub-concern… is INDISTINGUISHABLE from a keyless template at the lexer… and must be handled upstream."*

  - **Funds claim — EMPIRICALLY CONFIRMED** (built `cargo build --release --bin mnemonic` at `e092f679`, v0.78.0; full round-trip driven against the real binary, not merely traced):

6. **A related, already-fixed sibling case for calibration:** the truly-fixed-but-non-bare case (`wpkh([fp/path]xpub/0)`, a concrete key with a fixed derivation step but NO trailing wildcard) is **already caught** by the Cycle-A residue-reject floor in `lex_placeholders` (`parse_descriptor.rs:206-217`), because the `/0` residue after `@N[fp/path]` fails the terminator check. This narrows the fix surface further: the new check only needs to fire when `concrete_keys_to_placeholders` sees a concrete-key match immediately followed (in the ORIGINAL descriptor text) by a terminator/EOF, with no `/` at all.
