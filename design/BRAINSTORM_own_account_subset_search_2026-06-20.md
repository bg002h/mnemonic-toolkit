## 1. Problem
v0.60.0 requires the supplied keys to EXACTLY fill the N slots (`pool.len() == n`); the engine enumerates `n!` orderings via `unrank_permutation` over the first `n` pool entries. `--own-account-max K` (the operator over-supplying own-account candidates because they don't recall which account index the wallet uses) is wired but **refuses** — the engine can't select a *subset* of an over-sized pool, and any pool index `≥ n` is never evaluated (the P3a I-1 finding). This makes a documented recovery feature non-functional for the very-common "which account was it?" case.

## 2. Decisions (the agreed model — to be R0'd)

### D1 — Own-only subset-search by DEFAULT; opt-in + BOUNDED unowned (cosigner) search.
Two tiers, safe-by-default:
- **Default (own-only):** only the OWN seed may be over-supplied. The operator supplies own-account candidates via the existing `--account <list>` OR re-enabled `--own-account-max K` (own seed derived at accounts `0..K-1` → K own candidates), and **ALL `M` cosigner mk1 cards EXACTLY** (no extra). The own-slot-count `j = N − M` is then DETERMINED. The search selects which `j` of the `K` own candidates fill the `j` own slots, and assigns all `N` keys to the `N` slots (positions of own-vs-cosigner slots are themselves unknown). Bounded: `K` is a small account range; `M` exact ⇒ the space stays modest.
- **Opt-in (unowned/cosigner search):** a new flag ENABLES over-supplying/uncertain cosigner candidates too ("I'm not sure which of these mk1 cards belong / how many cosigners there are"). This makes `j` uncertain and grows the space toward `P(pool, N)`. It MUST be explicitly enabled AND BOUNDED (a count cap on extra unowned candidates, on top of the adaptive time-cap), so it can never silently blow up. Default-off.

### D2 — Enumeration = subset-select k-permutation, OWN-ANCHORED. The load-bearing count is DERIVED here (R0-r1 I-1).
The engine gains a subset-select enumeration (generalize the `n!`-over-`(0..n)` `unrank_permutation(rank, n)` → a pool-aware `unrank_kperm`). **Notation (R0-r1 M-1 — disambiguate `K`):** `N` = slots; `M` = cosigner cards supplied (own-only: EXACT, all used); `j = N − M` = own-slot-count (own-only: DETERMINED); `K_own` = number of OWN-account candidates the operator supplies (the `--account` list length, or `--own-account-max K_own` ⇒ own seed at accounts `0..K_own−1`). Requires `K_own ≥ j` and `K_own + M ≥ N` (= `K_own ≥ j`, same thing). The pinned `prefix_ladder_own_account_max_subset_space` test (`permutation_search.rs:741`) and #28 SPEC §6.1 use `K` = EXTRA accounts beyond `j` (pool `= (N−own)+K`); this doc's `K_own` = total own candidates = `j + extra`. State the relation when updating them.

**Own-anchored enumerated count — DERIVED (two independent methods, equal):**
> Own-only premise: exactly `j = N−M` of the `K_own` own candidates are used, ALL `M` cosigners are used. (a) choose which `j` own candidates: `C(K_own, j)`; (b) the chosen `j` own + the `M` cosigners = exactly `N` distinct keys; assign them bijectively to the `N` ordered slots: `N!`. → **`S_own = C(K_own, j) · N!`**, `j = N−M`.
> Cross-check (place-directly): place `M` cosigners into `N` slots `P(N,M)=N!/j!`, then `j` own from `K_own` into the remaining `j` slots `P(K_own,j)=K_own!/(K_own−j)!` → `N!/j! · K_own!/(K_own−j)! = C(K_own,j)·N!`. ✓ equal.
> Backward-compat: at `K_own = j` (no over-supply), `C(j,j)·N! = N!` — byte-identical to v0.60.0's `perm_count_u128(n,n)` (`restore.rs:1661`). ✓

**This REFINES the canonical `realized_s = P(pool, N)` DOWN to `S_own`** (strictly `≤ P(pool,N)`: it forbids the cosigner-DROPPING placements — using `<M` cosigners + `>j` own — that `P(K_own+M, N)` includes but the own-only premise excludes). It therefore **SUPERSEDES** the four sources that pin `P(pool,N)`: the pinned test `prefix_ladder_own_account_max_subset_space` (`permutation_search.rs:739-755` — must be UPDATED to `S_own = C(K_own,j)·N!`); the FOLLOWUP "Fix" line (`FOLLOWUPS.md:48`); #28 SPEC §6.1 (`:110`); the P3a I-1-review option-1 (`template-multisig-p3a-…-review.md:64`). Each is re-cited in the SPEC.

**The enumerated≡counted invariant (funds-safety FLOOR — the I-1 danger direction):** sizing the prefix to the SMALLER `S_own` is safe ONLY IF the enumerator scans EXACTLY the `S_own` set — every assignment using exactly `j` own + all `M` cosigners, each once, and NOTHING ELSE. A scanned-but-uncounted cosigner-dropping placement would be a collision the prefix wasn't sized for. So `unrank_kperm` (or the own-anchored generator) MUST be proven to enumerate the `S_own` set bijectively, verified by an exhaustive small-N brute-force-reference test (engine's enumerated set ≡ the independently-generated valid-assignment set). `S_own` is closed-form (`C(K_own,j)·N!`) ⇒ computable UP-FRONT for the cap ETA + `validate_prefix_strength`, satisfying D4.

### D3 — Ambiguity full-scan PRESERVED for prefix id-search; safe early-exit for address-search.
The funds-safety gate is unchanged: **prefix id-search MUST scan the full realized space to certify uniqueness** (≥2 matching assignments → `Ambiguous` → refuse, never silently pick one). Ordering ("own first") does NOT shortcut this — it's the time-cap that bounds prefix id-search. **Address-search is collision-free** (full scriptPubKey ⇒ a match is provably unique), so own-anchored ordering + early-exit on first match is SAFE there and makes the common case fast. (Optional: a *full* — non-prefix — wallet-id is also collision-safe for early-exit; SPEC to decide.)

### D4 — Prefix-strength sizing to the REALIZED space; address-search recommended for large pools.
`realized_s` = the ACTUAL enumerated count: own-only ⇒ `S_own = C(K_own,j)·N!` (D2); opt-in ⇒ the bounded count for that mode (I-2/D6) — NEVER `n!` under over-supply, NEVER the looser `P(pool,N)` for own-only (D2). `validate_prefix_strength` (`restore.rs:1700`) sizes the required `--expect-wallet-id` prefix to `realized_s` via the #28 `ceil((log2(S)+32)/8)` ladder (collision-prob `≤ ~2e-10`). **Worked margin (R0-r1 I-2), `N=11, j=4, M=7`, K_own=32 own candidates:** `S_own = C(32,4)·11! = 35,960 · 39,916,800 ≈ 1.435e12`; `log2 ≈ 40.4`; `ceil((40.4+32)/8) = ceil(9.05) = 10` bytes — a 10-byte (20-hex) `--expect-wallet-id` keeps the margin. (The looser uniform `P(pool,N) = P(K_own+M, N) = P(39,11)` the pinned ladder would demand is strictly larger ⇒ over-sizes the prefix; `S_own` is the faithful, tighter sizing.) A short prefix over a large pool REFUSES (point the operator at a longer id or address-search). Address-search stays collision-free regardless of pool size (I-2) ⇒ the recommended mode for large pools.

### D6 — Unknown own-slot-count handling.
- Own-only mode: `j = N − M` is DETERMINED (cosigners exact) → no declaration needed.
- Opt-in mode: `j` uncertain (extra cosigners) → the search tries the valid `j` range, bounded; OR the operator may pin `j`/own-count. SPEC to decide the inference vs declaration (the prune is tighter if `j` is pinned).

## 3. Funds-safety floors (carried from #28 + new)
- **Distinct-keys** (`reject_duplicate_keys` on the WHOLE pool, `restore.rs:1648`, before `realized_s`/search) — unchanged mechanism, but now LOAD-BEARING for the NEW collision axis (R0-r1 I-2): over-supply adds distinct injective SUBSETS (not just distinct orderings of a fixed set), and address-search collision-freeness over the over-supplied pool RESTS on this floor — two distinct subsets have distinct key SETS ⇒ distinct scriptPubKey ONLY because no two pool keys are byte-identical (`pool_key_blobs` compares the 65-byte `key65`, so an own-derived key equal to a cosigner card is caught regardless of origin). State this dependency in the SPEC.
- **Own-anchor enumerate≡count correctness:** the constrained enumeration MUST cover EXACTLY the `S_own = C(K_own,j)·N!` set (every assignment using exactly `j` own + all `M` cosigners, each once, NOTHING else). Skip a valid assignment ⇒ legitimate wallet NO-MATCHES (the I-1 class); scan an UNcounted (cosigner-dropping) placement ⇒ a collision the prefix wasn't sized for (the dangerous direction). The k-perm unranking + own-anchor generator are the R0-load-bearing combinatorics (off-by-one = funds-safety bug) → exhaustive small-N brute-force-reference test (engine's enumerated set ≡ the independently-generated valid set) + `realized_s == enumerated count` assertion.
- **Ambiguity** (≥2) + **no-match** → refuse (D3). id-search full-scan ambiguity certification is over the OWN-ANCHORED enumerated set (the same set `realized_s` counts) — closes the I-1↔I-2 coupling.
- **Adaptive time-cap** (the ~1hr ceiling + `--accept-search-time`) — the universal blow-up bound; the bounded opt-in count-cap is an ADDITIONAL up-front bound; AND a hard `K_own`/pool ceiling (open-point 6) refuses absurd inputs BEFORE calibration (`perm_count_u128`/`factorial` already REFUSE on `u128` overflow, `restore.rs:1943` [RELAUNCH]/`permutation_search.rs:481` → `bad(...)`, never panic — #28 M1, confirmed live).
- **Prefix-strength sized to `realized_s` = `S_own`** (D4).
- **Per-slot origins BUILT fresh from supplied keys; carried origin never loaded (C1)** — unchanged from #28. (RELAUNCH: "built fresh" now subsumes the L8 network coin-type substitution — see SPEC §5.)
- **Secret-hygiene of the over-supply own-key derivation = a CONSUMED dependency, NOT in scope (RELAUNCH).** Deriving `K_own` own candidates materializes more transient own private material; the design DECOUPLES that hygiene treatment out of this cycle — the over-supply loop consumes `derive_account_xpub_only(...)` so it holds only PUBLIC xpubs and never owns an `Xpriv`. Authoring the move-only `ScrubbedXpriv` RAII scrub backing it is tracked as FOLLOWUP `derive-slot-account-xpriv-scrub-confinement` (filed, decoupled on architect advice), NOT authored here. (SPEC §4.x / plan P2 state the dependency.)

### 3a. Own-anchor premise-violation failure modes — all fail SAFE (R0-r1 I-3)
Own-only mode's prune assumes the operator (i) owns EXACTLY `j = N−M` slots via a SINGLE own seed (multiple accounts), and (ii) supplied ALL `M` cosigners (no more, no fewer). Every violation must fail SAFE (refuse / NO-MATCH), never silent-wrong:
| Violation | Outcome | Why safe |
|---|---|---|
| Under-supplies cosigners (`M' < M`) | NO-MATCH → refuse | own-anchored search forces `j_assumed = N−M' > j_true` + `M'` cosigners; the real assignment needs `M` cosigner keys it lacks → unreachable. Message: "did you supply ALL cosigner cards?" |
| Over-supplies cosigners in own-only (`M' > M`) | **REFUSE up front** | own-only REQUIRES exact cosigners → refuse with "own-only mode needs exact cosigner cards; use `<opt-in flag>` for uncertain cosigners." (Without this gate the prune silently mis-fires.) |
| Supplies own key ALSO as a cosigner card | refuse (distinct-keys floor) | `key65` byte-dup caught before search, regardless of origin/account. |
| Owns >`j` slots via MULTIPLE independent seeds | out of own-only scope | own-only = ONE own seed (multi-account); a multi-SEED operator uses explicit `--account`/`--cosigner @N=` or the opt-in tier. State the scope. |
