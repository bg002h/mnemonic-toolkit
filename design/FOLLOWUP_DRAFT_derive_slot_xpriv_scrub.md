### `derive-slot-account-xpriv-scrub-confinement` — `derive_bip32_from_entropy_at_path` discards the account `Xpriv` un-scrubbed; confine it behind a move-only RAII newtype + public-only entrypoint

- **Surfaced:** 2026-06-21, own-account subset-search re-brainstorm recon (§3.3) + a follow-on architect pass over the secret-keymat sweep. The 2026-06-21 sweep (`design/agent-reports/sweep-keymat-toolkit.md`, findings 1-6) MISSED this site — it is a previously-uncounted sibling of the same "derived-output secret not zeroized" class. Verified vs `origin/master` (re-grep `derive_slot.rs` / `derive.rs` line numbers at write time).
- **Secret type / gap class:** account-level `bitcoin::bip32::Xpriv` (full spending authority over the account branch). Gap class 1 (zeroize-on-drop gap) — an owned `Xpriv` held in a bare struct field.
- **Where:** `crates/mnemonic-toolkit/src/derive_slot.rs` `derive_bip32_from_entropy_at_path` (`:65`) — `account_xpriv = master.derive_priv(&secp, path)` (`:84-85`), `account_xpub = Xpub::from_priv(&secp, &account_xpriv)` (`:87`), then BOTH stored into the returned `DerivedAccount`. `crates/mnemonic-toolkit/src/derive.rs` `pub struct DerivedAccount` (`:23`) — `pub entropy: Zeroizing<Vec<u8>>` (`:24`) but `pub account_xpriv: Xpriv` (`:27`) is a bare, un-scrubbed field that drops without erase. Any caller needing only the public `account_xpub` (the subset-search over-supply loop derives K_own such keys) holds the secret `Xpriv` for no reason.
- **What:** the seed buffer is `Zeroizing<[u8;64]>` and the entropy is `Zeroizing<Vec<u8>>`, but the derived account `Xpriv` — equal spending authority — sits in a bare struct field. Callers that need only the public xpub hold the secret `Xpriv` un-scrubbed.
- **Fix (architect-validated; PROVISIONAL — re-verify the `derive_slot.rs` substrate post-burndown, it is an active hygiene area):**
  - `struct ScrubbedXpriv(Xpriv)` — NOT `Copy`/`Clone` (move-only; `Copy`+`Drop` mutually exclusive, E0184); `impl Drop` → `self.0.private_key.non_secure_erase()` + a VOLATILE `chain_code` zero-write; no escape hatch (only `&self` accessors + `xpub(&self, &secp) -> Xpub`; NO `into_inner` / `Deref<Target=Xpriv>` / `Clone` / `Copy` / pub field); compile-time `assert_not_impl_any!(ScrubbedXpriv: Copy, Clone)`.
  - `derive_account_xpub_only(...) -> Result<(Xpub, Fingerprint), ToolkitError>` in `derive_slot.rs` — confines the `Xpriv` inside the fn (wrap in `ScrubbedXpriv`, read xpub, scrub on drop) so callers never own an `Xpriv`.
- **Caveat:** the by-value `derive_priv() -> Xpriv` return temp + secp internals remain under the open FOLLOWUP `rust-bitcoin-xpriv-zeroize-upstream`; `non_secure_erase` is best-effort (its own crate docs note the compiler may have pre-copied). The newtype MINIMIZES (one move-in per derive) but cannot eliminate this.
- **Severity:** MED (defense-in-depth; full account spending authority held un-scrubbed in a struct field). Not a leak-to-disk/argv/log → not Critical.
