## 1. Goal

Add a verified, decrypt-only implementation of Electrum's **BIE1 (user-password) whole-file storage encryption** to the `electrum_crypto.rs` library, so a future Phase B can wire it into `import-wallet`. BIE2 (hardware-device/xpub) is permanently out of scope (no password decrypt path).

## 2. Verified crypto (primary-source, pinned SHA above)

`WalletStorage.decrypt(password)` = `zlib.decompress(crypto.ecies_decrypt_message(get_eckey_from_password(password), raw))`.

**`get_eckey_from_password(password)`** (`storage.py:193-197`):
`secret = PBKDF2-HMAC-SHA512(password.utf8, salt=b"", iterations=1024, dklen=64)`; privkey scalar = `int.from_bytes(secret,'big') mod n` (secp256k1 order); reject scalar==0. **NOT sha256d; salt is EMPTY.**

**`ecies_decrypt_message(privkey, blob)`** (`crypto.py:470-494`), magic `b"BIE1"`:
1. `raw = base64decode(blob)`; require `len(raw) >= 85`.
2. `magic = raw[0:4]` (must == `b"BIE1"`; `b"BIE2"` ⇒ unsupported), `ephemeral_pubkey = raw[4:37]` (33B compressed), `ciphertext = raw[37:-32]`, `mac = raw[-32:]` (HMAC-SHA256, 32B).
3. `ecdh = (ephemeral_pubkey × privkey_scalar)` serialized **compressed (33B)**; `key = sha512(ecdh)`; `iv, key_e, key_m = key[0:16], key[16:32], key[32:64]`.
4. verify `HMAC-SHA256(key_m, raw[:-32]) == mac` (this is the wrong-password detector — verify BEFORE decrypt to avoid a PKCS7 padding oracle).
5. `plaintext = AES-128-CBC-decrypt(key_e, iv, ciphertext)` with PKCS7. (**AES-128**, not the AES-256 used elsewhere in the crate.)

Storage adds `zlib.decompress(plaintext)` → wallet JSON.

## 4. Dependencies

**New (2):**
- `flate2` (zlib `decompress`; default features = pure-Rust `miniz_oxide` backend, no system zlib).
- `crypto-bigint` for the 64→32 mod-n reduction. **R0-confirmed** over `num-bigint` (constant-time over secret-key material, RustCrypto family) and over `k256` (rejected — would duplicate secp256k1 in-tree). Reduction (R0 M1): let `N64 = N zero-LEFT-padded to 64 bytes`; `r = U512::from_be_slice(&pbkdf2_64) % NonZero::new(U512::from_be_slice(&N64)).unwrap()`; scalar = `r.to_be_bytes()[32..64]` (low 32 BE bytes). KAT target = the pw123 scalar above.

**Already in tree (no new feature needed — R0 I4):** `sha2::Sha512` (used at `bsms_crypto.rs:51`), `aes::Aes128` (the `aes = "0.8"` crate exposes it alongside the `Aes256` used today), `cbc`, `hmac`, `pbkdf2`, `zeroize`, `base64`, `hex`, and secp256k1 via the `bitcoin` re-export (see §5).

## 5. API design (extend `crates/mnemonic-toolkit/src/electrum_crypto.rs`)

New library-local error (hand-rolled `Display`/`Error`, alphabetical variants per CLAUDE.md):
```rust
pub enum EciesDecryptError {
    AesDecryptFailure,            // PKCS7 unpad refused (post-MAC; rare)
    Base64DecodeFailure(String),
    Bie2Unsupported,              // magic == b"BIE2"
    HmacMismatch,                 // wrong password / tampered (primary detector)
    InvalidEphemeralPubkey,       // raw[4:37] not a valid point
    InvalidMagic([u8; 4]),        // neither BIE1 nor BIE2
    InvalidScalar,                // mod-n reduction == 0 (unreachable in practice)
    TooShort { got: usize },      // len < 85
    ZlibDecompressFailure(String),
}
```
Layered, independently-testable functions (mirror `bsms_crypto.rs` helper style):
```rust
pub fn derive_storage_eckey(password: &[u8]) -> Result<Zeroizing<[u8; 32]>, EciesDecryptError>;
//   PBKDF2-HMAC-SHA512(password, b"", 1024, 64) -> int % n -> 32 BE bytes; Err(InvalidScalar) if 0.

pub fn ecies_decrypt_message(blob_b64: &str, privkey: &[u8; 32])
    -> Result<Zeroizing<Vec<u8>>, EciesDecryptError>;
//   The BIE1 envelope (steps 1-5 of §2). Validated by the 3 Electrum KATs.

pub fn ecies_decrypt_storage(blob_b64: &str, password: &[u8])
    -> Result<Zeroizing<Vec<u8>>, EciesDecryptError>;
//   derive_storage_eckey -> ecies_decrypt_message -> zlib.decompress. The Phase-B entry point.
```
Notes:
- `decrypt_field`/`derive_key` (Format A, AES-256/sha256d) are untouched and NOT reused. New AES-128 alias `type Aes128CbcDec = cbc::Decryptor<aes::Aes128>;` alongside the existing AES-256 alias.
- **EC access path (R0 I2):** `secp256k1` is NOT a direct dependency — use the `bitcoin` 0.32.8 re-export: `use bitcoin::secp256k1::{PublicKey, Scalar, Secp256k1};` (this resolves to `secp256k1` 0.29.1). Writing `secp256k1::…` at crate root will not compile.
- **Point-mul:** `PublicKey::from_slice(&raw[4..37])` (Err ⇒ `InvalidEphemeralPubkey`), then `pk.mul_tweak(&secp, &Scalar::from_be_bytes(scalar)?)`, then `.serialize()` (compressed 33B SEC). `Secp256k1::new()` context.
- **`mul_tweak` cannot fail here (R0 I3):** the scalar is guaranteed in `[1, n-1]` (the `derive_storage_eckey` `InvalidScalar`-on-zero guard runs first; reduction guarantees `< n`), and a valid (non-infinity) order-`n` point times a nonzero scalar `< n` is never the point at infinity. So `.expect("valid point × nonzero scalar in [1,n-1] is never the identity")` with that proof-comment. `mul_tweak`'s Err is therefore NOT mapped to `InvalidEphemeralPubkey` (that variant is reserved for `PublicKey::from_slice` failure).
- **Error unification deferred to Phase B (R0 M2):** in Phase A all variants stay DISTINCT (KAT cells assert on them). The `HmacMismatch | AesDecryptFailure` → single non-leaky "decryption failed" merge is a Phase-B CLI-boundary concern (mirror `cmd/electrum_decrypt.rs`); a pure library returning distinct variants is not an oracle.

## 7. Secret hygiene

`derive_storage_eckey` returns `Zeroizing<[u8;32]>`. `Zeroizing` also on the intermediates that touch secret/derived material (R0 M3): the 64-byte PBKDF2 output, the `crypto-bigint` reduction output, the compressed `ecdh` point bytes, the `key`/`key_e`/`key_m` split, and the decrypted output `Zeroizing<Vec<u8>>` (whole-file plaintext can carry seed/xprv). Password slice borrowed (caller owns/zeroizes — Phase B does the mlock/pin at the CLI boundary, mirroring `cmd/electrum_decrypt.rs`). No `mlock` inside the pure-library functions (consistent with `bsms_crypto`/`electrum_crypto` today).

- **BIE2** (`crypto.py`/`storage.py` XPUB_PASSWORD): detected (`Bie2Unsupported`) but never decryptable from a password. Phase B emits the clear refusal advisory.
- **CLI / sniff / orchestrator / `flag_is_secret` / GUI schema-mirror / manual** — all Phase B.
- **No version bump / tag** — Phase A is a pre-tag internal-library commit on master.

## 9. Risks (carried from architect §10)

mod-n reduction correctness (KAT-pinned to pw123 scalar); empty-salt + 1024-iter PBKDF2 (distinct from BSMS c=2048 / Format A sha256d — don't copy-paste a KDF); AES-128 not 256; HMAC-before-decrypt ordering; compressed (33B) point in KDF; exact slice offsets / len≥85; crypto-citation decay (re-grepped at pinned SHA above).
