## M2 — SLIP-39 `combine` digest-error message falsely implies passphrase verification
**Gap:** `cmd/slip39.rs:757` — `DigestVerificationFailed => "…wrong --passphrase OR a share was substituted"`. SLIP-0039's digest runs on the Feistel-encrypted master secret **before** passphrase decryption, so this error can NEVER be caused by a wrong passphrase; worse, it implies the *absence* of this error verifies the passphrase (it does not — a mistyped passphrase exits 0 and prints a different valid-looking master). Funds risk: a user could retire the real backup.
**Fix:** rewrite the message to state the digest check catches only share substitution/corruption/wrong-secret, and explicitly does NOT verify `--passphrase` (this check runs pre-decryption). Keep the unconditional "verify the recovered wallet's expected derived address before trusting" note (`:683`, unaffected).
**Loci + lockstep (same PR):** `cmd/slip39.rs:757` (message); `tests/cli_slip39_refusals.rs:336` (asserts the exact string — update in lockstep); `docs/manual/src/40-cli-reference/41-mnemonic.md:2194` (mirrors the string — manual-mirror invariant). No GUI/schema hit (grep of mnemonic-gui/src + docs/manual-gui for the string = 0).
**SemVer:** PATCH-class (rides the MINOR).

The refusal is a factual error — `language.rs` already maps `Portuguese => 9` in three other tables (ms1 wire-language). Fail-closed today (exit 1, no wrong output) but blocks recovery of a spec-conformant child wallet.
**Fix:** replace the `Portuguese => Err(...)` arm (`:353-357`) with `CliLanguage::Portuguese => (9, bip39::Language::Portuguese),`; add `| 9 | Portuguese |` to the doc-comment table (`:325-341`, currently stops at 8/Czech + states "Portuguese … refused").
**Oracle (no official BIP-85 non-English vector exists):** flip `tests/cli_derive_child.rs:535-560` `bip39_portuguese_refused_no_bip85_code` from a refusal assertion to a positive **divergence** test mirroring `bip39_japanese_diverges_from_english` (`:492-533`): derive English vs Portuguese at the same master+index → assert outputs differ + Portuguese output is valid BIP-39 Portuguese (wordlist membership). Same weak-oracle strength as the other 6 non-English languages. **R0 m4:** `bip39` is ALREADY a runtime `[dependencies]` entry (`Cargo.toml:52`, `all-languages`) and tests already `use bip39` (`cli_derive_child.rs:390`) → **NO dep change needed** (do NOT add a dev-dep — it would duplicate). ## M4 — BSMS Round-1 lenient-mode returns exit 0 even when a signature verify FAILED
**Gap:** `cmd/import_wallet.rs:418` `return Ok(0)` inside standalone Round-1 mode (`args.blob.is_none()`) fires regardless of whether any `Round1Verification.status` is `Failed`. Lenient mode (`verify_bsms_round1_files` `:2514-2639`) pushes `Failed{reason}` + a stderr NOTICE (`:2621-2626`) but still returns `Ok(...)` — inverting the `verify && proceed` contract for `$?`-gated automation.
**Fix (user decision = Option A):** compute `any(status == Failed)` over `round1_verifications`; if any failed, return a **distinct nonzero exit = 4** (the established VERIFY-ME idiom) instead of `Ok(0)`; still print the full per-record report first (lenient preserved — no abort, unlike strict's exit-2). Strict mode (`--bsms-verify-strict`, typed `BsmsSignatureMismatch`, exit 2, aborts pre-output) UNCHANGED.
**exit-4 non-collision RESOLVED (R0 m2):** `import-wallet` already uses exit 4 (`ToolkitError::ImportWalletSeedMismatch`, from `wallet_import/overlay.rs:207`) but that is **blob-mode-only** (overlay runs after the standalone early-return → mode-disjoint) AND the same semantic class (toolkit-wide exit 4 = uniform "mismatch/VERIFY-ME — do not trust", per `BundleMismatch`/`RestoreMismatch`/repair-candidate, documented `41-mnemonic.md:751-753`). Exit 2=strict abort, 3=`FutureFormat`, 5=`REPAIR_APPLIED` all taken → **4 is correct, non-colliding.**
**BOTH modes (R0 I1):** the user decision "nonzero on ANY Failed" has no mode qualifier. Apply the `any(Failed)`→exit-4 rule at BOTH (a) the standalone-mode early return `cmd/import_wallet.rs:404-419` AND (b) the combined `--blob`+`--bsms-round1` path's tail return `:1363` (`round1_verifications` with `Failed` entries flow into the `--json` envelope but the invocation exits 0 today — same `$?`-inversion, arguably worse since import proceeds + synthesizes cards). ~3 lines each.
**Loci + lockstep (same PR):** `cmd/import_wallet.rs:404-419` + `:1363`; `docs/manual/src/40-cli-reference/41-mnemonic.md:1350-1351` (add the lenient-failure exit-4 row/prose for BOTH modes). **Test lockstep (R0 I2 — two existing tests pin the old exit-0 and MUST flip to `.code(4)`):** `tests/cli_bsms_round1.rs:110-131` `cell_5_flipped_sig_lenient_default_emits_notice_and_verified_false` (standalone flipped-sig; also its header comment `:18`) — **cell_5 IS the "signature fails to verify" fixture, no new fixture needed**; `tests/cli_import_wallet_bsms_encrypted.rs:422-468` `round1_encrypted_decrypt_ok_but_sig_fail` (`.success()`→`.code(4)`; comment `:426`). Passing-sig guards (cells 1-3) + strict-exit-2 (cell 6) already exist. **NEW combined-mode test (R0 I4 — the combined `--blob`+`--bsms-round1` arm at `:1363` otherwise ships UNTESTED; cell_13 uses a verified record so a `:1363`-check-revert survives the suite):** add `combined_blob_round1_lenient_failed_exits_4` — blob `tests/fixtures/wallet_import/bsms-1of1-singlesig.txt` (cell_13's fixture) + cell_5's flipped-sig TV1 tempfile → assert `.code(4)` + the `--json` envelope/report still emitted. cell_13 (`cli_bsms_round1.rs:287-314`, verified record `.code(0)`) stays the combined passing guard. **CHANGELOG: a loud "may break `$?`-gated scripts relying on exit 0 for `import-wallet --bsms-round1` (standalone OR combined) with an unverified signature" callout** (+ note the GUI run-modal will present these as failed at the next GUI pin bump — desired, lands lagging).
**Oracle + RED-proof:** cell_5 (+ the encrypted case) assert lenient exits 4 (was 0) + report still prints; passing-sig cells assert 0; strict asserts 2. RED under reverting the `any(Failed)` check (→ exit 0).
**SemVer:** MINOR (default exit-code behavior change).
