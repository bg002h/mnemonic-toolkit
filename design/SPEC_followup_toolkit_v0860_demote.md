## #5 (FUNDS-ADJACENT) — DEMOTE single-string mk1/md1 repair to Unverified candidate
**Gap:** a valid-but-wrong single-string mk1/md1 alias is Blessed by `repair` — no cross-chunk hash exists for a single card, so a beyond-t (≥5-error) miscorrection landing on a DIFFERENT valid codeword is reported "confirmed" (exit 5) with no in-band signal. Risk: ~0 at e≤4 (bounded-distance recovers the original), ~1e-4/attempt at e≥5 (T2-measured), funds-losing. ms1 already got this treatment in Cycle F.
**Fix (R0 I1 — predicate pinned at the ORACLE boundary, NOT `count==1`):** demote ONLY the single-card form that LACKS the content-id oracle. TWO md1 single-string forms exist: **non-chunked** (chunked_flag=0; the v0.35 bypass `vendor/md-codec/src/chunk.rs:615-631` SKIPS the content-id check → no oracle → DEMOTE) vs **chunked-of-1** (flag=1, count=1; goes through `reassemble` → RETAINS the ~2⁻²⁰ content-id check → HAS the oracle → stays **Blessed**). Toolkit `bundle` emits chunked-of-1 (`synthesize.rs:392…` via `chunk::split`) so it MUST NOT demote. → **demote iff the corrected md1 single string is NON-CHUNKED** (chunked_flag bit 0 of the first payload symbol), at bless site `repair.rs:1599` (`repair_via_md_codec` — the executable site, NOT the doc comment `:451-453`, M1) + the single-string mk1 arm (`repair.rs:1049-1062`, `GroupKey::SingleString` singleton — encoder-UNREACHABLE, so DEFENSIVE/untested, M3). Set `SetVerify::Unverified{reason}` with its OWN reason (M4: "a non-chunked single-string card has no cross-chunk/content-id oracle — re-derive the wallet/address to confirm", NOT the reassembly reason).
