## 1. Problem

The published static-musl release binaries are **integrity-only**, not **provenance**: the per-asset `SHA256SUMS.<arch>` lets a downloader confirm a file matches what the maintainer uploaded, but NOT that the maintainer built it from the claimed source. We want **bit-for-bit reproducible** musl binaries so the published hash becomes a *provenance* statement — "provably built from this exact source@tag" — verifiable by any independent rebuilder (cf. Bitcoin Core Guix, Tor).

### 1.1 PROVEN facts (from the recon — do NOT re-derive)

- **NON-REPRODUCIBLE TODAY (empirically measured).** Two release builds of `mnemonic` from byte-identical source (`4329859`, pinned 1.85.0) at two different absolute paths are NOT byte-identical: SHA256 `2dcb759a…` vs `a2996b83…`; `cmp` first differs at byte 41; 304-byte size delta; differing GNU build-ids.
- **Root cause = SINGLE-SOURCE: absolute build-path leak in `.rodata`.** Rust `file!()`/`#[track_caller]` panic-`Location` string literals embed the absolute project-source-root (the diverging `wt-build-A/B` substring is the literal inter-build divergence); the byte-41 differ is `e_shoff` shifting because `.strtab` size differs (mangled symbol names encode the crate path). Build-id divergence is a downstream symptom, not an independent cause. **This is also a `$HOME` PRIVACY leak** — every shipped binary currently bakes in the maintainer's exact build directory.
- **Fix PROVEN byte-identical on the pinned 1.85.0, zero toolchain change:** the recon ran `RUSTFLAGS="--remap-path-prefix=$PWD=/build --remap-path-prefix=$CARGO_HOME=/cargo --remap-path-prefix=$SYSROOT=/rust"` (the shell expanded `$PWD`/`$CARGO_HOME`/`$SYSROOT` before rustc saw them) across two builds at two DIFFERENT absolute paths (`wt-build-A` vs `wt-build-B`). Both remapped cross-path builds → identical SHA256 (`aa5b5fd4…`) + identical build-id; binary still runs. NOTE the flag is the **top-level `--remap-path-prefix`** (verified: `-Cremap-path-prefix` errors `unknown codegen option` on 1.85.0). The `$SYSROOT` remap is a harmless no-op (rustc already virtualizes to `/rustc/<hash>`). **The two-DIFFERENT-paths shape is load-bearing: the remap is only proven effective when the two real paths differ (R0-r3-I1) — a same-path A/B would match trivially without exercising the remap.**
- **Wall-clock-timestamp scope is NARROWER than first stated.** No embedded wall-clock timestamps in **normal Rust** output (no `build.rs` in the toolkit crate, `[profile.release]` defaults deterministic, `debug=0` → no DWARF). **BUT this does NOT extend to the cc/libsecp leg** — `cc`-under-musl `__DATE__`/`__TIME__` is UN-MEASURED (recon L64) and is the showstopper of §1.2; the A/B-same-day double-build does NOT by itself prove the cc timestamp class is neutralized (see §5 third probe).
- **`trim-paths` (the clean Cargo-native form) is NIGHTLY-GATED on 1.85.0.** `--config 'profile.release.trim-paths=true'` build-FAILS on the pinned Cargo 1.85.0 (trips the unstable gate, emits NO binary). On the deliberate 1.85.0 pin the portable fix is the `--remap-path-prefix` form — NOT `trim-paths` — unless we bump the toolchain.

## 4. Hermetic-container design (digest-pinned)

- **Base:** the official `rust:1.85.0` image (Debian-based), pinned **by digest** (`rust@sha256:<digest>`), with `musl-tools` (x86_64 `musl-gcc`) layered via a thin `Dockerfile.repro`. **Container-provenance (R0-r2-I3a):** the BUILT, layered image is published **by digest** and is the source of truth — rebuilders `docker pull <image>@sha256:<digest>` rather than rebuild `Dockerfile.repro` from apt (whose transitive musl-tools deps are NOT pinned by the base digest and resolve to whatever the Debian mirror serves that day). ## 5. secp256k1-sys musl-cc validation plan (the diffoscope / double-build GATE)

**This is the FIRST POC action — it MUST pass before any "musl reproducible" claim.** The proven double-build was on gnu; musl libsecp objects are un-measured.

**Per-arch double-build gate:**
1. In the pinned container, build the target twice **at two different absolute source paths** (e.g. `/build-a/src` and `/build-b/src`), each with the full recipe (§3) — the ENV-channel remap targets adjusted so BOTH map to the same `/build` (different real paths → identical remapped output). **The two-DIFFERENT-paths shape is mandatory and load-bearing (R0-r3-I1): it is the only configuration that actually exercises `--remap-path-prefix`; a same-path A/B would match trivially without proving the remap works. This is also why the P4 gate reuses this two-path build (§6 P4) rather than a same-path rebuild.**
2. **diffoscope** (or `cmp` + `readelf`/`nm` triage) the two binaries AND the two **`.tar.gz` tarballs** (the gzip-pinned §3.13 form — diffoscope also catches a gzip-header mtime regression).
3. **Drill into the secp256k1-sys object specifically:** locate `target/<triple>/release/build/secp256k1-sys-*/out/*.o` (or the archived libsecp) from each build and `cmp` them directly — isolates a C-frontier failure from a Rust-frontier failure.
4. **`SOURCE_DATE_EPOCH` load-bearing probe (R0-I3) — REQUIRED, not optional.** The A/B builds in step 1 run on the same calendar day, so a baked-in `__DATE__` could match by coincidence and green-light a build that is NON-reproducible across days. Therefore additionally:
   - (a) build the libsecp `.o` ONCE with `SOURCE_DATE_EPOCH` **unset** (or set to a deliberately DIFFERENT value) and confirm the `.o` **DIFFERS** from the pinned-epoch `.o` — this PROVES the epoch is honored and load-bearing for the cc leg;
   - (b) then confirm two builds at the pinned epoch match;
   - (c) `readelf -p .comment` / grep the `.o` (and final binary) for any `__DATE__`-shaped residue (e.g. `Jun [ 0-9][0-9] 20[0-9][0-9]`, `__TIME__` `HH:MM:SS`) — expect ZERO. If (a) shows NO difference, `SOURCE_DATE_EPOCH` is NOT being honored (wrong cc version / not passed through) → P-blocker.
5. **`cross` passthrough effectiveness probe (R0-I2) — REQUIRED for aarch64.** Because `cross` A/B runs share the container's INTERNAL paths, A and B can match each other while BOTH retaining an un-stripped host path or a `__DATE__`. So for aarch64 do NOT rely on A/B-equality alone: assert directly that the passthrough took effect — `readelf -p .comment` shows the expected pinned compiler string, and grep the `.o` for absence of any `/build-a`-vs-`/build-b` host-path residue AND absence of `__DATE__` residue. `cross`'s own A/B determinism is WEAKER evidence than the host-native x86_64 leg; treat the direct-residue assertions as the primary aarch64 gate.
6. **gzip-header residue probe (R0-r2-I1 + R0-r4-m4) — REQUIRED.** After packaging, assert the `.tar.gz` gzip header **mtime field is ZERO** (`gzip -l` shows no/zero timestamp, or inspect header bytes 4-7) **AND the OS byte (offset 9) equals the pinned value** (observed `03`/Unix). The OS byte is PRIMARILY pinned by the same container-digest constraint that fixes the compressor build (R0-r2-I3); this residue assertion is the cheap secondary check that a divergent-gzip-build OS-byte (or a non-`-n` mtime) regression is caught — either would ship a non-provenance tarball hash even with a reproducible inner binary.
7. **x86_64-musl (host `musl-gcc`)** and **aarch64-musl (`cross` bundled toolchain)** are SEPARATE gates — pass x86_64 first (P1), aarch64 second (P2). A failure on either is a P-blocker: triage via `readelf -p .comment` (compiler version), `__DATE__`/`__TIME__` residue (→ epoch not honored / wrong cc / passthrough gap), or absolute-path residue (→ `-ffile-prefix-map`/passthrough gap).
8. **Acceptance:** (i) both A/B binaries identical, (ii) both A/B `.tar.gz` tarballs identical, (iii) both A/B libsecp `.o` identical, (iv) the epoch-load-bearing probe (4a) shows a DIFFERENCE when the epoch changes, (v) zero `__DATE__`/host-path residue in the `.o` and final binary, (vi) for aarch64 the passthrough/comment assertions pass, (vii) the gzip-header mtime field is zero (6), AND (viii) the binary still runs (`mnemonic --version`). Record the expected SHA256 of the canonical artifact **alongside the source COMMIT SHA + SOURCE_DATE_EPOCH + container digest** (the §3.12 provenance tuple). **Note (R0-r3-I1): because acceptance (i)/(ii) are measured ACROSS two distinct paths, passing this gate is what licenses the provenance claim — and the SAME two-path build is what P4 re-runs in CI against the published hash, so the remap stays under continuous test release-over-release rather than being proven once and silently regressing.**
