## Phase P0 — (a) payload-digest into `array_id` (wc-codec `raid.rs`, RED-first)
- **PRE-(a) CHECKPOINT (does double duty — I-1 + G-E):** BEFORE writing (a), against the CURRENT (old-derivation) binary: (1) **RED-proof** — construct two arrays with the SAME cosigner-fingerprint seed but DIFFERENT payloads (different xpubs, equal-length 73-B), cross-mix their plates through `raid_reconstruct`, and assert the pre-fix mix returns **`Ok(wrong payload bytes ≠ either original)`** (M-A: at the wc-codec layer the length-prefix XOR-cancels and `stripe_to_payload` `raid.rs:159-173` does no integrity check ⇒ `Ok` with prob 1; the literal "wrong-XPUB" only manifests at the toolkit mk1 re-parse ~50% secp density, so an optional toolkit companion cell is flaky — keep the wc-codec `Ok(wrong bytes)` assertion as the RED-proof). (2) **GENERATE + PIN LEGACY FIXTURES (I-1, load-bearing for P1):** capture the full plate word-lists of two same-seed different-payload **r=2** arrays (+ an r=1 pair) as HARDCODED fixtures in `tests/raid.rs` — these are byte-faithful "plates engraved under the old derivation." `decode` never re-derives `array_id` (SPEC load-bearing recon fact), so these legacy plates pass the equality gate (`raid.rs:363-373`) and are the ONLY way to exercise (b)'s oracle after (a) makes fresh same-seed arrays diverge. This fixture generation MUST happen at THIS pre-(a) checkpoint (post-(a) code cannot emit colliding-id plates).
- **TDD (a) GREEN:** after (a), the same-quorum-mix cross-reconstruct built from FRESH (post-fix) arrays refuses at the equality gate (`RaidArrayMismatch`) — different payloads → different `array_id`.
- **Impl:** in `raid_encode` (`raid.rs:~238`), replace `array_id = array_id_from_seed(seed)` with `top22(SHA-256(seed ‖ SHA-256(canonical)))` where `canonical` = **u32-BE `n` ‖ per index: u16-BE `payload_bits` ‖ exactly `ceil(payload_bits/8)` payload bytes** (SPEC §1a; trailing slack MUST be zero). **M-C decision: DO tighten `raid.rs:233` to reject non-minimal `bytes.len()` (> `ceil(bits/8)`)** — it enforces the injectivity precondition the frozen digest layout depends on, is unreachable from the real caller (`word_card_adapter.rs:85-94`, byte-aligned minimal), and removes an impl/KAT-7-re-derivation divergence vector. Deterministic; EXCLUDE `r` (KAT 4 append-only).
- **Fold the KAT-7 delta (I-1):** update `tests/raid.rs:397-402`'s in-test derivation to re-compute the new formula (re-derive `canonical` identically in-test) — it becomes the (a) determinism pin.
- **Doc-comments (M-6):** update the FIVE stale sites — `pipeline.rs:249-251`, `raid.rs:208-210`, `lib.rs:103` (public field rustdoc), `pipeline.rs:228`, `pipeline.rs:237`. Re-grep `SHA-256(seed)`/`top 22 bits` for residue.
- **KATs:** determinism (same inputs→same id); KAT 4 append-only still green; identical-payload re-issue = no-op mix.

## Phase P1 — (b) spare-parity consistency oracle (wc-codec `raid.rs`)
- **Impl:** in `raid_reconstruct`, when `#parity-present > #data-missing` (≥1 spare equation; `r_available` at `:412`), verify the unused parity equation(s) over all n reconstructed stripes AFTER the solve; also verify parity on a 0-data-missing decode with ≥1 parity present. On mismatch → refuse `RaidArrayMismatch`. **M-2:** extend the `RaidArrayMismatch` doc (`lib.rs:147-151`) + `Display` (`lib.rs:190-193`) to cover the parity-inconsistency cause.
- **KATs (built on the P0 pre-(a) PINNED LEGACY FIXTURES — I-1; fresh post-(a) arrays can't reach the oracle):** cross-mix the two pinned legacy r=2 arrays' plates + 1 missing → the spare parity equation catches it → refuse `RaidArrayMismatch`; a 0-data-missing chimera of legacy plates WITH ≥1 parity plate presented → refuse; a GENUINE r=2 array with ≤1 missing (fresh, post-fix) STILL reconstructs — G-B/G3 no-over-rejection, exact integer algebra (also machine-gated by the existing KATs 1/2/3/9/10 + the 40-case proptest `tests/raid.rs:566-616`, which now newly run the 0-missing parity verification and must stay green). Pin the accepted legacy residual explicitly: a 0-missing pure-data legacy chimera with NO parity plate is NOT caught (SPEC §1b) — a test documents this is the expected (undetectable) behavior, not a bug.
