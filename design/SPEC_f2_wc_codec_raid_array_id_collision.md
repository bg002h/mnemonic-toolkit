## 0. Problem
`wc-codec`'s RAID `array_id` = top-22 bits of `SHA-256(seed)` where `seed` is built ONLY from the ordered cosigner MASTER fingerprints (`word_card.rs:223-244`, hash `pipeline.rs:252-257`) — network/account/script-type-independent; privacy cards contribute `[0u8;4]`. So two arrays for the SAME cosigner set (mainnet vs testnet, account 1' vs 2', BIP-48 vs BIP-45, or a re-issue) carry **identical `array_id` with probability 1**. `raid_reconstruct`'s only cross-array coherence checks are `array_id` + `n` + stripe-width (`raid.rs:363-373`) — none discriminate two same-quorum arrays. Mix n−1 data plates of array A with a parity plate of array B (1 missing): the MDS solve returns `Ok` unconditionally (`raid.rs:466-503`), the 16-bit length prefix XOR-cancels (equal-length same-quorum payloads), the solved stripe is **never integrity-tag-checked** (its tag died with the lost plate; mk1 canonical bytecode has no checksum — `C4`), and mk1 structural re-parse passes ≈50% (secp curve density) → `*recovered` prints a **valid-but-wrong cosigner xpub at exit 0** (eval empirical 21/36). Violates the `raid.rs:329-330` refuse contract + `RaidArrayMismatch` doc (`lib.rs:147-151`). KAT 6 (`tests/raid.rs:316-348`) tests only DIFFERENT-seed arrays — the exact blind spot.

**Load-bearing recon fact:** `raid_reconstruct` NEVER re-derives `array_id` — it equality-compares the engraved values (`array_id_from_seed` has one caller: `raid_encode`, `raid.rs:238`). So changing the DERIVATION is a **non-wire-LAYOUT** change: existing engraved plates keep decoding AND reconstructing among themselves; only cross-vintage grouping refuses loudly (fail-closed). The array-id field stays 2 words / 22 bits at the same header position; `H0_VERSION` (=0) untouched.

## 1. Fix (three parts — recon §6; skip the wire-breaking option (d))

### (a) Fold a deterministic payload digest into `array_id` — PREVENTION for all future arrays (incl. r=1), NON-WIRE-LAYOUT
In `raid_encode` (`raid.rs:~238`), derive:
`array_id = top22(SHA-256(seed ‖ digest))` where `digest = SHA-256(canonical)` and
`canonical` is an INJECTIVE fixed-width serialization: **`n` as u32-BE** ‖ for each index in order: **`payload_bits` as u16-BE** (`≤0xFFFF`, enforced `raid.rs:233`) ‖ **exactly `ceil(payload_bits/8)` payload bytes** (M-5 — the minimal byte-count; trailing slack bits MUST be zero). `H` = SHA-256 throughout. The byte-count rule is load-bearing for INJECTIVITY: `raid_encode`'s domain today permits `bytes.len() > ceil(bits/8)` (`raid.rs:233`), which would make a naive `payload_bits ‖ bytes` serialization non-injective (e.g. `{(1,[80 00 02 FF]),(8,[AA])}` vs `{(1,[80]),(2,[FF 00 08 AA])}` collide) — unreachable from the real mk1 caller (byte-aligned minimal, `payload_bits=8·len`, `word_card_adapter.rs:85-94`) but this derivation FREEZES into KAT 7 + engraved steel, so pin the minimal count now (optionally also tighten `raid.rs:233` to reject non-minimal `bytes.len()`). This exact byte layout is the single source both the impl and the KAT-7 in-test re-derivation read.
- **Deterministic** (repro builds; same-array reprints still group). **KATs intact EXCEPT KAT 7's in-test derivation assertion** (`tests/raid.rs:397-402`), which pins the OLD `top22(SHA-256(seed))` formula and is updated in lockstep to the new formula — it BECOMES the (a) determinism pin (I-1; see §2). **EXCLUDE `r`** from the digest so P1 stays byte-identical for r=1 vs r=2 (preserves the append-only property — KAT 4 `tests/raid.rs:220-239`, which compares ParityA words across r=1/r=2).
- **Privacy-mode bonus (recon):** today two all-privacy-card arrays with equal `n` collide with probability 1 (seed = `0^{4n}`); the payload digest differentiates them — (a) strictly IMPROVES the privacy-mode case.
- Two DIFFERENT wallets with the same cosigner set now get DIFFERENT `array_id`s → the cross-array mix is caught by the existing equality gate → `RaidArrayMismatch`. An identical-payload re-issue still shares the id — HARMLESS (identical payloads ⇒ identical stripes ⇒ mixing is a no-op).
- **Engine-side** (in `raid.rs`, not caller-side `word_card.rs`) so every future caller inherits it. ~15 LOC.
- **Compat:** a plate re-encoded on the new binary won't group with survivors engraved under the old derivation → loud `RaidArrayMismatch` (never silent). The replacement workflow re-encodes the whole array from the n cards anyway. Old-vs-old arrays keep colliding → covered by (b)+(c).
- **Covers:** r=1 AND r=2, all future arrays; collision 1 → ~2⁻²².

### (b) Spare-parity consistency oracle — DEFENSE-IN-DEPTH + legacy r=2, NON-WIRE, decode-only
In `raid_reconstruct` (`raid.rs`), the oracle FIRES exactly when **#parity-plates-present > #data-plates-missing** (i.e. ≥1 SPARE parity equation exists after the solve): verify the unused parity equation(s) over all n reconstructed stripes; also verify parity when 0 plates are missing but ≥1 parity plate is present (catches "chimera" mixes that today return each plate's genuine-but-mismatched payload). A foreign mix is detected with probability ≈ 1 − 2⁻¹¹·ᵂ (W ≈ 55+ for real xpubs → effectively certain). On mismatch → refuse (`RaidArrayMismatch` — see §4). ~40-70 LOC.
- **Exact coverage matrix (I-2 — honest; no exposure silently open):**
  - **(a)** prevents ALL FUTURE arrays (r=1 and r=2) — collision 1 → ~2⁻²².
  - **(b)** catches, RETROACTIVELY on legacy arrays: r=2 with ≤1 data missing AND ≥1 parity present (a spare equation exists); and any 0-data-missing decode where ≥1 parity plate is presented.
  - **(c)** advisory covers every decode that produces a `*recovered` (MDS-solved) plate — legacy r=1-with-1-missing, r=2-with-1-data-and-1-parity-missing, and the residual r=1 tail.
  - **ACCEPTED, DOCUMENTED LEGACY RESIDUAL (in-band undetectable):** a legacy **0-missing pure-data chimera with NO parity plate presented** (n data plates of mixed vintage, no P1/P2 in the input) — array_id/n/W all equal, no parity equation, no `*recovered` plate → (a) is future-only, (b) has no equation, (c) never fires (G4). Info-theoretically undetectable in-band for legacy plates. **Honest mitigation = documentation + a manual note:** "when decoding a card SET, include a parity plate — it turns a mixed-vintage chimera into a loud `RaidArrayMismatch` refusal." (Future arrays under (a) are immune regardless.)

### (c) Always-on `*recovered` verify-this-xpub advisory — UNIVERSAL mitigation (legacy r=1 + residual)
Any `*recovered` plate (a plate whose value came from an MDS solve, not its own engraving) carries a loud advisory to independently verify the reconstructed xpub, because a wrong-fit mix cannot always be prevented (legacy arrays, r=1-with-1-missing).
- **Text/stderr:** advisory at the `*recovered` marker site (`word_card.rs:377-390`, marker `:383`) + stderr.
- **`--json`:** add a boolean/advisory field on the recovered entry (`RecoveredJson`, `word_card.rs:516-517`) → **bump `WORD_CARD_SCHEMA_VERSION`** (`word_card.rs:35-39`). This is a `--json` wire-shape change → paired-PR GUI companion note (NOT `schema_mirror` — no flag change).
- **Covers:** every decode that produces a `*recovered` (MDS-solved) plate — legacy r=1-with-1-missing + the residual r=1 tail + r=2-with-1-data-and-1-parity-missing — as human-check mitigation, not prevention. Does NOT cover the 0-missing pure-data chimera (no `*recovered` plate — the §1b accepted residual). ~15-40 LOC.
