## Important (2)

### I1 — verify_bundle.rs descriptor loop has NO Entropy arm to mirror — site 15's "derive like the Entropy arm" is non-implementable as written — `verify_bundle.rs:776-855`
The SPEC §2 says "Each call site does its OWN derivation using `derive_language` (mirroring its existing Entropy arm)" and §5 site 15 says "helper §2; push `language: emit_language`". But `verify_bundle.rs`'s descriptor binding loop (`:776-855`) has only THREE arms: Phrase||Seedqr (`:781`), Xpub (`:823`), and `else → DescriptorReparseFailed` (`:849`). **There is no Entropy arm here to mirror** (unlike `bundle.rs:1375` which does have one). The verify_bundle Phrase arm derives via `bip39::Mnemonic::parse_in(lang, phrase)` → `derive_master_seed` → `new_master` → `derive_priv(&anno_path)` (`:811-821`); an entropy-bearing Ms1 arm must instead use `bip39::Mnemonic::from_entropy_in(derive_language, &entropy)` then the same seed→master→`derive_priv(&anno_path)` chain (the pattern `bundle.rs:1393-1406` uses), OR call `derive_slot::derive_bip32_from_entropy_at_path` directly. **Fix:** §2/§5 site 15 must spell out the verify_bundle Ms1 arm's derivation explicitly (via the `derive_slot` helper), and note the else-branch error type there is `DescriptorReparseFailed` (not `BadInput`). Additionally flag (Minor) that this loop lacks an Entropy arm at all — pre-existing asymmetry with `bundle.rs`, out of scope but worth a one-line FOLLOWUP note.

### I2 — I3 Seedqr-fix narrative + test 9 assert the wrong pre-fix baseline — `bundle.rs:1142-1162` + `:1408` + SPEC §4/§8 test 9/§9
The SPEC §4 claims "`[Seedqr, Path]` in canonical mode currently slips past" and test 9 asserts it is "rejected (the widened gate)". Verified: the canonical gate at `:1142` is wrapped in `if !is_non_canonical` and only checks `has_phrase && has_path` (`:1153`), so a `[Seedqr, Path]` slot does pass the gate (has_phrase=false). BUT `bundle_run_unified_descriptor`'s binding loop (`:1305-1430`) has **no Seedqr arm** (only Phrase `:1306`, Xpub `:1345`, Entropy `:1375`, else→`BadInput` `:1408`), so `[Seedqr, Path]` then hits `else→BadInput` (exit 1). So today's behavior is **exit-1 BadInput rejection, not acceptance** — the input was never accepted; the SPEC's "slips past" framing implies a silent-acceptance bug that does not exist. The widened gate changes the rejection from exit-1 `BadInput` to exit-2 `SlotInputViolation{kind:"conflict"}`. **Fix:** (a) §4/§9 must restate the Seedqr change as an exit-code/error-class normalization (1→2, BadInput→SlotInputViolation) for an already-rejected input; (b) test 9 must assert the post-fix exit 2 + `kind:"conflict"` AND document that the pre-fix path returned exit-1 BadInput (`feedback_recapture_golden_only_when_current_correct`); (c) CHANGELOG should say "normalize `[Seedqr, Path]` canonical-mode rejection to exit-2 SlotInputViolation (was exit-1 BadInput)". For Ms1 the gate-widening IS load-bearing (without it, `[Ms1, Path]` would pass the gate and, once the site-13 arm exists, be silently accepted with the explicit path overriding the canonical origin).
