### 7. The §5a fail-safe (THE adversarial item) — fail-SAFE, CANNOT silent-wrong — CONFIRMED
The implementer's deviation (flag #2): in own-only mode the toolkit knows `N` (`d.n`, `:1443`) but not the true cosigner count `M`. It infers `j = n − m_cosigners` from the SUPPLIED cosigner count `M'`. The up-front refuse fires only when STRUCTURALLY detectable: `m_cosigners >= n` (no own slot left) → ModeViolation naming `--search-cosigner-subset` (`:1769-1781`). When `M' > M` but `M' < N`, the up-front gate does NOT fire. I proved this fails SAFE:
- The own-anchored generator uses ALL `m_cosigners` cosigner cards in EVERY assignment (`own_anchored_unrank` appends `k_own..k_own+m` whole, `permutation_search.rs:719-722`); `build_keyed_template_descriptor` requires exactly `n` slots (`synthesize.rs:288`), and the id/scriptPubKey compare is over the full `n`-key descriptor.
- So if the operator over-supplies a WRONG cosigner, that wrong key poisons EVERY candidate's key-set ⇒ no candidate can equal the real wallet's key-set ⇒ NO-MATCH (`SearchOutcome::None` → refuse exit 4). It can never complete the operator's true wallet, NOR a different wrong wallet (that would require a wallet containing the exact `M'` supplied keys whose id-prefix/scriptPubKey collides — the SAME residual governed by `validate_prefix_strength(prefix.len(), realized_s)` and the full-scriptPubKey address match, sized to the LARGER `s_own`, exactly as on the exact path).
- **Adversarial probe (reviewer-authored, ran GREEN):** a 3-of-3 real wallet {A@1, B@0, C@0}, over-supplied with cosigner B (real) + an OUTSIDER card (wrong, replacing C), `--own-account-max 4`, `m_cosigners=2 < n=3`, `j=1`. Result: **exit 4 NO-MATCH** — never a silent-wrong completion.
**Assessment:** "refuse up front only when structurally detectable (`m≥n`), else fail-safe NO-MATCH" is funds-safe and reconciles acceptably with SPEC §5a's "REFUSE up front" framing — the up-front refuse is structurally impossible when `M` is unknown, and the NO-MATCH fallback never silent-wrongs.
