## IMPORTANT

- **I-1 — `--own-account-max K` (D10) and EVERY `pool.len() > n` completion is BROKEN: the search never places the over-supplied / cosigner keys, so a legitimate wallet returns NO MATCH; and the realized-`S` the prefix/cap are sized to is NOT the space actually enumerated (SPEC §6.1/§6.2/§7-floor-5 + §2 `--own-account-max` fidelity break).**
  `restore.rs:1332-1335,1407-1414,1423-1424,1691`. The engine entrypoint is `ps::search(n, evaluator, mode)` with `n = d.n` (the SLOT count). `unrank_permutation(rank, n)` (`permutation_search.rs:494-505`) builds `elems = (0..n)` and permutes ONLY those — every `assignment` it yields is a bijection of `{0,…,n-1}`. The evaluator does `pool[pi]` for `pi ∈ assignment`, so pool indices `≥ n` are NEVER evaluated. But the code lets `pool.len() > n` (`--own-account-max K` derives K own keys, `pool = own(K) ++ unassigned_cosigners`, gated only `pool.len() >= n`, `:1407`) and sizes the strong-prefix + cap to `realized_s = perm_count_u128(pool.len(), n) = P(pool,n) > n!` (`:1423`, `run_capped_search:1674`). So:
  - **(funds-recovery break)** With `pool.len() > n` the cosigners (at pool indices `≥ K`) and the over-supplied own accounts (indices `n…K-1`) are unreachable → the real key→slot assignment is never tried → `SearchOutcome::None` → refuse, even though the wallet exists and is correctly specified.
  - **(sizing fidelity break)** `validate_prefix_strength(prefix.len(), P(pool,n))` and the cap estimate over `P(pool,n)×outer` describe a space the engine does not scan (it scans `n!×outer`). The prefix demand is over-strict (safe direction for collisions — does NOT create a weak-prefix hole), but it contradicts the P1 review item 5 / SPEC §6.2 claim that "S = the actual enumerated candidate count" and the cap ETA is wrong.
  **Reviewer evidence (two independent proofs):**
  1. Engine-enumeration probe (faithful port of `unrank_permutation`): `n=2, pool_len=4` → engine `perms=2`, `realized_S=P(4,2)=12`, **max pool index ever produced = 1**, the cosigner at pool idx 3 is **never** evaluated.
  2. End-to-end CLI reproduction (temp test, since reverted): a real 2-of-2 `wsh-multi` with the OWN seed at account 1 + a cosigner at account 0, run with `--own-account-max 3` (pool = own@0,own@1,own@2,cosigner = 4, n=2) and the correct recorded `--expect-wallet-id` → **`✗ NO MATCH`, exit 4** — the wallet is unfindable though the golden builds fine and the id is the real recorded one.
  **Note** the multi-account EXACT-list case is NOT affected (`--account 0,1` → `pool.len()==n`, full `n!` permuted) — `multi_account_own_resolves_both_slots:621` passes precisely because pool==n. The defect is confined to `pool.len() > n`, which ONLY `--own-account-max` (or, in principle, a future over-supply) reaches — and there is no test for `--own-account-max` at all, so it shipped untested.
  **Class:** real correctness gap (a documented recovery feature is non-functional + a SPEC sizing invariant is misimplemented). Fail-safe (refuse, never wrong), hence Important not Critical.

## MINOR

- **M-1 — the own-origin deviation's CENTRAL claim (reproduce a DEFAULT BIP-87 / non-`bip48` `--multisig-path-family` wallet) is UNTESTED.** Every integration vector emits at explicit BIP-48 `48'/0'/acct'/2'` slots (`canonical_path:112-118`); none exercises the default BIP-87 multisig emit (`parse.rs:62-94`, default `Bip87`) nor `--multisig-path-family bip48` end-to-end through emit→complete. The logic is correct by construction (`own_origin_from_family` substitutes the account into whatever family the cosigner carries) but the load-bearing reproduction property the section banner (`:1075-1085`) rests on has no pin. Add a vector that emits a multisig template at the DEFAULT family and completes it.
- **M-2 — `cosigner_family` picks the FIRST cosigner arbitrarily.** `:1307-1311` `.next()`. Sound for homogeneous wallets and fail-safe for heterogeneous ones (wrong guess → refuse, not wrong), but a heterogeneous-family multisig (mixed-vendor cosigners with different purposes) would false-refuse in id/address search unless `--origin` is given. Consider: try each distinct cosigner family for the own slot, or document `--origin` as the escape hatch in the no-match message.
- **M-3 — SPEC §7-floor-1(ii) full-policy `all_verified`-promotion not done in P3a.** `restore.rs:2906-2917` still computes a "partial" advisory status rather than the hard refuse the SPEC/plan §4/floor-1(ii) called for ("promote the today-advisory `all_verified` to a hard refuse — closes the v0.44.0 unsupplied-slot regression"). This is on the KEYED full-policy `run_multisig` path (separate from the template completion, which has its own hard every-slot gate `:1393-1414`), so it does not affect P3a's template funds-safety — but the floor-1(ii) item is unmet on the path the SPEC named. Confirm it is intentionally deferred (P3b / separate) and tracked.
- **M-4 — explicit mode auto-infers the own origin but the warning speaks only to the key→slot assignment.** In `complete_explicit_assignment` the own key's origin is still the auto-guessed `cosigner_family`/canonical value (built in `own_keys`, `:1352-1380`); the operator asserts the assignment but not the own origin family. The warning (`:1752-1757`) covers "a wrong assignment produces a wrong wallet silently" — origin is part of that, so it is within the warned envelope — but a line noting the own origin is INFERRED (and to pass `--origin` if the own family is non-default) would close the gap. Within design intent (SPEC §4.3: explicit mode = no verification, operator's risk), hence Minor.

## To turn GREEN

1. **Make the engine subset-aware** — extend `permutation_search::search` to enumerate `P(pool, n)` injective placements of a `pool`-sized candidate set into `n` slots (k-permutations, not just `n!` full permutations), so the over-supplied own candidates AND the cosigners are all reachable and `realized_s = P(pool,n)` is the truly-enumerated space. OR
2. **Refuse the over-supply for now** — if a subset-search is out of P3a scope, hard-reject `pool.len() > n` at `:1407` (i.e. require `pool.len() == n`, dropping the range branch) and gate `--own-account-max` to "P3b / not yet" so a user is told it is unsupported instead of getting a silent NO MATCH for a real wallet; restore `realized_s = n!` (= `perm_count_u128(n,n)`) so the sizing matches the scan. Either way, ADD a test that exercises the chosen behavior (a working `--own-account-max` completion, or its explicit refusal).
