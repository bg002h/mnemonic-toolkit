#### N-1 — SPEC §2.5 row 7 stem template has unfilled `(got {I1, I2, ...})` interpolation slot; handler emits a shorter literal that omits this slot, and the test asserts the handler form

**Source ground truth:**

- `design/SPEC_slip39_v0_13_0.md:218` — row 7 template:
  `slip39 combine: shares disagree on identifier (got {I1, I2, ...}); shares must come from the same secret`
- `cmd/slip39.rs:664` — handler emission:
  `"slip39 combine: shares disagree on identifier; shares must come from the same secret"`
- `tests/cli_slip39_refusals.rs:275-277` — test assertion matches the handler (omits the `(got {I1, I2, ...})` slot)

The SPEC template includes a `(got {I1, I2, ...})` interpolated identifier-set slot
that the handler does not render. The `Slip39Error::IdentifierMismatch` variant is a
unit variant (no carried data — see `slip39/error.rs`), so the lib does not surface
the identifier set anyway. Two reconciliation options:

1. **Patch SPEC** to drop `(got {I1, I2, ...})`, aligning with the handler/test.
2. **Promote `IdentifierMismatch` to a struct variant** carrying `Vec<u16>` and render
   the slot. This is a lib-shape change with API surface impact; defer.

Recommend option 1 — pure SPEC tightening, zero code churn. Could be folded inline at
PE rollup or as a P2.3 SPEC patch when other refinements queue.

Note: the disjunctive row 21 + row 22 assertions in `cli_slip39_refusals.rs:581-589`
and `:609-614` use `stderr.contains("slip39 combine: shares disagree on identifier")`
as one branch — a STRICT SUBSTRING of the handler's stem — so the substring matches
correctly. The disjunctive tests are not affected by this divergence.

### 6. mlock pin discipline (per-share O(N))

for (g_idx, g) in rendered.iter().enumerate() {
