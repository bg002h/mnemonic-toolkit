## Implementation summary

New module `crates/mnemonic-toolkit/src/wallet_export.rs` (~210 LOC) carries:
- `REFUSAL_SECRET_INPUT` constant — SPEC §3 byte-exact stderr.
- `format_stub_message(name)` — SPEC §7 sparrow/specter byte-exact stderr factory.
- `validate_watch_only(slots)` — refuses phrase / entropy / xprv / wif subkeys (SPEC §3).
- `build_descriptor_string(template, slots, k, network, account)` — assembles the canonical multipath `<0;1>` descriptor from `ResolvedSlot` xpubs + key-origin metadata, then round-trips through `miniscript::Descriptor::<DescriptorPublicKey>::from_str` + `to_string()` to emit the canonical form with `#checksum` suffix (SPEC §4 lock).
- `format_bitcoin_core_importdescriptors(canonical, range, ts, _ver)` — SPEC §5 emitter. Multipath descriptors split into 2 entries via `Descriptor::into_single_descriptors()`; each split is re-`.to_string()`-ed to capture the per-form checksum (receive `#wc3n3van` vs change `#lv5jvedt` for the BIP-84 Trezor vector).
- `format_bip388_wallet_policy(template, slots, k, network, account)` — SPEC §6 emitter; `description_template` uses `@N/**` placeholders, `keys_info` is `[fp/path]xpub` strings in slot-index order.

New subcommand at `crates/mnemonic-toolkit/src/cmd/export_wallet.rs` (~210 LOC):
- `ExportWalletArgs` clap struct with all SPEC §2 flags (`--template` / `--descriptor` mutually exclusive; `--slot` / `--threshold` / `--network` / `--account` / `--language` / `--multisig-path-family`; `--format` / `--output` / `--range` / `--timestamp` / `--bitcoin-core-version`).
- Custom value-parsers `parse_range` (`<u32>,<u32>`) and `parse_timestamp` (`now` | unix seconds).
- `run()` dispatcher: stubs sparrow/specter first → `validate_watch_only` → mutual-exclusion checks → reuses `cmd::bundle::resolve_slots` (already `pub(crate)`) for slot resolution → routes to format-specific emitter → serializes pretty JSON to stdout (or `--output` file).

Wired to clap via:
- `cmd/mod.rs`: `pub mod export_wallet;`.
- `main.rs`: new `Command::ExportWallet(...)` variant + dispatch arm + `mod wallet_export;` declaration.
- `error.rs`: `ExportWalletSecretInput` (no payload) and `ExportWalletFormatStub(&'static str)` variants; both exit 2; `kind()` and `message()` arms wired; messages source from `wallet_export::REFUSAL_SECRET_INPUT` and `wallet_export::format_stub_message(name)` for single-source-of-truth.

## Hazards encountered + resolutions

1. **`<0;1>` split checksum hazard (SPEC §4 / R1-C2):** Verified during implementation by capturing miniscript's `Display` output for the multipath input vs the two split forms — the 3 checksums (`#hpg6d6w2` multipath, `#wc3n3van` receive, `#lv5jvedt` change) are all distinct and all auto-generated by miniscript. The implementation never manually computes checksums; it round-trips through `from_str` + `to_string()` for each split. **Resolution:** byte-pinned all 3 in tests.

2. **`--bitcoin-core-version 24` vs 25 shape diff (SPEC §9 cell 6 conditional):** No diff was found for the field set this toolkit emits (`desc` / `active` / `internal` / `range` / `timestamp`). Bitcoin Core 24 + 25 both accept and require this same shape. The conditional clause in SPEC §9 explicitly allows reduction to a documentation test if the diff is empty — cell 6 asserts byte-identical output between `--bitcoin-core-version 24` and `--bitcoin-core-version 25`. The flag is retained for future-proofing (e.g., `next_index` in 25+ which the toolkit does not yet emit).

3. **slip0132 input-normalization info-line on stderr (SPEC §5.5.a):** Defaulted to **suppressed** (Option B) per orchestrator instruction, mirroring `verify-bundle`'s read-only checker semantics. The `_slip0132_signals` returned by `cmd::bundle::resolve_slots` is `_`-bound at both call sites in `cmd/export_wallet.rs::run`. A future cycle can elevate to Option A (surfacing the info-line) by uncomment-and-thread; no SPEC change required since the suppression is consistent with existing `verify-bundle` behavior.

4. **`--descriptor` passthrough + `--format bip388`:** Mutually unsupported in v0.7 — BIP-388 requires `@N/**` placeholders, but `--descriptor` substitutes real xpubs at the canonical layer. Refusal text emitted from the dispatcher; this is not a SPEC-mandated cell but tightens the surface. Future-direction: could synthesize `@N/**` placeholders by template-detecting the descriptor structure, but punted to v0.8 if demand surfaces.

## Known caveats / FOLLOWUPS for v0.8

- **`--format hwi` (Hardware Wallet Interface JSON):** SPEC §11 deferred. Adopt when demand signal surfaces.
- **Sparrow / Specter actual wallet-import-JSON formats:** SPEC §7 stubs are stable. Sparrow's format is JSON-with-metadata; Specter's is a label dictionary plus descriptor. v0.8 brainstorm if user demand surfaces.
- **slip0132 info-line on `export-wallet`:** suppressed by default per `verify-bundle` Option B precedent. v0.8 cycle can elevate to Option A if user direction shifts.
- **`--descriptor` + `--format bip388` interop:** refused at runtime in v0.7. Could be implemented via reverse-template-detection in v0.8.
