## 1. Guard fail-safe ruling — the load-bearing one. GENUINELY FAIL-SAFE.

**Claim under test:** could a `git fetch`/`git diff` ERROR silently yield `relevant=false` → heavy steps skip → the *required* `examples` check reports GREEN without ever running the gate (a silent false-green on a code change = Critical)?

**Verdict: NO. Definitively closed on four independent legs.**

**(a) Default shell is fail-fast.** The workflow has **no `defaults:` block and no per-step `shell:` override** (grep confirmed), so every `run:` step uses the GitHub-Actions default `bash --noprofile --norc -e -o pipefail {0}` — errexit + pipefail ON.

**(b) Errexit fires on BOTH failure points — empirically reproduced** under the exact `bash --noprofile --norc -e -o pipefail` shell:
- `git fetch --no-tags --depth=1 origin "$GITHUB_BASE_REF"` fails → **exit 128, `relevant=false` NEVER echoed** (job dies before the branch).
- `changed="$(git diff --name-only FETCH_HEAD HEAD)"` — this is the subtle one: a *bare* assignment (not `local`/`export`/`declare`, so no SC2155 masking) whose command substitution fails **does** trigger errexit → **exit 128, `relevant=false` NEVER echoed**. Verified empirically, not assumed.
- Clean success (empty diff, exit 0) correctly reaches the grep and the docs-only false branch → exit 0.

So the ONLY path to `relevant=false` is: fetch succeeded **and** diff succeeded **and** grep found no gated match — a genuine docs-only PR. There is **no** error-path that yields a green skip.

**(c) Two-dot diff base is correct and over-includes (safe direction).** The guard uses the two-arg `git diff FETCH_HEAD HEAD` (== two-dot `FETCH_HEAD..HEAD`) after `git fetch`-ing the PR base into `FETCH_HEAD`; checkout uses `fetch-depth: 0`. Two-dot compares the base-tip tree to the (merge-commit) HEAD tree, so it **over-includes** base-side drift → at worst it runs the gate unnecessarily; it can **never** miss a PR-side edit to a gated path. `GITHUB_BASE_REF` is always set (guard is `if: github.event_name == 'pull_request'`).

**(d) Grep set == gated push `paths:` set — no gap.** Regex `^(\.examples-build/|docs/Examples\.pdf$|scripts/install\.sh$|crates/|Cargo\.lock$|Cargo\.toml$|\.github/workflows/examples\.yml$)` matched against representative inputs:
- TRUE (correct): `crates/mnemonic-toolkit/src/main.rs` (nested), `.examples-build/gen.sh` (nested), `.github/workflows/examples.yml` (self — so this workflow's own PR exercises born-green, not a self-no-op), `Cargo.lock`, `Cargo.toml`, `scripts/install.sh`, `docs/Examples.pdf`.
- false (correct): `docs/manual-gui/**`, `docs/manual/**`, `.github/workflows/manual.yml`, `design/FOLLOWUPS.md`, `README.md`, and — via `$`-anchoring — `Cargo.lock.bak`, `docs/Examples.pdf.tmp`.

All 7 gated push-path classes are covered; `crates/` + `.examples-build/` are unanchored (match nested), single-file paths are `$`-anchored (no over-match).

**Three-scenario trace (brief item 1):**
