## §B Findings

### Critical

#### C1 — `\index{share!SLIP-39}` LaTeX sub-entry will fail `lint.sh` bidirectional check; cannot match the proposed table row `share, SLIP-39`

**Location:** Plan §3.2 final paragraph + §8 row 2 + §8 NOTE.

**Issue:** The `lint.sh` step 6/6 source-side scan extracts `\index{X}` markers via `grep -oE '\\index\{[^}]*\}' | sed -E 's/^\\index\{([^}]*)\}$/\1/' | sed -E 's/\\_/_/g'`. The only escape transform is `\_` → `_`. The `!` LaTeX sub-entry separator is NOT stripped. Result: source-side term is the literal string `share!SLIP-39`. The table-side term parser is `grep -oE '^\| `[^`]+`' | sed -E 's/^\| `([^`]*)`$/\1/'`, which extracts the backtick-quoted leading cell verbatim — so the proposed row `| `share, SLIP-39` | ...` extracts to `share, SLIP-39`. **`share!SLIP-39` ≠ `share, SLIP-39`**, so both bidirectional grep checks fail (source-side: marker has no matching row; table-side: row has no matching marker). The lint emits two FAIL lines and exits 1. LOCK criterion §10.8 (`make ... lint` returns `[lint] OK`) will not pass.

**Reproduced:** `echo '\index{share!SLIP-39}' | grep -oE '\\index\{[^}]*\}' | sed -E 's/^\\index\{([^}]*)\}$/\1/' | sed -E 's/\\_/_/g'` → output `share!SLIP-39`. Confirmed against the live `lint.sh` lines 124-125.

**Plan acknowledges the risk but defers:** §8 NOTE says "Verify the lint behavior at GREEN-write time before transcribing." This is exactly the kind of check that should be folded at R0 because the resolution determines the marker-set scheme — there are three viable resolutions, each with a different cascade:
- (a) Drop the LaTeX sub-entry: use `\index{share, SLIP-39}` (literal comma in marker), table row term `share, SLIP-39`. Pro: zero lint changes, table reads naturally. Con: PDF render's `makeindex` step interprets the comma as part of the term, not as a sub-entry separator (renders as a flat single-line entry, not nested under `share`).
- (b) Drop the marker entirely: use `\index{share (SLIP-39)}` or `\index{SLIP-39 share}`. Pro: zero lint changes, no comma. Con: alphabetical placement in the PDF index loses the desired "see also `share`" co-locality.
- (c) Patch `lint.sh` to strip `!` to `, ` in the source-side normalizer (`sed -E 's/!/, /g'` after the existing strip). Pro: keeps the plan's marker semantics (true LaTeX sub-entry); table reads naturally. Con: out-of-scope `lint.sh` change introduces a new normalization rule; sets a non-obvious convention; may need a sibling change to handle other LaTeX index syntax (`@` sort key, `|` formatting cmd, etc.) the same way.

**Fix:** Pick one of (a)/(b)/(c) at R0 fold. Recommend (b) — `\index{SLIP-39 share}` — because it (i) keeps the source-side scan + table form trivially matching, (ii) avoids any `lint.sh` patch (out of P3 scope per §2.3), (iii) preserves the chapter-discoverability intent (a reader looking up "share" in the PDF index finds the explicit cross-entry under `share` via the existing `\index{share}` should that get added in a sibling chapter; SLIP-39 keeps its own entry under "S"), and (iv) is the form most other technical books use (e.g. "Bitcoin" / "Bitcoin transaction" / "Bitcoin script" as siblings, not as "Bitcoin!transaction" sub-entries) — flat is normally fine for a 5-marker chapter. (c) is the "purest" answer per LaTeX intent but adds machinery the rest of the manual doesn't use.

**Confidence: 95.**

### Important

#### I1 — Plan §3.10 + §7 part 3 says SLIP-39 imports onto Trezor One, but the SPEC's own §3 OOS row says Trezor One predates SLIP-39

**Location:** Plan §3.10 outline + §7 part 3 (Trezor hardware compatibility note).

**Issue:** The plan's §7 part 3 prose says: *"Users with a Trezor device can verify by importing via `trezorctl recovery-device --backup-type slip39-basic` (Trezor One) or the Trezor Suite recovery wizard (Trezor Model T)."* This contradicts SPEC §3 OOS table row `OOS-slip39-import-trezor-onev-format` ("Trezor One predates SLIP-39 (uses raw BIP-39)"). SLIP-39 is supported on Trezor Model T and Trezor Safe family only — NOT on Trezor One. Shipping the chapter with this claim would mislead a Trezor One owner into a `trezorctl` command that returns an error.

**Second issue, same paragraph:** the plan says `slip39-basic` is the import mode. SLIP-39 has TWO modes per SLIP-0039: `slip39-basic` (single-group, e.g. the §4.3 2-of-3 example) and `slip39-advanced` (multi-group, the §4.4 2-of-3-of-2-of-3 example which is the recipe input to §3.10). If the chapter pins the multi-group example as the recipe input AND tells the reader to use `--backup-type slip39-basic` import, the import will fail (mode mismatch with the encoded shares). At least one mode mention is wrong; both should be addressed.

**Fix:** Rewrite §7 part 3 prose to (a) drop the Trezor One reference entirely (per SPEC OOS row), (b) clarify both `slip39-basic` and `slip39-advanced` import modes — match the example shape, OR keep the recipe-input as example 4's multi-group shares and reference `slip39-advanced`, OR if the architect / planner is uncertain about current Trezor firmware mode names, soften the prose to "via the Trezor Suite recovery wizard (consult Trezor's current docs for the exact backup-type flag, which has historically varied by firmware version)" — i.e., name nothing version-specific. The plan's existing N3 escape hatch ("their precise UI flow varies by firmware version") supports the soften-and-defer path.

**Confidence: 90.**

#### I3 — Plan §7 part 2 + §11.4 should disclose the 0.3.0 ↔ commit-hash relationship in chapter prose, not just in the plan risk row

**Location:** Plan §7 part 2 (Trezor recipe) + §11.4 (Risk row 4: `python-shamir-mnemonic` API drift).

**Issue:** Plan §7 part 2 pins `pip install shamir-mnemonic==0.3.0`. Plan §11.4 mitigation says "R0 reviewer should verify the pinned version matches `python-shamir-mnemonic@17fcce14` SemVer." Architect cannot directly verify that 0.3.0's source tree corresponds to commit `17fcce14` without cloning + comparing trees; this is out of architect-scope. **What's verifiable:** 0.3.0 IS the latest released PyPI version (per `pip index versions shamir-mnemonic`); the toolkit's lib is bit-exact verified against fixture vectors vendored at commit `17fcce14` (per `tests/lib_slip39_vectors.rs:3-4`); 0.3.0 may or may not encode the exact same wire format depending on what's changed since the vendored commit.

The risk is that a reader runs the recipe with 0.3.0 and the recipe FAILS because some SLIP-39 mnemonic format detail changed between `17fcce14` and the released 0.3.0 (e.g., extendable-bit encoding, identifier-bit-width, etc.). The plan's mitigation is "verify the pinned version" — but verifying this is non-trivial (requires actually running the recipe end-to-end, which is the chapter's claim).

**Fix:** Add disclosure to chapter prose (within §3.10 or §7 part 2): "Recipe pinned to `shamir-mnemonic==0.3.0` (latest at chapter-write 2026-05-14). Toolkit's library compatibility is verified against the upstream commit `17fcce14`; if the recipe fails for you, the upstream PyPI release may have introduced a wire-format change since the toolkit's vendored vectors. File a toolkit issue + use the version-pinned PyPI archive at `https://pypi.org/project/shamir-mnemonic/0.3.0/`." This is honest about the moving-target risk and gives the reader a recovery path. **Stronger fix:** validate the recipe end-to-end at GREEN-write (run the actual `pip install shamir-mnemonic==0.3.0 && python -m shamir_mnemonic.cli combine ...` against shares produced by `mnemonic slip39 split`) and note "validated 2026-05-14 against shamir-mnemonic 0.3.0 on Linux x86_64" in chapter prose — converts the recipe from a documentation claim to a reproduced-by-author claim.
