# Architect Verdict (brainstorm consult) — BRAINSTORM_pathless_partial_decode — Fable, adversarial

**Persisted verbatim per CLAUDE.md.** Pre-R0 brainstorm consult (the formal SPEC R0 gate follows the written SPEC). Verified vs descriptor-mnemonic HEAD `a39c9d9f` + toolkit consuming md-codec 0.40.0 (vendored). Two findings proven by a runtime probe the reviewer built. VERDICT: **OPEN (1C / 3I)**.

## Bottom line
The brainstorm's headline safety claim (Approach A: relax the decode gate, rely on `expand_per_at_n` as universal backstop) is FALSE as stated, precisely on the funds-sensitive path the design flags. `verify-bundle` never calls `expand_per_at_n`/derive/policy-id on the supplied card, and the "same predicate" claim is demonstrably wrong (the two diverge on an empty-override wire). Approach A is unsafe for `verify-bundle` → explicit gate (C) required there.

## CRITICAL
**C1 — `verify-bundle` loses its ONLY origin-completeness gate under Approach A → verify(PASS) ↔ restore(REFUSE) divergence (funds false-assurance).** Supplied card accepted solely by the decode step (`cmd/verify_bundle.rs:2450/:2846/:3045`); no `expand_per_at_n`/`derive_address`/`to_miniscript`/`compute_wallet_policy_id` runs on the supplied `desc`. The verdict compares EXCLUDE origin by design: multisig `policy_match` = tree/use_site/use_site_overrides only (`:2903-2906`, comment `:2883-2887` "Origins … are EXCLUDED"), pubkeys a sorted multiset (`:2907-2923`); singlesig `xpub_match` first pubkey only (`:3077`); template bundles bind via `compute_wallet_descriptor_template_id` (origin-invariant, `identity.rs:71`). Today a keyed general-policy dead card (`canonical_origin==None`, empty origin) is rejected ONLY by the decode gate. Remove it → card decodes, `policy_match` matches, `verify-bundle` emits `result: ok`/exit 0 — for a card `restore --md1` still REFUSES (`expand_per_at_n` fail-close `restore.rs:3181`). User verifies backup as GOOD then cannot restore. Same class as v0.76/v0.79 false-passes. NOTE asymmetry: md-cli's own `md verify` byte-compares the full re-encoded payload incl. origin (`md-cli/src/cmd/verify.rs:37-45`) → SAFE; the toolkit's origin-excluding STRUCTURAL compare is NOT. **This is an internal contradiction in the design** (Component 4 states the requirement; Approach A's mechanism can't satisfy it — the mechanism isn't on this path). REQUIRED: explicit partial gate on `verify-bundle` — after decode, query unresolved-origin indices; if non-empty → `result: partial`/exit 4, never pass, regardless of structural match.

## IMPORTANT
**I1 — the two "copies of the same predicate" are NOT the same; `expand_per_at_n`'s check has a hole the decode gate covers.** `validate_explicit_origin_required` (`validate.rs:221-246`): empty `OriginPathOverrides[idx]` → `if !op.components.is_empty()` does NOT continue (`:229`); falls through to `path_decl` (`:234-243`), rejecting only if that's also empty. `expand_per_at_n` (`canonicalize.rs:434-469`): reject includes `sparse_lookup(origin_path_overrides, idx).is_none()` (`:465`) — an empty override IS present → `is_none()` false → reject skipped → returns origin `[]` SILENTLY. Probe (`sh(sortedmulti(2,@0,@1))`, `path_decl=Shared(m/48'/0'/0'/1')`, empty `OriginPathOverrides[0]`): decode OK (via path_decl fallback), expand OK with resolved `@0=[]`, policy-id `e54fed29…` vs no-override ref `eb6d5e10…` DIVERGES. Latent bug independent of this feature, but it undermines A: for the COMMON dead-card shape (`Shared(empty)`, no override) expand DOES fail-close (restore/address safe for the realistic case), but the SPEC must (a) build `unresolved_origin_indices()` on VALIDATE semantics not expand's, and (b) add an `EmptyTlvEntry`-style reject for empty origin overrides so the two converge. Don't ship a query inheriting expand's blind spot.

**I2 — reframe claim #3 ("origin never surfaced in output today") is WRONG for `--json`.** `md decode --json`/`inspect --json` emit origin via `JsonDescriptor.path_decl` → `format_origin_path` (`md-cli/src/format/json.rs:222-263`); probe: `"path_decl": {"data":"m","tag":"Shared"}`. What's never surfaced is the RESOLVED/canonical-filled origin (the internal `m/86'/0'/0'`). Narrow claim holds; broad claim false. The additive `--json` `partial` field must be coherent with the already-emitted `path_decl:"m"`; keep `wallet-policy-id` omitted on partial.

**I3 — relaxing decode enlarges `repair`'s candidate-validity set (funds-adjacent).** `md repair` (`repair.rs:118`→`decode_with_correction`) + toolkit repair use SUCCESSFUL DECODE as the BCH-correction correctness oracle. A correction producing a dead card is pruned today; post-relaxation it "decodes" partial and could be blessed. Chunked content-id oracle (`chunk.rs:379-387`) + v0.86.0 non-chunked demote mitigate multi-error alias, but the SPEC must decide: a correction resolving to a PARTIAL card is NOT `Blessed`/exit-5 → demote to Unverified/exit-4 (or exclude). Currently unaddressed.
