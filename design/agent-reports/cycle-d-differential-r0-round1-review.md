## Important
- **I1 — Canonicity claim INVERTED; the FOLLOWUP would point the fix the wrong way.** Empirically: md-codec 0.35.1 has NO canonical wrapper for `c:pk_k` in wsh — `canonical_origin::canonical_origin` returns a default-origin only for pkh/wpkh/tr-keypath/wsh(multi|sortedmulti)/sh(wsh(multi)); `wsh(pk(@N))` in EITHER form is a "non-canonical wrapper" needing explicit origin. The encoder + `compute_wallet_policy_id` faithfully encode whatever tree the walker hands them and do NOT normalize `Check(PkK)`↔bare`PkK` (only placeholder-index ordering is canonicalized). So at the md-codec wire level NEITHER tool is "wrong"; the only authority preferring bare-PkK is **descriptor-mnemonic SPEC v0.30 §5.1** ("Walker emits bare `Tag::PkK`/`Tag::PkH`… regardless of context; `Tag::Check` is never emitted wrapping a key leaf on the wire"). Per that SPEC, **the TOOLKIT is the non-conformant side** — and its deviation is DELIBERATE + tested (`walk_check_kept_in_non_tap_context`, parse_descriptor.rs:2568). The FOLLOWUP must state: md-codec defines no canonical wrapper; descriptor-mnemonic SPEC §5.1 picks bare-PkK; the toolkit consciously emits Check(PkK) in wsh/sh; fix direction = **toolkit adopts SPEC §5.1 bare-PkK in non-tap** (drop the tap_context gate at parse_descriptor.rs:603), unless the toolkit has its own SPEC reason — verify first, invert the existing test. NOT "fix md-cli" and NOT "define canonicity" (already defined).
- **I2 — Input-form pairing is load-bearing + under-specified; naïve pairing produces spurious divergences even on expected-match shapes.** Proven: md-cli `--key` takes a BARE xpub with NO `[fp/path]` bracket (a bracket → base58 decode error), derives BIP-32 depth from the xpub (rejects depth≠4 for wsh/MultiSig, depth≠3 for wpkh/SingleSig), takes fingerprint via `--fingerprint @i=HEX` and origin via `--path` — three separate inputs. The toolkit takes ONE concrete descriptor with a mandatory `[fp/path]xpub` bracket. Demo: `wpkh` (a no-Check expected-MATCH shape) DIVERGED on policy-id (`e25d7948` vs `1c0170fe`) purely because md-cli got `--fingerprint` but not `--path`; adding `--path m/84'/0'/0'` made them identical (`1c0170fe`==`1c0170fe`). The plan must specify, per corpus entry, the exact md-cli triple (bare depth-N xpub, `--fingerprint`, `--path`) reconstructing the toolkit's `[fp/path]xpub`, with depth-matched xpubs (depth-4 wsh, depth-3 wpkh).
- **I3 — Dual-error vacuity is real; the oracle must guard it.** Illegal `pk(@0)` top-level: both error but with different messages + exit codes (md-cli=1, toolkit=2). A naïve "compare stdout md1" could score two-absent-md1s as Match. The oracle MUST require BOTH tools exit 0 AND emit parseable md1 before any Match/Diverge; else `Expect::BothError`/`Expect::ToolError` (a THIRD state), never silently Match. O3's `Match | Diverge` table is missing this arm — add it.
- **I4 — md1-string equality (O1) is the WRONG primary; wallet-policy-id (O2) must be primary.** The tools chunk differently: md-cli `encode` → single-string md1; toolkit `bundle` → 3-chunk md1 (different HRP-data prefixes). O1 raw-string compare would ALWAYS "diverge" on chunking. Robust identity = `wallet-policy-id` (decode both → `md inspect` → wallet-policy-id): cleanly distinguishes real walker divergence (`58d18033` vs `9ad78e4f` for wsh(pk)) from agreement (`1c0170fe`==`1c0170fe` for wpkh). Make O2 primary; O1 secondary/informational (or normalize both to unchunked single-string before compare). `wallet-descriptor-template-id` (key-independent tree hash) is even better walker-ISOLATING (differs `9208f590` vs `ef980fcc` for wsh(pk), matches `45775d4d`==`45775d4d` for wpkh) — add O2' = template-id compare.
