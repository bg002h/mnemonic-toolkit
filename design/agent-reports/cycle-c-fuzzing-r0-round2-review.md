## Important
- **[I3-residual] The `md1_decode_payload` target as folded crashes vacuously on its first large input — there is no `total_bits > len*8` "validation" to exercise; there is a `debug_assert!` that aborts.** `decode_payload(bytes, total_bits)` (md decode.rs:15, `pub use`-exported at lib.rs:45) immediately calls `BitReader::with_bit_limit(bytes, total_bits)` which carries `debug_assert!(bit_limit <= bytes.len() * 8)` at bitstream.rs:114. cargo-fuzz's default build is **release-with-debug-assertions** (its `--help`: "`cargo build --release`, but with debug assertions and overflow checks enabled"; `cargo-fuzz-0.13.2/src/project.rs:259-260` pushes `-Cdebug-assertions` whenever `!release || debug_assertions || careful_mode`). I built a scratch crate path-dep'ing md-codec with `-Cdebug-assertions=on` and called `decode_payload(&[0x01,0x02,0x03], 65535)` (the exact shape the spec mandates: first 2 LE bytes `0xffff` = 65535, remainder = 3 bytes): **it panics at bitstream.rs:114 `assertion failed: bit_limit <= bytes.len() * 8`**, it does NOT return `Err`. Since a random 2-byte LE prefix is almost always `> remainder.len()*8`, the target aborts on essentially the first exec, permanently, on a non-bug. The spec text at lines 124-125 ("UNCLAMPED, also exercising total_bits > len*8 validation") and the round-1 [I3] claim ("it should return Err, not panic, on total_bits > len*8") are both incorrect. **Fix:** clamp the candidate to the data budget — `total_bits = candidate.min(remainder.len()*8)` — which still exceeds P3's `len*8` pin (it fuzzes every partial-byte trailing-bit count `0..=len*8`, the real blind spot P3 leaves), and `read_bits` correctly returns `Err(BitStreamTruncated)` on a genuine short read (bitstream.rs:129-133), so clamped values produce findings-free truncation errors. Drop the "exercises total_bits > len*8 validation" rationale (false). If exercising the `>len*8` path is genuinely wanted, it must go through a wrapper that maps the assert to an `Err`, not the raw `decode_payload`.

## Minor
- **[gen-corpus × splitter validity gate]** The validity gate (spec line 179, "`gen-corpus` MUST assert every committed valid-class seed *decodes Ok*") is phrased for single-string targets and is ambiguous for the four **splitter** targets (md1_reassemble, md1_decode_with_correction, ms1_combine, mk1_decode), whose corpus seeds must be `chunk0\nchunk1\n…` (chunks joined by the `\n` sentinel), not a single decodable string. Pin the gate per-target: gen-corpus's validity assertion must run *the same split-then-call the target uses* (e.g. for md1_reassemble: split on `\n` → `reassemble(&parts)` is Ok), not "the raw bytes decode Ok." Also specify gen-corpus joins chunks **between** (not a trailing `\n` after the last) so single-chunk-into-splitter seeds don't acquire an empty trailing `""` part. `\n` (0x0A) is confirmed outside the bech32 alphabet/HRP/separator, so it cannot collide with valid card content — the splitter itself is sound; only the corpus-writing convention needs pinning.
- **[I1 ms coordinate left implicit]** The apply-details idempotence text spells out the md coordinate as "(chunk_index, position) in the input data-parts" (line 121) but the ms1_decode reference (line 131) just says "with apply-details idempotence [I1]" — ms's `CorrectionDetail` has **no `chunk_index`** (single-chunk; decode.rs:119-128 `{position, was, now}`), so the implementer must use the `position`-only coordinate into the single data-part. It's self-documenting from the struct (no chunk_index to misuse) but a one-clause note ("ms: single data-part, position-only; no chunk_index") would remove the asymmetry. Also note both `position` fields are **post-HRP-and-separator** data-part offsets (md chunk.rs:406-410, ms decode.rs:121-123) — applying `now` requires offsetting past `md1`/`ms1`; worth one explicit sentence so an implementer doesn't index the raw chunk string.
- **[C1 exclusion sufficiency rests on the 8-char window — worth a one-line note]** The exclusion set `Codex32(_) | WrongHrp{..}` is sufficient, but the *reason* deserves a sentence in the spec: of the 18 `ms_codec::Error` variants, only those two can emit an **≥8-char contiguous** input echo (`Codex32` Debug-wraps codex32's full-string variants `InvalidChecksum`/`MismatchedHrp`/`MismatchedId`; `WrongHrp.got` echoes the observed HRP region, which on a `1`-shift mutation can be a long secret prefix). Every other non-excluded variant echoes at most a `[u8;4]` tag (4 chars) or a single char — below the 8-char window threshold — so they cannot leak and need no exclusion. Documenting this makes the exclusion set's *minimality* auditable when the FOLLOWUP later deletes entries.

## Round-2 additional checks
- **(1) Sentinel `\n` splitter / gen-corpus interaction:** `\n` (0x0A) confirmed outside the bech32 alphabet (`qpzry9x8gf2tvdw0s3jn54khce6mua7l`), HRP, and separator — cannot collide with valid card content. Single-chunk targets take the whole input (no split) so trailing-`\n` is harmless. The only gap is the gen-corpus validity-gate wording for splitter targets → **Minor** above (mechanical fix).
- **(2) Fixed-point oracle equality level — PartialEq availability:** all four value types implement `PartialEq, Eq`: md `Descriptor` (encode.rs:16), mk `KeyCard` (key_card.rs:23), ms `Tag` (tag.rs:11), ms `Payload` (payload.rs:28). The oracle compiles for every target. ✓
- **(3) decode_with_correction extra panic surface vs decode:** read the full BCH path (decode.rs:221-279). `decode_regular_errors` returns `Option` → mapped to `Err(TooManyErrors)`; correction loop bounds-checks `pos >= corrected.len()` before indexing (line 246); all alphabet indexing masks `& 0x1F`; defensive re-verify catches pathological 5+-error patterns. `parse_ms1_symbols` uses `.unwrap_or`/`.ok_or` only. **No reachable unwrap/panic on arbitrary input** — decode_with_correction adds no panic surface over decode. ✓
- **(4) Phasing standalone-shippability:** no shared/common harness crate anywhere (grep clean). Each `<repo>/fuzz/` is its own `[workspace]` path-dep'ing only its own codec (splitter/oracle/scanner duplicated per repo — correct tradeoff for independent shipping). Each phase (md → ms → mk) is genuinely standalone. ✓
- **(5) Other vacuity/false-positive/unbuildable risks:** the I3 debug-assert is the one live hit (above). No other target has a reachable non-finding panic; all re-encode oracles compile; the no-leak oracle is non-vacuous; CI path filters non-overlapping (verified in round 1, unchanged).
