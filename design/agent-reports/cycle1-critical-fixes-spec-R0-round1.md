## CRITICAL FINDINGS

### C1 — H13 §3 decision "FAITHFULLY REPRESENT" is wrong; it must flip to **REJECT** (funds-safety)

**What's wrong.** The spec's load-bearing §3 decision is: *because the md1 wire's `Alternative` carries a `hardened: bool` and it's serialized, we should faithfully encode `<0';1'>` (set the bit) rather than reject it.* The reasoning chain is: wire-can-carry-it ⇒ faithful-encode is "strictly better than rejecting" (§3.2, §3.3). **This is a false inference.** The spec itself flags the load-bearing risk as Q-H13-1 ("R0 MUST validate that md-codec `derive_address` honors `Alternative.hardened` … otherwise we'd faithfully store a value nothing can derive, which would argue for reject-instead"). I validated it. **The answer is: md-codec does NOT derive it — it rejects it, by design, and CANNOT do otherwise.**

**Where / the evidence (all `origin/main` @ `54dd765`, md-codec):**
1. `derive.rs:105-107` — `derive_address` does, as its **first** pre-flight, `if crate::to_miniscript::has_hardened_use_site(self) { return Err(Error::HardenedPublicDerivation); }`. The doc-comment at `derive.rs:98-104` states this is the BIP-32 hardened-public-derivation refusal covering "a hardened wildcard OR any hardened multipath alternative, anywhere."
2. `to_miniscript.rs:101-108` — `use_site_is_hardened` returns true if **any** multipath alternative has `a.hardened`. So a `<0';1'>` card trips the refusal.
3. `to_miniscript.rs:277-291` — the inner single-chain resolver `use_site_to_derivation_path` (reached via `to_miniscript_descriptor` → `build_descriptor_public_key`, `to_miniscript.rs:148-163`) ALSO returns `Error::HardenedPublicDerivation` for a hardened alt. So the rejection is redundant/defense-in-depth: **there is no code path in md-codec that derives an address from a hardened multipath alt.** (`pub fn derive_address` is the only address entry point — `derive.rs:92`; grep confirms no alternate xpriv/signing derive path consumes `Alternative.hardened` for derivation.)

**Why this is structurally forced, not a fixable omission (the BIP-32 fact):**
- md1 cosigner keys are **xpubs** — `tlv.rs:14,29-32`: the Pubkeys TLV stores "32-byte chain code || 33-byte compressed pubkey (65 bytes)". **No private keys.** An md1 backup is watch-only for its cosigners.
- BIP-32 forbids hardened child derivation from a public key (you need the private key). **BIP-389 confirms this** (fetched live, `bip-0389.mediawiki`): the `<...>` group *syntactically* permits `h`/`H`/`'` markers, BUT "hardened derivation requires private keys, so `xpub/<0h;1h>` would be technically invalid since public keys cannot perform hardened derivation (per BIP 32 rules)." So a hardened-multipath alt on an xpub-only key is **fundamentally underivable** — this is *why* md-codec rejects it, *why* Bitcoin Core rejects it ("not a valid uint32"), and it cannot be "fixed" by making md-codec smarter.

**End-to-end un-derivability (toolkit side, `origin/master` @ `4d5872ed`):**
- `md address` calls `.derive_address(...)` (md-cli `cmd/address.rs:38-39`) → `HardenedPublicDerivation`.
- toolkit `restore --md1` has its OWN pre-check: `restore.rs:2779-2784` `if md_codec::to_miniscript::has_hardened_use_site(&d) { return Err(ToolkitError::ModeViolation{…"watch-only addresses cannot be derived from it … Faithful reconstruction is not supported"}) }` (refuses before either derivation arm); the plain wsh/sh-wsh arm at `restore.rs:2937` calls `d.derive_address(0, i, …)` which would reject anyway. Regression test `hardened_override_is_not_restorable` (`restore.rs:3464`) asserts the refusal.
- **rust-miniscript itself (empirically, against the exact pinned `rev 95fdd1c5` in toolkit `Cargo.toml:29`):** a hardened-multipath descriptor `from_str`s and `into_single_descriptors()`-splits fine, but `at_derivation_index`/`derive_at_index` on the branch selecting a hardened alt errors verbatim: **`"key with hardened derivation steps cannot be a DerivedDescriptorKey"`** (a hardened `/*h` wildcard poisons ALL branches; a hardened multipath alt poisons only its branch). So even bypassing md-codec's pre-flight, rust-miniscript refuses at derivation — the BIP-32 prohibition is enforced at every layer.

**The constellation ALREADY classifies these cards as unrestorable.** The toolkit has an *engrave-time* advisory `unrestorable_advisory.rs:102` that fires the SAME `has_hardened_use_site` predicate to warn the operator the card cannot be restored. So the current architecture is: faithful-encode the bit + uniform derive-time refusal + engrave-time "unrestorable" advisory. This is a strong additional argument for **encode-time REJECT**: the constellation already knows the card is unrestorable at creation time (it warns about it) — failing closed at encode is strictly cleaner than emitting a faithful-but-permanently-undecodable artifact and deferring the refusal to restore.

**The spec's own primary input agrees with REJECT, and the spec mis-reads it.** The bug-hunt report's *empirical* H13 finding (constellation-bughunt-2026-06-20.md, the differential-oracle wave) says verbatim: *"Core **rejects** `<0';1'>` ('not a valid uint32') → the correct behavior is to **ERROR, not collapse**."* The spec's §3.3 "Core-parity nuance" tries to reconcile this by arguing "the m-format md1 wire is a different, richer representation" that *should* faithfully record intent and let a downstream tool "handle the hardened-multipath semantics at *its* boundary." **There is no such downstream tool** — the constellation's own derive/restore/address paths are that boundary, and they all refuse. Faithful-encode therefore manufactures exactly the failure mode the spec claims to avoid: *a steel card that records a wallet no participant — including this very toolkit — can derive an address for, let alone spend from.* That is a worse funds-safety outcome than the current silent collapse only in that it's now *silently un-restorable* instead of *silently wrong-addressed*; either way the user's funds-intent is not faithfully reconstructable.

**Why it blocks the gate.** §3 is the entire H13 workstream's design premise. The concrete change surface (§3.4: widen both lexers to set `Alternative.hardened` from the captured bit), the SemVer call (§3.5), the differential row (§6.1 row H13 "round-trip yields the intended hardened-derived addresses"), and the unit tests (§6.2 "make_use_site_path → Alternative{hardened:true}") are ALL built on the faithful-encode premise. Proceeding would implement, test, and ship a card format whose hardened-multipath instances are un-restorable — a CRITICAL funds-safety regression dressed as a fix.

**Required fix (flip to REJECT, which the spec already scaffolds as the fallback):**
1. **Decision flips:** H13 = "extend both multipath lexers to **capture** `'`/`h` (so the hardening is no longer silently invisible), then **reject** a hardened multipath alternative with a typed error" (`CliError::TemplateParse` in md-cli; `ToolkitError::DescriptorParse` in toolkit — both exist, verified `md-cli/src/error.rs:6`, `toolkit/src/error.rs:123`; no new variant, no alphabetical-ordering concern). The key correctness property is **fail-closed, loud, never silent-collapse** — which the spec's §3.3(b) already endorses; it just must apply to the well-formed-`<0';1'>` case too, not only to malformed markers.
2. **`make_use_site_path`:** md-cli already returns `Result` (`template.rs:220`); the toolkit mirror returns plain `UseSitePath` (`parse_descriptor.rs:223`, 2 call sites `:193`,`:197`) and must be widened to `Result` to surface the typed reject. (The spec anticipates this thread-an-error work in §3.4.2 — keep it.)
3. **Differential row (§6.1 H13):** assert a typed ERROR at encode/parse (exit ≠ 0) for `<0';1'>` / `<0h;1h>`, NEVER a bare-key collapse (`bcrt1qq0kxm9…`). Keep the clean-negative: non-hardened `<0;1>` MUST still round-trip correctly (do not over-reject). This is exactly the spec's "If Q-H13-1 resolves to reject" branch at §6.1 — promote it to the primary assertion.
4. **No md-codec change** still holds (the wire bit stays; we simply never *write* a hardened multipath from the lexer — md-codec's existing refusal is the backstop). **No companion md-codec FOLLOWUP** is needed for a derive gap, because there is no gap to fix: md-codec's refusal is correct BIP-32 behavior, not a bug. (The spec's §3.4.3 conditional "If Q-H13-1 reveals derive_address doesn't honor it, THAT becomes a separate md-codec workstream" should be struck — md-codec is already correct.)
5. **SemVer unchanged:** md-cli MINOR is still right (behavior changes for a non-empty input class: silent-collapse → typed error). Toolkit MINOR via the pin bump is still right.

**Net:** the *mechanics* the spec describes (widen the regex to see `'`/`h`, stop silently dropping) are mostly reusable; only the terminal action flips from "set the bit and encode" to "see the marker and reject." This is a smaller, safer change than faithful-encode and matches Core, md-codec, BIP-32, and the bug-hunt's empirical recommendation.

## IMPORTANT FINDINGS

### I1 — H1 §5.2/§5.3: the hand-rolled four-part predicate is over-engineered; the codebase already has an exact, index-aware structural primitive (`tree ==`)

**What's suboptimal.** §5.3 proposes a bespoke predicate:
```
md1_policy_match := tags_equal(…) && thresholds_equal(…) && wrapper_equal(…) && pubkey_binding_equal(…)
```
and Q-H1-1 asks whether an existing primitive can be reused. **There is one, and it subsumes all four conjuncts in a single derived comparison:** the decoded `md_codec::Descriptor` and its `tree: Node` derive full `PartialEq/Eq` end-to-end:
- `md-codec/src/encode.rs:16-28` — `Descriptor { n, path_decl, use_site_path, tree, tlv }` derives `PartialEq, Eq`.
- `md-codec/src/tree.rs:8-15` — `Node { tag, body }` derives `PartialEq, Eq`.
- `md-codec/src/tree.rs:17-76` — `Body::MultiKeys { k: u8, indices: Vec<u8> }` carries **both threshold `k` AND the index-ordered per-slot binding** (the `@i` placeholders in slot order); `Body::Tr`, `Body::Variable` likewise. So `tree == tree` is **order-significant** on `indices` — exactly the index-aware binding §5.2.4 calls for, for free.
- `md-codec/src/tag.rs:14-35` — `Tag` (Multi/SortedMulti/MultiA/SortedMultiA/Wsh/Sh/Tr…) derives `PartialEq, Eq`; the wrapper IS the root `tree.tag`.

So `expected_md_decoded.tree == desc.tree` already compares **Tag + threshold k + wrapper + index-aware slot-order + nested policy structure** in one shot — covering §5.2.4 (1)-(5) without a hand-rolled traversal. This is materially simpler, less bug-prone, and harder to drift than four separate helpers.

**Why this is Important, not Minor.** A funds-safety verification predicate that is hand-assembled from four conjuncts is itself a place a future maintainer can introduce a false-GREEN (e.g. forget the wrapper conjunct, or mis-handle the `sh` vs `sh(wsh)` case below). Leaning on the derived `PartialEq` removes that surface. The spec should adopt `tree ==` (plus `use_site_path` equality if use-site divergence must also bind) as the core compare, and only fall back to extracting individual fields for **mismatch-detail strings** (which are cosmetic, not verdict-bearing).

**Critical sub-nuance the spec must encode (the `sh` vs `sh(wsh)` trap):** `md-codec/src/decode.rs:35-38` documents that root `Tag::Sh` covers BOTH `sh(multi)` and `sh(wsh(multi))` — they share the root tag and differ ONLY in the nested body (one extra `Wsh` child node). A naive "compare root tag + k" predicate would treat `sh(wsh(sortedmulti(2)))` and `sh(sortedmulti(2))` as equal — a false-GREEN, and `sh(wsh(...))` is one of the spec's own §6.1 row-H1 discriminator cases. **`tree ==` handles this correctly** (the nesting differs); a hand-rolled `wrapper_equal` that only looks at the root tag would NOT. This is a strong additional argument for the derived-`PartialEq` approach and against the bespoke predicate. The spec must call this out explicitly so the implementer does not hand-roll a root-tag-only wrapper check.

**On Q-H1-1 (the disqualified primitive):** the spec's instinct to avoid `compute_wallet_policy_id` is **correct and verified.** `md-codec/src/identity.rs:194-237` hashes the per-`@N` origin path verbatim (`:194-197`, re-emitted `:209-210`, concatenated `:235-237`) AND is fingerprint/xpub-presence-significant (`:219-230`) — so it would FALSE-FAIL across origin elision and fingerprint/xpub-presence variance (bug-hunt L14 confirmed correct; its "stable across origin elision" doc-comment at `identity.rs:108-115` is a lie, and the test `walletpolicyid_stable_across_origin_elision` is vacuous per L17). Keep `compute_wallet_policy_id` OUT. Note for completeness: the **keyless-template** arm at `verify_bundle.rs:882` correctly uses `compute_wallet_descriptor_template_id` (origin-INVARIANT, `identity.rs:73-106`) — that is the right precedent for *that* arm, but it is template-id (no keys), not the keyed-policy compare; do not conflate. The keyed-multisig arm should use `tree ==` on the decoded keyed descriptors.

**On Q-H1-2 (compare per-cosigner origin paths?):** with `tree ==` as the verdict primitive, the structural compare deliberately does NOT bind origin paths (the tree carries placeholder `indices`, not origins; origins live in `path_decl`/TLV). The spec's lean — let the §6 differential-oracle row (Core-derived address equality) arbitrate "same wallet," with the structural compare as the fast gate — is sound, BECAUSE binding origins in the fast gate would re-introduce the L14 origin-elision brittleness. **Confirmed: do NOT compare origin paths in `md1_policy_match`.** The oracle is the funds-truth backstop; the structural compare is necessary-not-sufficient and that layering is adequate (the H12 fix independently closes the origin-divergence wrong-address class, so the gate need not also police origins to be funds-safe).

**Required fix:** replace the §5.3 four-conjunct predicate with "`md1_policy_match := decoded_expected.tree == decoded_supplied.tree` (and `use_site_path` equality if use-site binding must be pinned)"; add an explicit note that this is chosen specifically because it (a) is index-aware on `Body::MultiKeys.indices` and (b) distinguishes `sh(multi)` from `sh(wsh(multi))` via nested-body equality, which a root-tag-only wrapper check would not. Retain `extract_multisig_threshold` (`bundle.rs:1197`, `pub(crate)`) only for human-readable mismatch detail, not the verdict.

### I2 — H12 §4.3/Q-H12-2: the third call site (`descriptor_intake.rs`) does NOT have an md-codec `Tag` in hand — it has a rust-miniscript `Descriptor`; the "thread the `Tag`" framing is wrong there

**What's imprecise.** §4.3 and Q-H12-2 frame the fix as "thread the descriptor's `Tag`/template into the helper" and ask "do all three call sites have the `Tag`/template in hand?" Verified answer: **two of three do; the third does not, and needs a different taproot-detection mechanism.**
- `bundle.rs:1445` call site: operates on `resolved_placeholders` (md-codec parse) and has the md-codec descriptor `tree` in scope via the `canonicity_probe = parse_descriptor(...)` at `:1396` (its `.tree` is used by `canonical_origin(&canonicity_probe.tree)` at `:1398`, immediately before the `compute_default_origin_path` call). So the md-codec `Tag` IS available here — taproot-detection via the md-codec tree works.
- `verify_bundle.rs:1373` mirror: same helper, same symmetric path — fixed in lockstep once the helper is taproot-aware. OK.
- `descriptor_intake.rs:324,345` call sites: these live in `parse_literal_xpub`, which operates on `parsed = MsDescriptor::<DescriptorPublicKey>::from_str(descriptor_str)` (`descriptor_intake.rs:297`) — a **rust-miniscript `Descriptor`, NOT an md-codec tree/`Tag`.** There is no md-codec `Tag` at this site. Taproot-detection here must be done on the miniscript descriptor (e.g. `matches!(parsed, miniscript::Descriptor::Tr(_))`), not by threading an md-codec `Tag`.

**Why Important.** If the implementer follows the spec literally ("thread the `Tag`") they will discover at the third site that there is no `Tag` to thread, and may either (a) plumb an unnecessary md-codec re-parse, or (b) — worse — leave `descriptor_intake.rs` at the literal `2` because "no Tag available," re-introducing the exact H12 defect on the xpub-search intake path (which `compute_default_origin_path` does NOT cover — `descriptor_intake.rs` has its own `bip48_default_path` helper at `:410`). The spec's "fix-the-class, kill the literal `2` at every site" intent is correct, but the mechanism differs per site and the spec must say so.

**Required fix:** §4.3 must specify the taproot-detection mechanism PER call site: md-codec `Tag::Tr` test (or `template.bip48_script_type()`) for `bundle.rs`/`verify_bundle.rs`; rust-miniscript `Descriptor::Tr(_)` (or equivalent `is_taproot` on the miniscript descriptor) for `descriptor_intake.rs`. The §4.3 recommendation that the helper "receive a precomputed `script_type: u32` from one shared caller-side computation" is the right shape — but each caller computes that `u32` from the descriptor representation IT holds (md-codec tree vs miniscript descriptor), and `template.rs::bip48_script_type()` remains the 1/2/3 authority only for the template/`Tag` callers, not the miniscript caller. (The miniscript caller maps `Tr → 3` directly.) Q-H12-2's "do all three call sites have the `Tag` in hand?" is now answered: **no — `descriptor_intake.rs` has a miniscript `Descriptor`, not a `Tag`** — fold this into §4.3.

## RESOLUTIONS TO THE LOAD-BEARING QUESTIONS

### Q-H13-1 (HIGH — decides H13's whole approach): **REJECT, not represent.**
md-codec's `derive_address` does NOT honor `Alternative.hardened` for derivation — it **refuses** it (`derive.rs:105-107` → `Error::HardenedPublicDerivation`; also `to_miniscript.rs:283-285`). This is correct BIP-32 behavior, not a gap: md1 cosigner keys are xpubs (`tlv.rs:14,29-32`), and BIP-32 forbids hardened derivation from a public key (BIP-389 live-confirmed: `xpub/<0h;1h>` is "technically invalid since public keys cannot perform hardened derivation"). End-to-end ALL THREE layers reject: `md address` (`md-cli cmd/address.rs:38-39`) → `HardenedPublicDerivation`; toolkit `restore --md1` (`restore.rs:2779` `ModeViolation` pre-check, `:2937` derive) → refusal; rust-miniscript directly (empirically, pinned `rev 95fdd1c5`) → `"key with hardened derivation steps cannot be a DerivedDescriptorKey"` at `derive_at_index`. The toolkit ALREADY warns these cards are unrestorable at engrave time (`unrestorable_advisory.rs:102`, same predicate). **Faithful-encode would manufacture a card the constellation already knows is permanently un-restorable** — the exact "store a value nothing can derive" failure the spec's own Q-H13-1 says argues for reject. The bug-hunt report's empirical finding agrees verbatim ("the correct behavior is to ERROR, not collapse"). ⇒ **C1: flip §3 to REJECT** (capture the `'`/`h` marker so it's no longer silently dropped, then typed-error; never silent-collapse). No md-codec change, no md-codec FOLLOWUP.

### Q-H12-2: **`bip48_script_type()` returns `Some(3)` for `TrMultiA/TrSortedMultiA` — CONFIRMED** (`template.rs:231-237`). The `3'`=P2TR interop target (Sparrow/Coldcard/Jade + Core diff-oracle) is correct AS A DE-FACTO CONVENTION, not BIP-48-standard (`template.rs:243-262` advisory documents this; preserve it). **BUT the `Tag` is NOT available at all 3 sites:** `bundle.rs:1445` and `verify_bundle.rs:1373` have the md-codec tree/`Tag`; `descriptor_intake.rs:324,345` have a rust-miniscript `Descriptor` (`from_str` at `:297`), not a `Tag` — taproot-detection there must be `Descriptor::Tr(_)`, not a threaded `Tag`. ⇒ **I2: §4.3 must specify per-site detection mechanism.** Emitting `3'` is what the diff-oracle proved coordinators expect (full addresses verified against the report).

### Q-H1-1/2: structural compare is sound; **reuse the derived `tree ==` primitive, not a hand-rolled predicate.** `md_codec::Descriptor`/`Node`/`Body`/`Tag` derive `PartialEq/Eq` (`encode.rs:16`, `tree.rs:8,17`, `tag.rs:14`); `tree ==` covers Tag + k + wrapper + **index-aware** slot-binding (`Body::MultiKeys{k,indices}`) + nesting in ONE compare, AND correctly distinguishes `sh(multi)` from `sh(wsh(multi))` (which a root-tag-only wrapper check would NOT — `decode.rs:35-38`). It catches wrong-threshold, unsorted-vs-sorted, script-type drift, and key-order permutation. It does NOT false-positive on origin elision because it compares the keyed `tree`, not the origin-significant policy-id. `compute_wallet_policy_id` IS disqualified (origin/fingerprint/xpub-presence significant — `identity.rs:194-237`; L14 confirmed).
