**Verdict:** NOT GREEN (0C / 5I)
**Reviewer:** opus architect, SHA d9063523
**Dispatched:** 2026-07-06 (Cycle B, SPEC R0 loop round 1). Persisted verbatim before fold per CLAUDE.md.

## Critical (must fix before implementation)
- none

The guard matrix's central safety property is **sound**: any entry pair satisfying conditions 1-7 is byte-identical except the final step (0 vs 1) and the `internal` flag, which — given identical `(fp, full origin path, xpub)` per key, identical script/threshold, and uniform per-key step change — is provably the external+internal chain of the *same* wallet. BIP-389 element-wise expansion of `<recv;chg>` yields exactly those two descriptors, so no wrong-merge (two different wallets) can pass. No direct fund-loss hole in the matrix itself.

## Important (must fix before implementation)

- **[I1] §5 `internal = None` mechanism: the recommended "detect-from-multipath" option is a `--select-descriptor` correctness regression** — `wallet_import/mod.rs:411-442` + fixture `core-multipath-receive-change-pair.json`. The SPEC calls shape-based detection ("multipath body ⇒ `internal = None`") "cleanest," but a **pre-existing** already-`<0;1>` Core entry (test `core_fixture_file_multipath_receive_change_pair_parses` @ `:952`) carries explicit `"internal": false` / `"internal": true` in its JSON, which `parse_entry` reads via `parse_bool_field` today → `Some(false)`/`Some(true)`. Shape-based detection would force BOTH those distinct entries to `None`, so `--select-descriptor active-receive` (new predicate `internal != Some(true)`) returns **2 bundles instead of 1**, and `active-change` likewise — a silent select-semantics regression on a shape the parser accepts today. **Fix:** thread an explicit merge-provenance flag from the pre-pass (option (b)); FORBID shape-based detection. Invariant must be "**pre-pass-merged entry ⇒ None**," not "multipath ⇒ None." Strengthen test #13 to assert `active-receive`/`active-change` on the multipath fixture still each return exactly the one `internal`-flagged entry.

- **[I2] §8.4/§8.10 address oracle is tautological, and verify-bundle FALSE-PASSES on a misfired multisig merge (the Cycle A C1 class)** — `parse_descriptor.rs` (merge output re-enters the same pipeline). If the string replacement rewrites only *some* keys' steps in a multisig (e.g. `wsh(sortedmulti(2, k1/<0;1>/*, k2/0/*))`), BIP-389 expands the change chain to `(k1/1, k2/0)` — WRONG change addresses — and verify-bundle re-derives the *same wrong descriptor* → PASS (exactly the C1 false-pass Cycle A closed). The §8.4 oracle "same addresses as a **hand-authored** `<0;1>/*` import" is near-tautological: a hand-authored `<0;1>` is the same construction the merge produces. **Fix:** (a) the oracle MUST anchor on **independently-derived** addresses from the ORIGINAL split descriptors — external addrs from `.../0/*`, internal addrs from `.../1/*` (rust-miniscript `at_derivation_index().address()` or a known vector), asserted equal to the merged bundle's chain-0/chain-1 addrs, for BOTH chains; (b) §4.3 must specify the replacement as **structurally anchored per parsed key use-site** (replace ALL keys' steps — never a first-only/`replacen(…,1)`); (c) test #10 ("addresses verified") must run this same original-anchored oracle, not a bare parse check.

- **[I3] Taproot (`tr`) is unscoped and untested, yet Core exports it** — fixture `core-bip86-mainnet.json` is `tr([b8688df1/86'/0'/0']xpub…/<0;1>/*)`. A bip86 wallet's *split* export is `tr(xpub/0/*)` + `tr(xpub/1/*)` and MUST merge to `tr(xpub/<0;1>/*)`; §4.2's uniformity condition (cond. 7) says "multisig" and never addresses a `tr`'s **internal key + tapscript-leaf keys**, where the "all keys uniform step change" invariant and the ALL-keys replacement (I2) are equally funds-critical. §8 has zero `tr` cases. **Fix:** (a) state that cond. 7 uniformity + the per-key replacement apply to a `tr` internal key AND every tapscript-leaf key; (b) add a bip86 single-key `tr` receive/change merge test; (c) either add a script-path `tr` pair test (internal + leaves uniform, addresses verified) OR explicitly declare script-path `tr` OUT of scope so it falls to the floor reject. Leaving it undefined means a shape Core actually emits is untested through a funds-critical merge.

- **[I4] §4.1 grouping-key definition is self-contradictory (central algorithm ambiguity)** — §4.1 defines the "merge key" to INCLUDE "the use-site final step(s) per key," but the entire purpose is to group entries that are identical EXCEPT the final step. As written, receive (`/0/*`) and change (`/1/*`) have *different* merge keys and would never group (feature fails safe-but-broken; only test #2 would catch it). **Fix:** state unambiguously that the **grouping key** = (script template, threshold, ordered per-key `(fp, full origin path, xpub)`, wildcard-hardness) **EXCLUDING** the final step; the differing final steps + disagreeing `internal` flags are the **within-group discriminant** (conditions 3-5). Make the comparison explicitly **positional/ordered** (not set-based) so a swapped-order `multi` cannot group.

- **[I5] Open "R0 to confirm/finalize" decisions on behavior-relevant items cannot pass an R0-to-0C/0I gate** — §4.3 (range-on-merge) and §5 (mechanism) and §7 (message/exit path) are left open. The funds/behavior-relevant one is **range**: it must be LOCKED and must NEVER block a merge — receive and change chains legitimately carry different per-chain scan state (`range`/next-index), so a range difference blocking the merge would break real Core wallets. **Fix:** lock "merged `range` = union/widening; a range difference never blocks the merge." (§5 mechanism is folded into I1; §7 message/token is Minor — see M7.) An R0-GREEN SPEC carries no open design decisions on merge eligibility.

## Guard-matrix adversarial findings
- Keys equal but origins differ (`84'/0'/0'` vs `84'/0'/1'`): **SAFE** — cond. 2 (identical full origin path) blocks; no merge → floor reject.
- Same xpub, different fingerprint: **SAFE** — cond. 2 (identical fp) blocks.
- Multisig key-order permutation (swapped-order `multi`): **SAFE** — grouping key is the ORDERED key list → different group; **but** make "ordered/positional, not set" explicit (I4) and add a synthetic test.
- `sortedmulti` vs `multi`: **SAFE** — cond. 1 (same template) blocks.
- Taproot `tr` (single-key bip86 + script-path): **HOLE** — unscoped/untested for a shape Core emits; internal-key+leaf uniformity + ALL-keys replacement undefined (I3, ties to I2).
- Nested `sh(wsh(...))`: **SAFE if replacement is per-key-anchored** — structurally like wsh; covered by the I2 replacement/oracle fix; add a test.
- Differing ranges: **SAFE but MUST NOT BLOCK** — per-chain scan state legitimately differs; lock union/widening (I5).
- Same xpub across ≥3 entries sharing the key: **SAFE** — cond. 6 (exactly two) → ambiguous → NOTICE + no merge.
- Receive with no change (lone `/0/*`): **SAFE** — floor reject (test #6).
- Hardened final step `/0'/*` or mismatched wildcard-hardness `/*` vs `/*'`: **SAFE (fail-closed) but messy** — lexer rejects downstream; exclude explicitly at cond. 3 for a clean message (M4).
- Step values ≠ 0/1 (`/5/*`+`/6/*`): **SAFE** — actual-values invariant + test #14; element-wise expansion still equals the two entries.
- Network prefix mismatch (mainnet xpub / testnet tpub): **SAFE** — cond. 2 (identical xpub string) blocks before `network_from_origins`.
- Two-receive (both internal:false, `/0/*`+`/2/*`): **SAFE** — cond. 5 (flags disagree) blocks.
- Misfired multisig replacement (only some keys rewritten): **HOLE** — verify-bundle FALSE-PASSES (Cycle A C1 class); requires original-anchored address oracle + all-keys replacement (I2).
- Corrupt input checksum on a mergeable pair: **SAFE (no fund-loss) but not fail-closed** — recompute masks a body-correct/checksum-wrong entry (M9).
- Vec-level leftover (remove one of pair, insert merged): **SAFE per §4.4** — but state "remove both, insert one" explicitly (M8).
- BIP-389 expansion equals union of split addrs: **SAFE (mathematically holds)** for all guarded cases — element-wise `<recv;chg>` = {receive desc, change desc} exactly; the `<;>` chars are in the BIP-380 checksum INPUT_CHARSET (confirmed feasible by the green `core_multipath_split_to_receive_change` test validating a `<0;1>` checksum).
