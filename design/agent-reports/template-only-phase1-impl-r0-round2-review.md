## I1 — CLOSED

The emit gate was changed from a TOP-LEVEL-only multi-tag guard to refuse with `TemplateFormUnsupportedShape` iff `cli_template_from_tree(&descriptor.tree).is_none()` (`synthesize.rs:1009-1018`, within `synthesize_template_descriptor`). All four sub-checks verified:

- **(a) `cli_template_from_tree` returns `None` for `wsh(multi/sortedmulti)` at ANY depth, `Some` ONLY for single-key pkh/wpkh/tr-keypath.** Body (`synthesize.rs:228-237`):
  ```
  (Tag::Pkh, Body::KeyArg{..}) => Bip44 ; (Tag::Wpkh, Body::KeyArg{..}) => Bip84 ;
  (Tag::Tr, Body::Tr{tree:None,..}) => Bip86 ; _ => None
  ```
  Verified against md-codec 0.37.0 `tree.rs:18` `enum Body`: a `wsh(sortedmulti(1,@0))` top node is `tag=Wsh, body=Children([...])` (`Body::Children` is the fixed-arity variant; `Wsh` is not a match arm) → falls to `_ => None`. The nested `SortedMulti` carries `Body::MultiKeys` but is never inspected because the top tag already misses. So a nested-multi 1-of-1 is `None` → refused. Single-sig pkh/wpkh/tr-keypath → `Some`. Confirmed.
- **(b) the two new descriptor-mode tests exercise the previously-uncovered n==1 path and are RED-first.** `template_form_refuses_nested_{sortedmulti,multi}_1of1_descriptor_mode` (`cli_bundle_md1_template_form.rs`, added in `d8b6eca`) drive `bundle --md1-form template --descriptor "wsh(sortedmulti(1,[704c7836/48'/0'/0'/2']xpub.../<0;1>/*))"` → assert `.failure().code(2)`. Confirmed RED-first against the pre-fold gate (`git show e59e4f7:…/synthesize.rs`): the old guard checked `descriptor.tree.body` for `MultiKeys|Variable` and `descriptor.tree.tag` for `Multi|SortedMulti` — TOP node only. For `wsh(sortedmulti(1,@0))` the top is `Wsh/Children`, so neither arm fired; `canonical_origin(&tree)` returns `Some(m/48'/0'/0'/2')` (md-codec 0.37.0 `canonical_origin.rs`: the `(Wsh, Children) if len==1 && is_wsh_inner_multi` arm) and `n==1` → a keyless MULTISIG template WAS emitted (exit 0). Under the old gate these tests would have failed. The new classifier refuses → they pass.
- **(c) existing accepts/refuses UNCHANGED.** The `n != 1` guard (`synthesize.rs:996-1004`) and the `canonical_origin().is_none()` guard (`synthesize.rs:1020-1029`) are retained verbatim around the new classifier check. For any tree passing `cli_template_from_tree.is_some()` (exactly the three single-sig shapes), `canonical_origin` is also `Some` for those same three (md-codec 0.37.0 `canonical_origin.rs` pkh/wpkh/tr-keypath arms) — so the canonical-origin guard is now redundant-but-harmless on the accept path (defense in depth), never newly-refusing an accepted shape. bip44/84/86 byte-identity, keyless, self-check accepts pass; multisig n>1 (pre-existing template-mode `N>1` guard), bip49 `sh(wpkh)` (`canonical_origin → None`) refuse. Test suite green (bundle template file 11/11, was 9 + 2 new).
- **(d) BOTH emission paths funnel through this one gate.** `synthesize_descriptor` early-returns to `synthesize_template_descriptor` when `md1_form.is_template()` (`synthesize.rs:354-356`). `synthesize_unified` delegates to `synthesize_descriptor` (`synthesize.rs:947`). All bundle.rs callers thread `args.md1_form`: `synthesize_unified` at `bundle.rs:434-442` (single/multi modes); direct `synthesize_descriptor` at `bundle.rs:1720-1726` (descriptor mode), `:1831-1837`, `:2075-2081` (import-json); `import_wallet.rs:1454`. There is no second emit path that bypasses `synthesize_template_descriptor`. Confirmed.

The false docstring round-1 cited is corrected: the new docstring (`synthesize.rs:951-963`) accurately explains the nested-multi-under-`Wsh` miss and why `cli_template_from_tree` closes it.

## I2 — CLOSED + no new bug

`--origin <path>` added to `VerifyBundleArgs` (`verify_bundle.rs:73`, `#[arg(long = "origin")] pub origin: Option<String>`) and threaded into `verify_singlesig_template`. All six sub-checks verified:

- **(a) the `--origin` xpub derivation matches restore.** Verify (`verify_bundle.rs:529-560`) and restore (`restore.rs:289-307`) both: parse `origin_str.trim_start_matches("m/").trim_start_matches('m')` with raw-string fallback (`DerivationPath::from_str`); call the SAME `crate::derive_slot::derive_bip32_from_entropy_at_path(entropy, passphrase, language, network, &path)`; build a `ResolvedSlot { xpub: acct.account_xpub, fingerprint: acct.master_fingerprint, path: acct.account_path.clone(), entropy: None, ... }`. `derive_bip32_from_entropy_at_path` (`derive_slot.rs:65-111`) returns `account_path: path.clone()` and `account_xpub` derived at that exact path. Byte-identical derivation → verify and restore agree. Pinned by test `verify_template_bundle_recomposes_under_origin_matches_restore`: asserts `v_desc == r_desc` (verify `--origin m/84'/0'/7'` descriptor == restore `--origin m/84'/0'/7'` descriptor) AND `v_desc.contains("/84'/0'/7'")`.
- **(b) the override slot feeds BOTH the expected-card synthesis AND the recompose.** Synthesis: `synthesize_unified(&resolved_for_synth, …)` (`verify_bundle.rs:574`), where `resolved_for_synth = vec![origin_slot]` under `--origin` else `resolved.clone()` (`:567-571`). Recompose: `build_descriptor_string(template, std::slice::from_ref(slot), 1, network, args.account, None)` (`:611-618`) and first-receive from `slot.xpub` (`:625`), where `slot = origin_slot.as_ref().unwrap_or(canonical_slot)` (`:572`). Both consume the override. Note `build_descriptor_string` still receives `args.account`, but it is IGNORED under override: `account` only feeds the `fallback` origin (`pipeline.rs:77`), and `key_origin_str` uses the `fallback` ONLY when `slot.origin_path_bare().is_empty()` (`pipeline.rs:41`) — under `--origin` the override slot's `path` is the explicit non-empty override path, so `slot.bracketed_origin()` is used and `account` is moot. No account double-application. Restore passes `args.account` identically (`restore.rs:177-184`), so the two agree.
- **(c) `--expect-wallet-id` skip-under-`--origin` correct + matches restore.** `verify_bundle.rs:638-644`: under `origin_slot.is_some()` it prints `notice: --expect-wallet-id is not checked when --origin overrides…` and skips the match; else the canonical recompute-and-match runs (`:645-665`). Identical carve-out to restore (`restore.rs:331-338`, same notice text). Rationale sound: D7 (`wallet_policy_id_for_singlesig`) hashes the fully-keyed, explicit-CANONICAL-origin preimage; a custom origin is a different preimage that would spuriously mismatch the canonical id. Pinned by `verify_template_bundle_expect_wallet_id_skipped_under_origin` (deliberately-wrong `deadbeefdeadbeef` under `--origin` → `.success()`, notice on stderr).
- **(d) the new tests genuinely exercise the path.** `template_cards("bip84", PHRASE_A, "7")` emits real cards at `--account 7` (mk1 origin = m/84'/0'/7'); verify under `--origin m/84'/0'/7'` re-derives that exact xpub → `md1_template_match` + `mk1_template_stub_bind` both genuinely match (non-vacuous: the test also asserts `v_stdout.contains("OK")` and the recomposed descriptor carries `/84'/0'/7'`). Recompose equals `restore --origin m/84'/0'/7'`.
- **(e) the previously-misleading comments are now accurate.** `verify_bundle.rs:308` (`--slot @0.<secret>= + --account/--origin`) and the docstring `:468` (`+ --account/--origin`) now correctly describe an implemented `--origin`. The `--expect-wallet-id` docstring (`:83`) gained the "NOT checked when `--origin` overrides" note.
- **(f) no funds-safety regression.** A verify that should FAIL still fails: `verify_template_bundle_without_seed_is_refused` (`.failure()`), `verify_template_bundle_expect_wallet_id_wrong_mismatch` (`.code(4)`), and the card-binding checks (`md1_template_match`/`mk1_template_stub_bind` → exit 4 on a card swap or policy/template cross-mix, confirmed round-1). Under `--origin`, the binding checks run against the override-synthesized expected bundle — an mk1 made at a different origin than `--origin` would mismatch and fail (correct). The skip is ONLY of `--expect-wallet-id` (a redundant D7 disambiguator), never of the structural md1/mk1/ms1 card comparison.
