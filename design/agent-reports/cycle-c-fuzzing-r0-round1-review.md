# R0 Review — Cycle C fuzzing (round 1)

Reviewer: Fable 5 architect agent (a92aa485e80ed1c8e), 2026-06-11.
Target: design/BRAINSTORM_stress_cycle_c_fuzzing.md @ toolkit e33c147.
Persisted verbatim per CLAUDE.md convention BEFORE the fold.

## Verdict: RED

The infrastructure premise is sound and now empirically proven end-to-end on this machine (nightly present, cargo-fuzz installs, a real md-codec fuzz target built, ran 6.5M execs/46s with the fixed-point oracle live on the success path, and found a planted bug in <1s with an artifact written). But two of the ten targets cannot be built or will crash vacuously on their first execution as currently specified, and the toolkit fuzz workspace as described would not compile. These are design decisions, not implementation details — they must be folded before any code.

## Critical (must fix before any further step)

- **[C1] The `ms1_no_secret_leak` oracle is DOA as specified — ms-codec's own error Display echoes the entire input share.** ms-codec `error.rs:11` wraps `Codex32(codex32::Error)` and `error.rs:118` Displays it as `write!(f, "codex32 parse error: {:?}", e)`. codex32-0.1.0 `src/lib.rs:58-63` defines `InvalidChecksum { checksum, string: String }` — **the full input string rides inside the error**. A single bit-flip of a valid share → checksum failure → `InvalidChecksum` → Display contains every ≥8-char window of the secret data-part. The oracle as spec'd crashes on essentially the first mutated exec, permanently, on a behavior the constellation already knows about (toolkit v0.53.4 exists precisely to withhold this echo at the *toolkit* layer). Additionally `Error::WrongHrp { got: String }` (`error.rs:15-17`, Display at `:122`) echoes attacker-controlled text, and a mutation that turns any data char into `1` shifts the bech32 separator so the parsed "HRP" contains a prefix of the real secret — another guaranteed window hit. The spec must decide, before implementation: (a) is the ms-codec-layer echo a bug? Then file the FOLLOWUP (bound/withhold `InvalidChecksum.string` and `WrongHrp.got`) and either fix first or build the oracle with a documented, variant-matched exclusion set (`matches!(e, Error::Codex32(_) | Error::WrongHrp{..})` → skip scan) that shrinks as fixes land; or (b) re-point the no-leak oracle at the layer that actually promises no-echo (toolkit `friendly_ms_codec`). Without this decision the target produces one pre-known finding and then nothing, forever.

- **[C2] The toolkit `descriptor_parse` target cannot be built as designed — `parse_descriptor` is bin-private and its compile closure is effectively the whole binary.** `parse_descriptor` is declared `mod parse_descriptor;` at `crates/mnemonic-toolkit/src/main.rs:23`; `lib.rs` deliberately does NOT export it (locked Option C crate shape, `SPEC_secret_memory_hygiene_v0_9_B.md` §4 P2 — `ToolkitError` is binary-private by design, and `parse_descriptor` returns `Result<_, ToolkitError>`, signature at `parse_descriptor.rs:747-751`; note it also takes `keys: &[ParsedKey]` and `fingerprints: &[ParsedFingerprint]`, not just `&str` — empty slices are valid). I mechanically computed the transitive `crate::` closure from `parse_descriptor`: **35 modules, ~56k lines**, including `cmd/`, `wallet_export`, `friendly`, `repair`, `cost` (dragged in by `error.rs:78/:269/:619-664`) and `derive`/`derive_slot` (`parse_descriptor.rs:953`, `synthesize.rs:400`). There is no small `#[path]`-mount hook. Viable options the spec must choose between: (i) `#[cfg(fuzzing)] pub mod …` bulk mount of the bin module graph in lib.rs (cargo-fuzz passes `--cfg fuzzing` via RUSTFLAGS; zero effect on normal builds; **but** `cfg(fuzzing)` trips `unexpected_cfgs` on both 1.85.0 and nightly — empirically verified — and toolkit CI runs `clippy --all-targets -- -D warnings` at rust.yml:149-150, so a `[lints.rust] unexpected_cfgs = { check-cfg = ['cfg(fuzzing)'] }` declaration is mandatory; this also amends the spec's "no library code changes" NO-BUMP claim and deserves its own mini-R0), or (ii) descope the toolkit target from Cycle C into a FOLLOWUP and ship 9 targets across 3 repos. I recommend (ii)-now-(i)-later or (i) as an explicit, gated phase 4 — but the spec as written ("the fuzz crate can path-dep the lib target and reach parse_descriptor") is factually wrong and unimplementable.

- **[C3] The toolkit fuzz workspace will not compile without replicating `[patch.crates-io]` — Cargo patches do not cross workspace boundaries.** Toolkit root `Cargo.toml` patches miniscript to git rev `95fdd1c`; the closure uses `Terminal::SortedMultiA` at `parse_descriptor.rs:595`, which exists only in the patched rev (the "two miniscripts" fact from the v0.49.1 cycle). A self-contained `fuzz/` workspace resolves miniscript from crates.io and either fails to compile or — worse — silently fuzzes different parser semantics than ships. The fuzz workspace's own root manifest must carry an identical `[patch.crates-io]` section. Corroborating evidence of the resolution-drift class: my scratch md-codec fuzz workspace resolved miniscript **13.1.0** while the repo's committed Cargo.lock pins **13.0.0**. Spec must mandate: commit `fuzz/Cargo.lock` per repo, aligned (`cargo update --precise`) with the root lock's shared-dep versions.

## Important (must fix before implementation)

- **[I1] The correction-honesty oracle as spec'd does not typecheck.** Both `md_codec::chunk::decode_with_correction` (chunk.rs:492, returns `Result<(Descriptor, Vec<CorrectionDetail>), Error>`) and `ms_codec::decode_with_correction` (decode.rs:221, returns `Result<(Tag, Payload, Vec<CorrectionDetail>)>`) return the *decoded value*, not corrected strings — "the corrected result must itself decode cleanly" is internally guaranteed and unverifiable from the return value. Re-specify as **apply-details idempotence**: `CorrectionDetail` carries `{chunk_index (md only), position, was, now}` (chunk.rs:405-415, decode.rs:120-128) — apply `now` at `position` in the input's data-part, re-run `decode_with_correction`, assert (a) equal decoded value, (b) empty details vector. That is implementable, meaningful (catches dishonest position/char reporting), and typechecks.
- **[I2] mk re-encode oracle must use `encode_with_chunk_set_id`, not `encode`.** `key_card.rs:95-100`: `encode` draws a **fresh 20-bit chunk_set_id from the system CSPRNG** on the multi-chunk path — nondeterminism inside a fuzz target breaks crash reproducibility and coverage stability. `KeyCard` does not carry the csi (wire-only), so the value-level fixed-point is otherwise sound; pin a fixed csi via `encode_with_chunk_set_id` (key_card.rs:110).
- **[I3] `md1_decode_payload` with `total_bits = len*8` reproduces P3's exact blind spot.** P3 pins the same (`proptest_roundtrip.rs:33`, "decode_payload arm pins total_bits = bytes*8" — verified). Partial-byte trailing-bits paths stay unfuzzed, contradicting the charter's point of exceeding P3. Make total_bits fuzz-chosen: e.g. first 2 input bytes = LE `total_bits` candidate (unclamped — also exercises the `total_bits > len*8` validation), remainder = payload.
- **[I4] Citation drift: ms `decode_with_correction` is at decode.rs:221, not :130** (:130 is the `CODEX32_ALPHABET` const region). All other spot-checked citations verified exact (md decode.rs:79/:15, chunk.rs:305/:492; mk key_card.rs:115; ms decode.rs:42, shares.rs:186; toolkit :747/:398/:60; P3 :32-42; pins 1.85.0×3 + mk none). Fix per the grep-at-write-time convention.
- **[I5] cargo-fuzz install has an MSRV footgun the CI section is silent on.** Empirically: `cargo install cargo-fuzz` run under the repo's 1.85.0 pin **fails** (`cargo-platform@0.3.3 requires rustc 1.91`, `cargo_metadata@0.23.1 requires 1.86`); `cargo +nightly install cargo-fuzz --locked` succeeds (v0.13.2). The CI workflow must install cargo-fuzz with the nightly toolchain active (or use a prebuilt-binary action, e.g. taiki-e/install-action) — and the same applies to anyone following the spec locally from a repo cwd.
- **[I6] Define re-encode-failure semantics + corpus-validity gate.** `encode_md1_string` (encode.rs:114) and `ms_codec::encode` are fallible. Spec should state explicitly: `Err` on a decode-accepted value is a **real finding** (assert/panic in-target; it is the decode/encode asymmetry class the charter targets — ms's encoder-symmetry rule at encode.rs:16-25 makes it genuinely unexpected), not noise to swallow. And `gen-corpus` must assert every committed seed decodes Ok — empirical motivation: of two md1 strings harvested from in-repo docs, one (`md1fz4awqq…`) does not decode; doc-harvested corpora silently rot.

## Minor (may defer)

- **[M1]** Use cargo-fuzz's default `fuzz/corpus/<target>/` layout (auto-loaded by `cargo fuzz run`, verified) instead of the spec's `corpora/<target>` — nonstandard paths need explicit args in every invocation.
- **[M2]** Splitter: prefer a sentinel-byte split (e.g. `\n`, which is outside the bech32 alphabet) over first-byte-count + even-split. libFuzzer insertions/deletions then move one chunk boundary locally; with count-prefix+even-split, any length change re-shears every chunk boundary, destroying corpus structure. Cap chunk count by truncation (≤8).
- **[M3]** No-leak oracle: scan Debug (`{:?}`) as well as Display (panic/log surfaces emit Debug); ≥8-char window over a 32-symbol alphabet is 40 bits — false-positive odds per error string are negligible; threshold is fine.
- **[M4]** ms rust.yml's header comment claims "ms-codec has its own separate workflow" — only rust.yml (ms-cli-scoped paths filter) exists. `fuzz-smoke.yml` will be the first CI ever exercising ms-codec in that repo; worth saying so in the spec.
- **[M5]** Nightly pinning (open Q7): agree — pinned `nightly-YYYY-MM-DD` in fuzz/rust-toolchain.toml + quarterly-bump FOLLOWUP. Precedent for nightly-in-CI exists (ms miri job uses floating `dtolnay/rust-toolchain@nightly`). Note mnemonic-key's fmt-job comment mandates keeping the repo free of a *root* rust-toolchain.toml — a `fuzz/`-scoped file does not violate it (rustup resolves from cwd upward), but cite the comment so nobody "fixes" it.
- **[M6]** descriptor-mnemonic CI clones the rust-miniscript fork (`ci.yml` test/clippy/doc jobs) but no manifest references it (no `[patch]`, no path dep; Cargo.lock pins crates.io 13.0.0) — appears vestigial. Unrelated cleanup candidate; do not touch this cycle.
- **[M7]** `from_utf8_lossy` is acceptable (md1/ms1/mk1 are ASCII; U+FFFD collapse just wastes a sliver of the input space). Alternative `str::from_utf8(...).ok()?`-style early-return keeps the corpus ASCII-clean. Either is fine; not a vacuity risk.
- **[M8]** Cron caveats: scheduled workflows run on the default branch only and are auto-disabled after 60 days of repo inactivity — add `workflow_dispatch` alongside `schedule` for manual runs and post-disable recovery.

## Answers to open questions

1. **One cycle vs split:** One cycle is feasible *only* with hard per-repo sub-gates: phase = repo, order **md → ms → mk → toolkit**, each phase lands fuzz dir + targets + bring-up proof + smoke CI green before the next opens. Each phase is standalone-valuable, so pre-authorize stopping after any phase. The toolkit phase is contingent on the C2 hook decision (own mini-R0 if option (i) chosen); descoping it to a FOLLOWUP is an acceptable RED-exit.
2. **CI cadence:** Spec's proposal is right-shaped. Concretely: `cargo fuzz build` compile gate on push/PR touching `fuzz/**` AND the codec src paths (catches API drift early — the lagging-indicator lesson); `cargo fuzz run -- -max_total_time=60` per target on cron (daily is fine: 10×60s + one nightly build ≈ 15-20 min, runs off the push-CI critical path) + `workflow_dispatch`. Do not run-fuzz on every push. New workflow files don't perturb existing path filters (toolkit rust.yml is `crates/**`-scoped; sibling-pin-check scans for `cargo install --git --tag` lines, which fuzz-smoke.yml won't contain).
3. **Workspace exclusion:** Verified empirically with a synthetic root-workspace + nested own-`[workspace]` fuzz dir containing deliberately misformatted code: root `cargo fmt --all --check`, `cargo test --workspace`, `cargo clippy --workspace --all-targets -- -D warnings` all ignore it completely. All four repos use explicit member lists (no globs), and `**/target/`-style gitignores already cover `fuzz/target`. No CI job in any repo globs all Cargo.toml. The claim holds — with the C3 patch-replication and committed-fuzz-lock caveats.
4. **Window length / Debug:** ≥8 chars is statistically safe (40 bits). Scan Display AND Debug. But the oracle's real problem is C1's exclusion-set decision, not the threshold.
5. **Re-encode reachability:** md: `encode_md1_string(&Descriptor)` ✓ (Cycle B P4 precedent). mk: `encode_with_chunk_set_id(&KeyCard, FIXED)` ✓ — must not use `encode` (I2; CSPRNG). ms: `encode(tag, &payload)` ✓ and the encoder's reserved-tag symmetry (encode.rs:16-25) means decode-accepted tags are always re-encodable — the oracle applies to all three codecs; treat re-encode `Err` as a finding (I6). `combine_shares` output `(Tag, Payload)` feeds the same `encode`.
6. **Toolkit hook:** Answered in C2 — bin-only, closure ≈ whole bin crate (35 modules/56k lines), signature needs `(input, &[], &[])`. Minimal *refactor-free* hook does not exist; the minimal *honest* hook is the cfg(fuzzing) bulk module mount in lib.rs with the check-cfg lint declaration, plus C3's patch replication; otherwise descope.
7. **Nightly drift:** Pinned nightly + quarterly-bump FOLLOWUP (M5). The locally-installed nightly (1.97.0-nightly 2026-04-27) built and ran everything; local validation is fully possible on this machine — no only-in-CI surface except the GitHub-Actions wiring itself.
