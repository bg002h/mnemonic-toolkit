## Important (must fix before implementer dispatch)

- **[I1-plan] The cited grouping-key / final-step extraction seam is UNUSABLE — `lex_placeholders` fail-closed-rejects the fixed step the guard matrix must read, and `extract_origin_components` never captures the use-site step at all** — `PLAN P1 step 2` + risk register #1. P1 says extract the grouping key + discriminant "via the existing key/use-site extractors (`extract_origin_components`, the `lex_placeholders` key structure)." Verified against source, **neither works for the final step**:
  - `extract_origin_components` (`pipeline.rs`) captures only `(fp_hex, origin-path-inside-brackets, bare-xpub)` — group 3 is the bare xpub with **no** `/0/*` suffix (it's fed straight to `Xpub::from_str`). The use-site final step is not in its output.
  - `lex_placeholders` (via `concrete_keys_to_placeholders` → `parse_descriptor`) is the **floor that rejects fixed steps** — the residue reject at `parse_descriptor.rs:206-211` fires *inside* `lex_placeholders` on exactly the `/0/*` / `/1/*` inputs the pre-pass must inspect. Calling it to read the step errors before returning it.

  So condition 4 ("final steps differ") and condition 3 (fixed unhardened step + wildcard-hardness) have **no working input source** as planned. This is a blocking mechanism defect in the funds-critical core.
  - **Fix (pin the mechanism):** extract per-candidate structure by **parsing the checksum-validated body via rust-miniscript** — `Descriptor::<DescriptorPublicKey>::from_str(body)` (which *accepts* fixed steps, unlike md1's `lex_placeholders`), then read each key's `derivation_path()` (final step) + `wildcard`, the template kind/threshold, and `Tr::tap_tree()` presence. This single mechanism ALSO resolves carry-forward 1 (tr key-path vs script-path = `tap_tree().is_none()` vs `.is_some()`), which already implicitly requires a miniscript parse ("from the parsed key/tap-tree structure") — the plan is internally inconsistent in citing `lex_placeholders` for the same step. The API is proven available: the test crate already uses `is_multipath()`/`into_single_descriptors()`/`derive_at_index()` (`tests/prop_backup_restore_roundtrip.rs:382`). Drop the `lex_placeholders` citation for extraction; keep `extract_origin_components` only for the `(fp, origin path, xpub)` key-material tuple.
  - **Reconcile the §4.3 construction constraint in the same fix:** SPEC §4.3 mandates the merged-string replacement be "anchored on the parsed per-key use-site positions (from the key/use-site lexer), **NOT a naive `str::replace`**." But `lex_placeholders` can't process the input (rejects it) and a miniscript AST parse **loses byte offsets**, so neither yields spans to anchor on. State the workable, funds-safe mechanism explicitly: after miniscript-verifying **condition-7 uniformity** (every key carries the identical `/<stepA>/*`), a `replace_all("/<stepA>/*" → "/<recv;chg>/*")` on the checksum-stripped body is correct and unambiguous (the trailing `/*` distinguishes a use-site step from any origin-path component, which ends in `']`), and is **oracle-backstopped** by §8.4/§8.10 (original-anchored address equality). This satisfies the SPEC's intent (no *first-only*/partial rewrite) while being implementable — note it as the sanctioned construction so an implementer doesn't stall on the literal "position-anchored from the lexer" wording.
