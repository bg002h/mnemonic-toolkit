### Critical

**C1 — The one-time mechanical migration CEMENTS drifted line-refs as wrong-but-existing symbols, inverting the cycle's stated purpose; and Phase 3 closes the drift FOLLOWUP while shipping that cemented drift.**

The rewrite resolves each ref by reading *current* source at the manual's *existing* line number, then pinning the enclosing symbol. For an accurate ref this is correct. For a **drifted** ref — exactly what this cycle exists to eliminate — the current line now sits in unrelated code, so the migration pins the WRONG symbol. G2 (`grep -wqF`) cannot detect this because the symbol exists. So §6's success criterion "every `::symbol` resolves" and a GREEN post-migration `make lint` measure existence, not correctness — a green lint is not evidence the migration is right.

### Important

**I1 — Multi-target comma-list refs are silently corrupted by the rewrite, AND survive G1.** `REF_RE` (`cycle-b-lineref-resolver.py:72`) matches only the first `:N[-M]`. Roughly 30+ refs in the **rendered** Part V chapters and glossary are comma-lists, e.g. `bch.rs:424, 440, 478, 493` (`52-mk-codec-api.md:285`), `tlv.rs:101, 125, 151, 175, 249, 380` (`51-md-codec-api.md:415`), `parse_descriptor.rs:673, 679` (`54-…:80`), `format.rs:120, 149` (`61-glossary.md:329`). The rewrite turns `bch.rs:424, 440, …` into `bch.rs::bch_correct_regular, 440, 478, 493` — orphaned bare line numbers that G1 (`\.rs:[0-9]`) will NOT catch (no `.rs:` precedes `440`). The SPEC never mentions comma-list refs; the resolver doesn't handle them. This is a second, independent migration defect in rendered output. The migration tool and G1 both need a comma-list rule (`\.rs:[0-9]+(\s*,\s*[0-9]+)*`).

**I2 — Gated scope (§4.3) should be `src/` ONLY; "both" is unsupported by the harness and provably wrong on the transcripts.** (a) The api-harvest transcripts are NOT in `MD_SRC`/`PDF_SRC` (Makefile:72-75 globs `src/` only) — they are unrendered authoring scaffolding, same class as the cli-help goldens deleted last week (MEMORY: `project_cli_help_golden_cleanup_shipped`). (b) Mechanical migration of them is demonstrably wrong: `api-harvest-mnemonic-toolkit.md:466` reads "the `format::BundleJson` doc-comment at `src/format.rs:114`," but the resolver maps `format.rs:114`→`MultisigInfo` (TSV 587), yielding "BundleJson doc-comment at `format.rs::MultisigInfo`." (c) The harness can't even gate transcripts as wired: Makefile:241-247 passes only `SRC_DIR` to `lint.sh`; gating transcripts requires plumbing `TRANSCRIPTS` the SPEC never specifies. All three vectors point to `src/`-only. (If the transcripts' staleness matters, file it as a separate FOLLOWUP — they're a known-stale harvest snapshot pinned to old crate versions, e.g. `Version | 0.8.0`, `md-codec-v0.16.1`.)

**I3 — Refs whose intent is a specific LINE's content, not a symbol, lose meaning under conversion (G2 still passes).** Several rendered/harvest refs cite a line *because of what is written there*, e.g. "the doc-comment at `format.rs:114` still says 'v0.2: schema_version 2'" (`61-glossary.md:329`, `api-harvest-…:466`). Converting to `format.rs::<symbol>` drops the line-level specificity the prose depends on, and (per I2's evidence) often resolves to the wrong symbol anyway. The SPEC's migration model assumes every `:N` is a "go look at this item" pointer; a non-trivial subset are "go look at this exact line." These need hand review, not the mechanical path — further evidence the "98.7% mechanical" figure is overstated.
