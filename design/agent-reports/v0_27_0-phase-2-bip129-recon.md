## Executive summary (5 bullets, copy of agent output)

1. **BIP-129 does NOT define a 6-line Round-2 bundle with a trailing signature.** Round-2 is a **4-line** descriptor record (`BSMS 1.0` / descriptor-or-template / path-restrictions / first-address) with **no SIG line**. Authentication of Round-2 records is achieved via the **encryption MAC** (HMAC-SHA256 keyed by SHA256(ENCRYPTION_KEY)), not a content signature; in `NO_ENCRYPTION` mode there is no MAC and no signature on Round-2 at all.
2. **Signatures exist only on Round-1 key records** (Signer → Coordinator) and are **5-line** records: `BSMS 1.0` / hex-TOKEN / KEY / Description / SIG. The SIG is a **BIP-322 legacy-format ECDSA recoverable signature** (Satoshi-style "Bitcoin Signed Message:\n" double-SHA256 digest), **base64-encoded** on line 5. The signing key is the private key corresponding to the public key / xpub on line 3 — there is **no `--coordinator-hmac-key`** in BIP-129; HMAC and PBKDF2 are encryption-mode primitives, never used for the SIG.
3. **The plan's framing ("TOKEN + descriptor + xpub → per-cosigner HMAC key → HMAC body → SIGNATURE") matches no part of BIP-129.** It conflates the Round-1 BIP-322 ECDSA signature (which uses the signer's own privkey, never the TOKEN) with the Round-2 encryption MAC (which is HMAC-SHA256 over plaintext keyed by SHA256(PBKDF2-SHA512(TOKEN))). Both surfaces exist; neither matches the proposed signature.
4. **The existing toolkit BSMS parser at `crates/mnemonic-toolkit/src/wallet_import/bsms.rs:104-125` recognizes a non-standard "6-line" shape** (header / TOKEN / descriptor / paths / first-addr / signature) that is **not BIP-129**. Treat this as either (a) a toolkit-local extension to be **specified and stamped explicitly** in the Phase 2 plan, or (b) a parser bug to be **corrected to the BIP-129 5-line Round-1 shape**. This decision must precede any verifier code.
5. **Phase 2 should pivot.** If the goal is BIP-129 signature verification, target the **Round-1 5-line key records** with BIP-322 verify (one signature per Signer record, each verified against that Signer's own pubkey/xpub). The Rust dep is `bitcoin::sign_message` (already in scope via `rust-bitcoin`) plus secp256k1 recovery — not HMAC. Test vectors are inline in BIP-129 (5 signers across 4 modes; all verifiable with bitcoin-core's `signmessage`/`verifymessage` legacy format).

## BIP-129 §Specification — verbatim quote of the signature mechanics

There is **no §5** in BIP-129. The BIP's section structure is `Introduction / Specification / QR Codes / Compatibility / Security / Privacy / Test Vectors / Acknowledgement / References`. The signature/MAC mechanics live in §Specification → §Setup Process → Round 1 → Signer (lines 77-84 of `https://raw.githubusercontent.com/bitcoin/bips/master/bip-0129.mediawiki`) and §Specification → §Encryption (lines 118-165).

**Round-1 5-line Signer record (BIP-129 line 81):**

> *"The first line in the record must be the specification version (`BSMS 1.0` as of this writing). The second line must be the hex-encoded `TOKEN`. The third line must be the `KEY`. The `KEY` is a public key or an XPUB plus the key origin information, written in the descriptor-defined format, i.e.: `[{master key fingerprint}/{derivation path}]{KEY}`. The fourth line is a text description of the key, 80 characters maximum. The fifth line must be a `SIG`, whereas `SIG` is the signature generated by using the private key associated with the public key or XPUB to sign the first four lines. The signature should follow [BIP-0322](https://github.com/bitcoin/bips/blob/master/bip-0322.mediawiki), legacy format accepted."*

**Round-2 4-line Coordinator descriptor record (BIP-129 line 96):**

> *"The first line in the descriptor record must be the specification version (`BSMS 1.0` as of this writing). The second line must be a descriptor or a descriptor template. The third line must be a comma-separated list of derivation path restrictions. The paths must start with `/` and use non-hardened derivation. If there are no template or restrictions, it must say `No path restrictions`. The fourth line must be the wallet's first address. If there are path restrictions, use the first address from the first path restriction."*

**Coordinator Round-2 verify of Signer Round-1 SIG (BIP-129 line 94):**

> *"The Coordinator verifies that the included `SIG` is valid given the `KEY`."*

**Encryption / MAC / KDF (BIP-129 lines 136-165):**

> *"The key derivation function is PBKDF2, with PRF = SHA512. Specifically: `DKey = PBKDF2(PRF, Password, Salt, c, dkLen)`. Whereas: PRF = SHA512, Password = "No SPOF", Salt = `TOKEN`, c = 2048, dkLen = 256 bits (32 bytes), DKey = Derived `ENCRYPTION_KEY`"*
>
> *"The encryption scheme is AES-256-CTR. `MAC = HMAC-SHA256(HMAC_Key, hex-encoded TOKEN || Data)` `IV = First 16 bytes of MAC` … Whereas: DKey = `ENCRYPTION_KEY`, HMAC_Key = SHA256(`ENCRYPTION_KEY`), Data = the plaintext, e.g. the entire key record in round 1 and the entire descriptor record in round 2"*

## Critical asymmetry the impl must respect

BIP-129 line 144 says PBKDF2 `Salt = TOKEN` — and the Coinkite Python reference (`bsms/encryption.py`) uses **raw bytes** of the TOKEN (unhex of line 2) as the salt. But line 152 says `MAC = HMAC-SHA256(HMAC_Key, hex-encoded TOKEN || Data)` — MAC uses the **ASCII-hex** of TOKEN as part of its input. Same TOKEN, two different byte representations in the same record. Foot-gun: any impl that conflates the two will fail TV-3's ENCRYPTION_KEY check.

## TOKEN modes (BIP-129 lines 123-125)

- `NO_ENCRYPTION` → exactly `"00"` (one byte 0x00, hex-encoded as two characters).
- `STANDARD` → 64-bit nonce, **16 hex chars** (e.g. test vector: `a54044308ceac9b7`).
- `EXTENDED` → 128-bit nonce, **32 hex chars** (e.g. test vector: `108a2360adb302774eb521daebbeda5e`).

## Signature format

- **Algorithm:** ECDSA over secp256k1, **recoverable form** (65 bytes = 1 header byte + 32 r + 32 s), per the legacy Bitcoin signmessage convention (BIP-322 legacy format, BIP-129 line 81).
- **Encoding:** **base64** (line 5 of the record). Each test vector in BIP-129 shows base64-encoded sigs (lines 230, 242, 269, 281, 311, 330, 379, 398, 417 of bip-0129.mediawiki) all of shape `[H\|I][A-Za-z0-9+/]{86}=` — classic Bitcoin signmessage form.
- **Header byte meaning:** the first byte encodes `27 + recovery_id [+ 4 if compressed]`. Test-vector sigs all begin with `H` or `I` which decode to first-byte = 31 or 32 → compressed pubkey, recovery_id ∈ {0,1}.
- **Message digest:** `dSHA256("\x18Bitcoin Signed Message:\n" || compact_size(len(body)) || body_bytes)` where `\x18` is the literal byte 0x18 = 24 = `len("Bitcoin Signed Message:\n")`. `rust-bitcoin` exposes `bitcoin::sign_message::signed_msg_hash(msg: &str) -> sha256d::Hash`.
- **Signed body construction:** `"BSMS 1.0\n{TOKEN_hex}\n{KEY}\n{description}"` (the 4 lines joined by `\n`, NO trailing newline). Both signer and verifier in `bsms/bip129.py` produce identical bytes.

## Proposed Rust engine signatures (replaces plan-doc §3.4)

The plan-doc's proposed `derive_per_cosigner_key(...) -> [u8; 32]` and `verify_signature(..., per_cosigner_key, body, declared_signature_hex)` are **wrong for BIP-129**. There is no per-cosigner key, no `[u8; 32]` HMAC key, no hex signature on the Round-1 SIG path.

```rust
/// Verify a BIP-129 Round-1 key-record signature.
///
/// Per BIP-129 §Round 1 Signer: the SIG on line 5 is a BIP-322 legacy-format
/// ECDSA recoverable signature over the four-line body (lines 1-4 joined by '\n',
/// no trailing newline), under the "Bitcoin Signed Message" double-SHA256 digest,
/// base64-encoded.
pub(crate) fn verify_round1_signature(
    signed_body_lines: [&str; 4],     // [version, hex-TOKEN, KEY, description]
    signer_pubkey: &secp256k1::PublicKey,
    signature_b64: &str,
) -> Result<(), BsmsVerifyError>;
```

**Pubkey extraction from line 3:** Line 3 is `[fp/path]KEY`. KEY can be either a raw 33-byte compressed pubkey (NO_ENCRYPTION/pubkey mode) or an xpub. For xpub mode, the signing pubkey is the xpub's **own** embedded pubkey (bytes 45-78 of serialized xpub), NOT a child-derived key. Per BIP-129 line 81 *"the private key associated with the public key or XPUB"*.

Optional companion (only if encryption-mode MAC verification enters scope):

```rust
/// Verify the BIP-129 §Encryption MAC envelope on a STANDARD/EXTENDED
/// Round-1 or Round-2 ciphertext blob.
pub(crate) fn verify_bsms_encryption_mac(
    token_raw_bytes: &[u8],    // unhex of line 2, NOT ASCII-hex bytes
    plaintext: &[u8],          // post-AES-CTR-decrypt full record
    declared_mac_hex: &str,    // 64-hex-char MAC prefix of .dat blob
) -> Result<(), BsmsVerifyError>;
```

Rust deps needed (all already in tree):
- `secp256k1` — ECDSA recover/verify
- `bitcoin::sign_message::signed_msg_hash` — exactly the BIP-129 digest
- `base64::engine::general_purpose::STANDARD.decode` — for line-5 SIG
- For MAC path (out of current scope): `hmac`, `sha2`, `pbkdf2`

## Ambiguities flagged

1. **Taproot `tr(...)`** is not addressed in BIP-129. Prerequisites listed in BIP-129 line 49 are BIP-32, BIP-322, BIP-380, BIP-381, BIP-382, BIP-383 — **BIP-386 (tr) absent**. Phase 2 should explicitly reject `tr(...)` Round-1 records and file a FOLLOWUP for BSMS-on-Taproot when the BIP is amended.
2. **Mixed pubkey/xpub Signer records** — verifier must handle both. Parse line 3 as `[{fp}/{path}]{KEY}`, then dispatch on whether KEY parses as 33-byte hex pubkey vs. base58 xpub.
3. **Description newline content** — BIP-129 doesn't explicitly forbid `\n` in the description. Recommendation: error on `\n` or `\r` in description bytes after CRLF→LF normalization; the SPEC text "80 characters maximum" implies a single line.
4. **Pubkey-encoding sub-ambiguity for xpub mode** — BIP-129 line 81 says "the private key associated with the public key or XPUB" but doesn't explicitly state **which** pubkey of an xpub is the signing pubkey. Coinkite ref (and consensus reading): the xpub's **own** pubkey (bytes 45-78 of the serialized xpub), NOT any child-derived key. Phase 2 must pin this explicitly in the toolkit SPEC.
5. **Toolkit's existing "6-line" parser shape vs. BIP-129** — `wallet_import/bsms.rs:104-125` accepts a "6-line bundle" `[header, TOKEN, descriptor, derivation_path, first_address, signature]` that is NOT in BIP-129. Three possible interpretations: (a) parser bug; (b) toolkit-local extension missing its SPEC; (c) deliberately-invented Coordinator-signed Round-2 variant. This decision precedes Phase 2 code.
6. **PBKDF2 password byte representation** — `Password = "No SPOF"` (7 bytes ASCII, no NUL). Pin explicitly: 7 raw bytes `b"No SPOF"` (0x4e 0x6f 0x20 0x53 0x50 0x4f 0x46).
7. **PBKDF2 salt vs MAC TOKEN byte representation asymmetry** — same TOKEN; PBKDF2 uses raw bytes (unhex of line 2); MAC uses ASCII-hex. Foot-gun; flag with inline comment in impl.
