## CRITICAL

### C1. (= D1, confirmed and re-demonstrated live) Chunk fragmenting: BIP mandates bit-boundary fragments and "accept any valid division"; the code splits at byte boundaries and floor-drops up to 7 bits per chunk

- **BIP:** line 246 "chunking splits the bit stream into fragments at encoder-chosen bit boundaries (no byte-aligned framing)"; line 773 "the exact division is implementation-defined … **Decoders MUST accept any valid division.** The boundary between header and fragment is exactly bit 37"; line 786 reassembly = "Concatenating each chunk's fragment bits".
- **Code:** `chunk.rs:262-291` (`split`): `bytes_per_chunk = payload_bytes.len().div_ceil(count)`; each chunk wire is exactly `37 + 8N` bits of whole bytes. `chunk.rs:326-336` (`reassemble`): `payload_byte_count = (symbol_aligned_bit_count - 37) / 8` — silently discards `(bits−37) mod 8` fragment bits per chunk. Only exact-multiple-of-8 fragments survive.
- **Live demo (this review):** encoded a 2-chunk card (`csi 0x8d5ed`), re-split the identical assembled payload at bit 259 (spec-legal), rebuilt valid-BCH chunks with correct headers → reference `md decode` rejects with `tag value 0x3b out of range`, while its own byte-division decodes fine. Symmetrically, a BIP-conforming decoder reassembling the reference's byte-padded fragments gets extra inter-chunk pad bits the BIP never mentions → stream corruption/CSI mismatch (the constellation-eval numbers: spec-rule CSI `0x453d1` vs wire `0x434df`). **Every real ≥2-chunk md1 card is unreadable by a spec-only implementation, and vice versa.**
- **Compounding internal contradiction:** line 201/306 say the 37-bit chunk header "occup[ies] eight bech32 characters with 3 bits of trailing slack absorbed by the symbol grid" and the fragment-max formula subtracts that slack (`(93−3−8−13)×5−3 = 342`), while line 773 says the fragment starts at exactly bit 37 (what the code does — contiguous, no slack). Three mutually incompatible framings (bit-exact / byte-aligned / 8-symbol+slack) coexist in the text.
- **Fix:** align BIP→code (fragments are whole bytes of the byte-padded assembled payload; assembled stream = `payload_bytes`, decode with ≤7-bit rollback tolerance; fragment boundary at bit 37, contiguous), delete "accept any valid division" and the 3-bit-slack framing, correct the fragment-max arithmetic, and commit a ≥2-chunk test vector (see I6). Code→BIP (true bit-boundary support) would need a wire-version bump and buys nothing.

### C2. Elided-origin ("`m` means canonical default") convention is entirely unspecified — a BIP-only decoder derives at the master node and recovers the wrong wallet

- **BIP:** §"Default derivation paths" (lines 150-168) says when no Tier-0/Tier-1 path exists, "encoders pick a per-top-level-type default path … **The path is encoded explicitly** into the Origin path declaration". Line 370: "A path of depth 0 (**the master node m**) encodes as just the 4-bit zero-depth field." No elision semantics exist anywhere (grep: only TLV 0x03's aside "overrides the canonical origin path … when the wrapper origin is non-canonical or undefined" — a concept never defined).
- **Code:** the reference encoder does the opposite — it encodes **depth-0 `m`** and resolves it at consume time through the canonical-origin table `canonical_origin.rs:10-19` (pkh→44', wpkh→84', tr-keyonly→86', wsh(multi)→48'/2', sh(wsh(multi))→48'/1'; everything else `None` = forced-explicit). Verified live: `md encode 'wsh(multi(2,@0/<0;1>/*,@1/<0;1>/*))'` (no `--path`) emits exactly the committed corpus phrase `md1yppqqxppsg2vlumagltz27le`, whose `wsh_multi_2of2.descriptor.json` shows `path_decl: "m"`. Most of the committed corpus (and hence the BIP's own §Test Vectors table) carries `m` on the wire.
- **Concrete failure:** an independent implementer follows the BIP, reads origin `m`, and derives xpubs at the master node → wrong xpubs, wrong addresses, recovery fails or lands on the wrong wallet — on the flagship vectors and on every default `md encode` output.
- **Also unspecified in the same cluster:** the decoder-side rule `validate_explicit_origin_required` (`validate.rs:221-246`) **rejects** elided origins whenever `canonical_origin` is `None` (`MissingExplicitOrigin`) — a wire-acceptance rule absent from the BIP, which presents depth-0 as universally legal. And the two tables disagree: the BIP's default table includes `sh(wpkh)`→49' and excludes pkh; the impl table includes pkh→44' and **omits `sh(wpkh)`** — with the demonstrated consequence that the reference encoder emits an elided `sh(wpkh)` card (`md1yqpqqxpsq258xsks3kh0ye`) that **the reference's own decoder rejects** ("non-canonical wrapper requires explicit origin for @0"). That is a reference bug the BIP review surfaced; file it in the repo regardless of which text wins.
- **Fix:** specify the elision convention normatively (depth-0 origin + canonical-origin table ⇒ resolved default; depth-0 with no table entry ⇒ decoder MUST reject `MissingExplicitOrigin`), reconcile the two tables, delete "encoded explicitly", and fix the `sh(wpkh)` encoder hole.

### C3. The long code exists only in the BIP; the implementation is regular-code-only — and the two documents disagree about where the 93 cap sits

- **BIP:** lines 262-269 normatively define the long code (BCH(108,93,8), `T_LONG`, 15-char checksum, `ms32_long_polymod`, lengths 96–108, "94 or 95 … invalid", "Encoders … exceeding 93 characters MUST use the long code"); line 848 decoders MUST implement "both regular and long codes"; length-envelope maxima 385/450 bits (lines 302-310).
- **Code:** `bch.rs:1-4` — "regular code only … long code dropped along with v0.x". Single-string cap = **80 data symbols** (`codex32.rs:25,89-94`), codeword cap 93 symbols **excluding** the 3-char `md1` prefix (`codex32.rs:32,174-179`) → max total string **96 chars, max payload 400 bits**, larger payloads must chunk. `md encode --force-long-code` is a **silent no-op** (verified live). BIP 93 (authoritative text) confirms the 93-char cap is on the **data part**, so the impl follows BIP 93 while the BIP's §Length envelope ("total string length capped at 93", `(93−3−13)×5 = 385`) wrongly counts the HRP into it, and §Checksum ("total data lengths up to … 93") accidentally matches the impl — the BIP contradicts itself.
- **Concrete failure:** both directions. A spec-only decoder rejects reference cards whose payloads are 386–400 bits (94–96-char strings the reference happily emits — "94 or 95 … invalid" per BIP). A spec-only encoder emits a long-code string for any 386–450-bit payload; the reference rejects it (`StringSymbolCountOutOfRange`, or BCH failure). Recovery-independence broken across a real size band.
- **Fix:** align BIP→code: delete the long code entirely (constants, guarantees, envelope rows, `ms32_long_polymod` references, "94 or 95" rule), restate the cap as ≤80 data symbols / ≤93 codeword symbols (HRP excluded) / payload ≤400 bits, chunking above that. Remove or implement `--force-long-code`.

### C4. Top-level scope: the BIP MUST-rejects `pkh` and `sh(multi/sortedmulti)`; the reference encodes and decodes both — and the BIP's own test corpus ships a `pkh` vector

- **BIP:** lines 100-111 "Conformant MD encoders MUST accept **exactly** the following …" (pkh, sh(multi), sh(sortedmulti), sh(KEY) rejected); "Decoders MUST reject these forms with a structured error"; §Sh wrapper matrix lines 129-148: inner tag other than wpkh/wsh → "Decoder MUST reject", with a MUST peek-before-recurse. Tag table line 430: pkh "Top-level: REJECTED".
- **Code + live demos:** `decode.rs:36-44` root allow-list is `{Sh, Wsh, Wpkh, **Pkh**, Tr}` with **no** sh-inner peek at the wire layer ("per-shape validation happens at the policy layer" — but no such rejection exists: `md encode 'sh(sortedmulti(2,…))' --path "m/45'"` → `md1yprddsqrqcuy9h42jehsvp9znl` → `md decode` round-trips it; `md encode/decode 'pkh(@0/<0;1>/*)'` round-trips the committed `pkh_basic` vector `md1yqpqqxzq2qwfv8urt848e`).
- **Internal contradictions in the BIP itself:** the error-taxonomy row (line 875) quotes the allow-list *including Pkh*; the §Test Vectors table (line 1118) lists `pkh_basic` — both contradict §Top-level scope. A conforming implementer literally cannot satisfy the corpus round-trip requirement (line 1107) and the scope MUSTs simultaneously.
- **Concrete failure:** users can engrave pkh / sh(sortedmulti) cards today (encoder accepts); a spec-conforming recovery tool MUST refuse them. And the FAQ's headline claim ("MD eliminates this footgun by rejecting sh(multi) at the spec level", line 1029) is untrue of the shipped code.
- **Fix:** pick one. Either enforce the scope in md-codec (encoder + decoder reject; drop `pkh_basic`; wire-behavior change but no version bump since conforming wires unchanged) — the safer direction given the FAQ's own footgun rationale — or re-scope the BIP to match code and delete the sh-matrix MUSTs. Lockstep with the mk/toolkit surface.

## IMPORTANT

**I1. Auto-dispatch is self-contradictory in the BIP and only half-implemented.** Line 194 (§Header): "discriminated **in-band**: decoders auto-dispatch on bit 0 of the first 5-bit symbol … **with no caller hint**" (+ Decoder-requirements #3 MUST). Line 330 (§General structure): "dispatch is **by reader role rather than by an in-band discriminator bit**". The code: `decode.rs:86-89` (`decode_md1_string`, backing `md decode`) does NOT auto-dispatch — verified live, it fails on the BIP's own chunked corpus vector with `wire-format version mismatch: got 9, expected 4` (a misleading mis-parse row absent from the §2.5 trace table); only `chunk.rs:503-639` `decode_with_correction` (backing `md repair`, v0.35.0) auto-dispatches, and only for `len==1`. Fix: make one story normative (in-band auto-dispatch is the better one), add the "got 9" row to the trace, and route `md decode` through the dispatching path.

**I2. PolicyId / WalletInstanceId as specified do not exist; the engraved 12-word anchor is computed over a completely different preimage.** BIP lines 792-815: `policy_id = SHA-256(canonical_bytecode)[0..16]`, engraved as the 12-word phrase; `PolicyId::fingerprint() -> [u8;4]` "(v0.10+)"; lines 817-838: `wallet_instance_id = SHA-256(canonical_bytecode || 78-byte bip32_serialize(xpubs))`. The codec has **no** `PolicyId` or `WalletInstanceId` types (grep-verified). What exists (`identity.rs`): `Md1EncodingId` (= the BIP's PolicyId formula, but line 217 asserts Md1EncodingId ≠ PolicyId — self-contradiction), `WalletDescriptorTemplateId` (not in the BIP), and `WalletPolicyId` (`identity.rs:186-269`) — the type carrying `to_phrase()` — hashed over an elaborate canonical-record preimage (template-tree bytes ‖ per-@N presence byte + varint-framed origin/use-site bits + fp? + xpub?) that no BIP reader could reconstruct. Note also the wire carries only 65-byte (chaincode‖pubkey) entries (TLV 0x02), so the BIP's 78-byte `bip32_serialize` concat is not even computable from card data (parent fingerprint/depth unavailable). Concrete failure: an independent tool's 12-word phrase never matches the reference's engraving — the sole purpose of the anchor (cross-verification) silently fails. Fix: rewrite §Policy identifier / §Wallet Instance ID to specify the actual `WalletPolicyId` preimage byte-exactly (it is documented in `identity.rs:141-186`) or re-point the phrase at `Md1EncodingId`; reconcile line 217 vs 798; fix the fingerprint API cite.

**I3. Decoder-side canonical-form validators are unspecified — the reference rejects wires the BIP permits.** `decode_payload` (decode.rs:56-76) enforces, beyond the BIP: `PlaceholderNotReferenced` + `PlaceholderFirstOccurrenceOutOfOrder` (validate.rs:17-33); `MultipathAltCountMismatch` (all multipath groups must share alt-count, validate.rs:127-148); `BaselineUseSiteOverride` (an `@0` use-site override is forbidden) + `RedundantUseSiteOverride` (override equal to baseline forbidden) (validate.rs:164-177); tap-leaf forbidden-tag set `{Wpkh,Tr,Wsh,Sh,Pkh,Multi,SortedMulti}` (validate.rs:203-208 — the BIP frames leaf subsets as MAY, line 689, and never gives this normative list); `MissingExplicitOrigin` (see C2); `InvalidXpubBytes` (secp point validation of TLV-0x02 pubkeys, validate.rs:255-264). A good-faith BIP encoder emitting, e.g., a redundant-but-well-formed override or a `multi()` tap leaf produces cards the reference refuses. Fix: enumerate every decode-rejection rule in the BIP (they are all reasonable rules — they just aren't written down).

**I4. Non-zero-padding rejection is specified with a fictional error and not implemented.** Line 250: decoders MUST reject non-zero trailing padding, "reference implementation surfaces this as `Error::InvalidBytecode { kind: BytecodeErrorKind::MalformedPayloadPadding }`" — no such variant/type exists anywhere (grep-verified). The TLV rollback (`tlv.rs:286-302`) tolerates ≤7 remaining bits **without inspecting their value** — non-zero pad is silently accepted. Acceptance-set divergence plus an un-pinnable conformance cite. Fix: either implement the zero-check or soften the BIP to match the rollback contract; delete the fictional variant.

**I5. Guided recovery, erasure decoding, and confidence reporting are REQUIRED by the BIP and entirely absent.** Lines 277-278 (correct up to 8 erasures / 13-15 consecutive erasures), line 849 (MUST implement erasure-aware decoding), lines 892-917 (MUST report outcome/method/confidence with a four-tier confidence ladder), lines 919-934 (guided recovery "REQUIRED in any MD-conformant tool", radius-12 constrained search). The workspace contains zero erasure code (grep), no structure-elicitation/candidate-search, and `md repair` reports corrections but no confidence tiers. The reference implementation is non-conformant with its own BIP's MUSTs. Fix: downgrade guided recovery + erasures to SHOULD/informative (or a separate companion spec) until implemented; a bitcoin-dev reviewer will test exactly this.

**I6. Test-vector section misdescribes the corpus machinery, and the one gap that hid D1 is still open.** No ≥2-chunk committed vector exists (`wsh_multi_chunked` is a chunked-of-1; `tests/chunking.rs` multi-chunk tests are impl-vs-impl only). §Regeneration's command `cargo test -p md-codec --test vectors -- --ignored vectors_regenerate` does not exist (verified: "no test target named `vectors`"); the loader is not `tests/vectors/manifest.rs` but `src/test_vectors.rs::MANIFEST` + the `md vectors --out` subcommand; "a tuple of three plain-text files" table lists four; `wsh_multi_chunked.phrase.txt` carries an undocumented extra `chunk-set-id:` line. Fix: commit a ≥2-chunk vector (post-C1-resolution bytes), plus a 94-96-char-boundary vector (C3) and a NUMS-taproot vector (none exists — the `is_nums=1` wire path is vectorless); correct the machinery description.

**I7. Unknown-TLV handling: BIP says "skip", impl preserves-and-re-emits — and the difference is CSI-visible.** Lines 560/575 ("skipped via length-prefix advancement") vs `tlv.rs:36-38,197-199,268-280` (unknown entries buffered verbatim and re-encoded). Reassembly's CSI check re-encodes the decoded descriptor (`chunk.rs:380-387`); a skip-style decoder drops the unknown TLV, re-encodes short, and CSI-mismatches any future chunked card carrying a reserved-tag TLV. Fix: specify preserve-verbatim (including re-emission ordering) as normative.

**I8. Varint canonicality unspecified.** `read_varint` (varint.rs:45-56) accepts non-minimal `L` and `l_high == 0` (the BIP says `L_high` selects 1..15; encoder never emits 0). A BIP-reading encoder has no minimality mandate; a non-minimal chunked card decodes but re-encodes minimally → wire CSI ≠ derived CSI → rejected. Fix: mandate minimal-length encodings for encoders and state the decoder's leniency (or reject non-minimal on decode).

**I9. Chunk-count sizing / auto-chunk mismatch (SHOULD-level).** Line 206 "Encoders SHOULD use the smallest count that fits"; impl sizes chunks by a 320-bit budget (`chunk.rs:220,249-260`) where 360 payload bits/chunk fit → emits more chunks than minimal. §Chunking's "the encoding MUST be chunked" reads as auto-chunk; the CLI instead errors and demands `--force-chunked` (verified). Document both.

- **M2.** Line 798: "12-word BIP 39 … (128 bits, including a 4-bit checksum word)" — 12 words = 132 bits: 128 entropy + 4 checksum.
- **M3.** Stale/fabricated references: `md-signer-compat` described as a workspace member with named subsets + CLI (line 704) — not in the repo (only md-cli, md-codec); FAQ §generator-string (lines 1066-1068) describes `gen_vectors --output` / `v0.2.json` machinery that exists nowhere; "Bindings for Python and TypeScript … planned for v0.2" (line 1163); `apoelstra/rust-miniscript#1` link (line 1158) almost certainly wrong; CLI list (line 1161) omits `repair`/`address`/`compile`/`gen-man`.
- **M4.** Rationale §"Why descriptor-codec" (lines 976-984) is stale v0.x text: claims MD "reuses the underlying tag values", adds "a path dictionary" (contradicting line 372's "does not carry a path dictionary"), and "@i … encodes as 2 bytes" (v0.30: kiw ≤5 bits).
- **M5.** Error-taxonomy nits: `PathDepthExceeded` "Decoders MUST reject depth > 15" is wire-impossible (4-bit field); same for decode-side `KeyCountOutOfRange`/`AltCountOutOfRange`/`ChunkCountOutOfRange` (all `x−1`-coded fields). State them as encoder-side.
- **M6.** Preamble is not BIP-2-conformant: `BIP: ?` with self-suggested numbers, non-standard `Status: Pre-Draft, AI + reference implementation, awaiting human review`, empty Comments-URI.

## Cross-format note (mk1/ms1 consistency)

md1's checksum layer = BIP-93 `ms32_polymod` verbatim (init `0x23181b3`, same GEN) with a per-HRP NUMS target (`0x0815c07747a3392e7`) — matching the constellation architecture (md/mk share the polymod; per-HRP targets; ms uses codex32's own constant). Whatever the mk1 BIP says here must mirror this exactly, including the "93 = data-part symbols, HRP excluded" cap semantics (C3) and must not inherit `bch.rs`'s wrong "codex32 init is 1" comment (M1). The §Authority-precedence pointer into the MK BIP (line 384) is the right pattern — keep both sides' sections cross-cited when C2's elision semantics land, since MK's `origin_path` authority interacts with md's canonical-default resolution.

## Submission-readiness verdict

**Not submission-ready.** The constants, bit layouts, worked examples, and single-string template-only happy path are excellent — genuinely byte-verified against the shipped code, which is better than most draft BIPs. But an independent implementer following only this document today would: fail to read any multi-chunk card (C1), derive the wrong wallet from most committed vectors (C2), reject legal reference cards in the 94–96-char band and emit long-code strings nothing can read (C3), refuse the corpus's own `pkh` vector (C4), and engrave a 12-word anchor that matches nothing (I2). The document also contradicts itself at four load-bearing points (bit-vs-byte-vs-slack framing; auto-dispatch vs reader-role; total-string vs data-part 93; scope-vs-corpus-vs-allow-list) and mandates a recovery subsystem the reference doesn't have (I5).

**Top-3 must-fix before any external review:**
1. **C1 — rewrite §Chunking to the byte-aligned wire that actually ships** (fragment = whole bytes, boundary at bit 37, delete "any valid division" and the 3-bit-slack framing) **and commit a ≥2-chunk vector.**
2. **C2 — specify the elided-origin/canonical-default convention** (and reconcile the default-path table with `canonical_origin.rs`, fixing the self-rejecting `sh(wpkh)` encoder hole in code).
3. **C3 — excise the long code and fix the length envelope** to 80-data-symbol/400-bit regular-only semantics (with C4's scope reconciliation as the immediate #4).

Repo-side follow-ups surfaced by the review, independent of BIP edits: the `sh(wpkh)` encode-without-path self-rejection bug; `--force-long-code` silent no-op; `bch.rs` init comment; `md decode` non-dispatch on chunked strings (misleading `got 9` error).
