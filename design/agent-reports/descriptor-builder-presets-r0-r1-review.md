## Critical

**C1 — The §2/§3.3 provenance mechanism cannot produce the §7 expected annotations (spec is internally contradictory).**
- Evidence: `ArchetypeDef.provenance: &[(&str, &str)]` is a *node_path-prefix → flag* map, and §3.3 says gate diagnostics get `flag` "by longest-prefix match". But §7's gate-flow-through cells demand **two different flags for diagnostics that land on the SAME node_path**:
  - `--threshold 5` with 2 keys → `SchemaField` is pushed at the **multi node's own path** (`gate.rs:148-149` calls `check_threshold(m.k, …, path, …)`, `gate.rs:212-221` pushes at `path`) — e.g. `root.or_d[0]`. §7 expects `flag: "--threshold"`.
  - duplicate `--key` values → `RepeatedKeys` localizes via `localize()` post-order (`gate.rs:338-364`) to the deepest subtree exhibiting `has_repeated_keys` — the **same multi node**, `root.or_d[0]` (its key children are strings, not nodes). §7 expects `flag: "--key"`.
  A prefix→flag map can hold only one flag per prefix; whichever is chosen, one §7 cell is unimplementable as specced.
- Why it blocks: this is the core new `--json` wire surface of the release; an implementer hits the contradiction in Phase 2 TDD and has to invent the resolution ad hoc — exactly what an R0 gate exists to prevent.
- Fix: make provenance **kind-aware**, e.g. entries `(prefix, Option<DiagnosticKind>, flag)` with longest-prefix + kind-specific override: for each quorum node, `(multi-path, Some(SchemaField)) → --threshold` / `(multi-path, None) → --key`, plus `(…multi.keys, None) → --key` for the `keys[i]` `SecretKey` paths. (Producer `min_count ≥ 2` guarantees the only `SchemaField` reaching the gate at a quorum node is threshold-flavored, so the mapping is unambiguous.) Note the cross-branch duplicate (`--key X --recovery-key X`) localizes to `root` → `flag: None` is acceptable; say so.

## Important

**I1 — Preset-mode dispatch order vs. the spec-intake path is unspecified; the natural wrong ordering survives the spec's own test matrix in the TTY case.**
- Evidence: `cmd/build_descriptor.rs:75` calls `read_spec` unconditionally before the gate; `read_spec` (`:91-114`) errors on TTY-no-`--spec` and otherwise **reads stdin to EOF**. §1 only says "if neither is given and stdin is a TTY, the existing no-input error path applies unchanged" — it never states that `--archetype` dispatch must branch BEFORE `read_spec` and that preset mode never reads stdin. An implementation that lowers after `read_spec` fails interactively (`--archetype …` at a terminal → spurious "no spec" error, or blocks on an open pipe); CI layer-2 cells (empty non-TTY stdin) would catch the pipe case but not pin the TTY contract.
- Fix: one sentence in §1 ("`--archetype` dispatch precedes spec intake; preset mode MUST NOT call `read_spec`/touch stdin") + a negative/positive cell in §7 (preset run with no stdin attached succeeds; preset run ignores piped stdin content).

**I2 — Phasing dependency error: Phase 1's producer-negative cells require `DiagnosticKind::Param`, which §9 places in Phase 2.**
- Evidence: §9 Phase 1 includes "producer negatives" (and the decay-ordering cell, §7: "producer `Param` diagnostic naming both flags"), but the `Param` variant + `"param"` `as_str` arm are listed under Phase 2 ("§3.3 provenance (`flag` field, `Param` kind, annotation…)"). `validate_params` returns `Vec<Diagnostic>` (`gate.rs:44-49` — `kind` is non-optional), so Phase 1 has no kind to construct.
- Fix: move `DiagnosticKind::Param` (+ `as_str`) into Phase 1; Phase 2 keeps only the `flag: Option<String>` field, the provenance annotation pass, and the spec-mode byte-stability self-test (c).

## Minor

**M1 — "Infallible `lower` makes 'no second validation path' structural" overclaims.** The infallibility is convention-coupled, not type-enforced: `lower` will `unwrap()` Options/index Vecs on the promise that `def.params` declared them required/min-counted. A mis-declared table for a future archetype #6 = production panic, against the codebase's own "never panic in a funds tool" ethos (`gate.rs:124-125`). Keep the `Result`-free signature (it is the right call — layer-1 tests make panic unreachable for shipped archetypes), but the SPEC should mandate `expect("declared required in ARCHETYPE_REGISTRY for <id>")`-style messages and note the coupling explicitly.

**M2 — The "non-repeatable param repeated" negative has two different failure surfaces depending on clap type.** `Option<String>`/`Option<u32>` flags (`--final-key`, `--older`, …) are rejected by clap itself when repeated (usage error, no producer diagnostic); only the `Vec<String>` flags (`--key`, `--recovery-key`) reach `validate_params`' `repeatable: false` check. §7 should pin the producer-path cell to a Vec-typed flag (e.g. `--key`×2 on `simple-timelocked-inheritance`) and note the clap-handled case separately.

**M3 — `--emit-spec` + `--network` is accepted-but-inert** (network only feeds the human-view address, `build_descriptor.rs:194-203`; the emitted SpecDoc has no network field). Consistent with `--format descriptor` ignoring `--network`, but since `--emit-spec` conflicts `--format`/`--json` and not `--network`, say in §4 (and the manual) that `--network` is accepted and ignored under `--emit-spec`.

**M4 — The §8 GUI FOLLOWUP content should also enumerate the deliberately un-projected clap rules.** Verified: `build_subcommand_conditional_rules` (`cmd/gui_schema.rs:336-347`) has no build-descriptor arm and projection is hand-coded per subcommand, so the SPEC's "no conditional-rule projection" claim holds. But the repo's precedent is that clap `conflicts_with` mutexes DO get hand-projected later for drift-gating (compare-cost, `gui_schema.rs:374-409`), and the GUI's `SubcommandSchema.conditional` is currently `None` for build-descriptor (`mnemonic-gui/src/schema/mnemonic.rs:3524`) — GUI forms will be able to emit `--spec`+`--archetype` / bare `--key` argv that clap refuses. Acceptable (CLI is the gate), but the FOLLOWUP should name the `--archetype↔--spec` mutex, the 10 `requires="archetype"` edges, the `--emit-spec` conflicts, and the `node_path: "params"` sentinel so the GUI cycle decides knowingly.

**M5 — Drift self-test (c)'s "byte-identical" needs an operational definition.** There is no pre-existing byte-golden of spec-mode `--json` diagnostics (existing tests assert fields only — `cli_build_descriptor.rs:126-138`); the test must pin a literal golden now and assert the `"flag"` key is absent. Empirical note: `emit_diagnostics` routes through `serde_json::Value` (`json!` at `build_descriptor.rs:123`), so object keys serialize **alphabetically** (`kind, message, node_path` — probe below); a present `flag` key will sort first. Worth one sentence so the implementer doesn't fight serde field order.

**M6 — Presentation: §1's `--archetype` value list is in non-alphabetical order while §2.1 pins alphabetical declaration order** (`decaying-multisig` first). Align §1's enumeration to remove ambiguity about `CliArchetype` declaration order.
