### 2. `mem::take` correctness at the Share boundary

`mod.rs:170-188`: `for (m_idx, mut mval) in member_shares { let inner: Vec<u8> = std::mem::take(&mut *mval); Share::from_parts(inner, ...) }`. The chain:
1. `mval: Zeroizing<Vec<u8>>` is owned by the loop binding.
2. `&mut *mval` derefs the wrapper to `&mut Vec<u8>`.
3. `std::mem::take(&mut Vec<u8>)` returns the original Vec (with its heap-allocated buffer of secret bytes) and replaces the wrapper's interior with `Vec::default()` (an empty Vec, no allocation).
4. `inner` (the secret-bearing Vec) is moved into `Share::from_parts`'s `value` parameter, which writes it to `Share.value` — a `#[zeroize]`-marked `Vec<u8>` field within the `#[derive(Zeroize, ZeroizeOnDrop)]` Share struct (`src/slip39/share.rs:77-120`).
5. `mval` (the now-empty Zeroizing wrapper) drops at end-of-iteration: Zeroize on an empty Vec is a no-op (nothing to scrub), then the empty Vec is freed.
6. The secret bytes now live in `share.value` and will be scrubbed when the Share itself drops.

The hand-off is sound. The inline comment at `:171-175` accurately describes this chain.

### 3. mlock pin lifetimes (Rust reverse-drop-order)

Both pin callsites declare `_ems_pin` AFTER their respective `ems` local, so the pin Drops FIRST at scope exit (Rust drops locals in reverse declaration order), while the `ems` Vec is still allocated — exactly the pin contract.

- `slip39_split` (`mod.rs:149` + `:159`): `ems` ← feistel::encrypt; `_ems_pin` ← pin_pages_for(&ems[..]). `ems` is only ever passed to `split_secret(..., &ems, rng)` (deref to `&[u8]`); no `.push` / `.extend` / `.reserve` on `ems`. Heap address stable; pin valid for the function body.
- `slip39_combine` (`mod.rs:309` + `:314`): `ems` ← `recover_secret(gt, &group_shares)?` (a `Zeroizing<Vec<u8>>`); `_ems_pin` ← `pin_pages_for(&ems[..])`. `ems` only passed to `feistel::decrypt(&ems, ...)` via Deref coercion (`&Zeroizing<Vec<u8>>` → `&Vec<u8>` → `&[u8]`). No realloc. Pin lifetime covers the PBKDF2-heavy decrypt window. On `Ok(master)` return: `master` moves into the Result first, then `_ems_pin` drops (munlock), then `ems` drops (frees Vec) — correct order.

Note on the `:158` inline comment ("pin drops at function exit, after `ems` itself is no longer referenced"): semantically correct but slightly informal — Rust drops in REVERSE declaration order regardless of last-use, so `_ems_pin` drops BEFORE `ems` at scope exit. Not flagged: the code is correct; the comment captures the right invariant in different language. (Nit-deferrable; FOLLOWUP not warranted.)

## Optional architecture notes (FOLLOWUP candidates only — explicitly NOT findings)

These are surfaced per the dispatch's "architectural questions worth surfacing" prompt; none are blockers, none merit FOLLOWUPs in their current form.

- **Share API ergonomics**: `Share::from_parts(value: Vec<u8>, ...)` requires the `mem::take(&mut *mval)` dance to hand off bytes from a `Zeroizing<Vec<u8>>` carrier without leaving a duplicate. Changing the signature to `Share::from_parts(value: Zeroizing<Vec<u8>>, ...)` would eliminate the dance, but Share already has `ZeroizeOnDrop` on its `value` field, so the current pattern is sound. No threat-model gap. Defer.
- **Third mlock pin on recovered master**: `slip39_combine` returns `master: Zeroizing<Vec<u8>>` without pinning it. Per the bip85/derive_slot precedent, the caller owns pinned-master responsibility. No concrete threat-model gap. Defer.
- **Lint anchor brittleness**: `Zeroizing::new(vec![0u8; random_len])` is a literal-identifier anchor — a rename of `random_len` would trip the lint. Acceptable for a strict invariant lint (the rename forces a deliberate lint-row update). No action.
