## D7
**Claim:** `WalletDescriptorTemplateId` is origin-invariant; emit elides ALL origins to `Shared(empty)`, dissolving the "Shared vs Divergent origin gate"; the keyless template round-trips (encode→decode→template-id stable).

**Origin-invariance of the template-id: SOUND, source-verified.** `compute_wallet_descriptor_template_id` (`identity.rs:71-104`) hashes ONLY `d.use_site_path.write` + `tree::write_node` + the `UseSitePathOverrides` TLV entry — it never reads `d.path_decl` (origins) and never calls `expand_per_at_n`. Pinned by `wdt_id_invariant_to_origin_path_change` (`identity.rs:324-338`, changes `components[2]` 0'→1' and asserts equal id) and `wdt_id_differs_for_different_use_site_paths` (`:340-351`). Empirically reproduced: same keyless degrade2 tree under `--path bip84/48/86` → WDT-Id `5b48af35d4321a3ac18b43045e2523cc` IDENTICAL across all three; wallet-policy-id and md1-encoding-id differ each time. So "template membership = spending conditions + use-site; wallet identity = addresses / WalletPolicyId" is correct, and `#25` use-site overrides ARE in the template-id domain (hashed at `identity.rs:79-98`). Use-site preservation is fine.

**The round-trip premise is FALSE for general policies — see Critical C1.** Eliding ALL origins to `Shared(empty)` produces a keyless template that **`md decode` REJECTS** for any non-canonical wrapper (degrade2, bare `wsh`, `tr(.,TapTree)`, `sh(sortedmulti)` legacy), because `decode_payload` runs `validate_explicit_origin_required` (`decode.rs:75` → `validate.rs:221-246`), which raises `Error::MissingExplicitOrigin{idx}` when `canonical_origin(&tree).is_none()` AND the per-`@N` path_decl/override is empty. `canonical_origin(degrade2.tree)` is `None` (the `wsh(or_i(...))` miniscript body hits the catch-all `_ => None`, `canonical_origin.rs:76-77`; cf. test `wsh_with_miniscript_body_returns_none` `:254-272`). The "Shared vs Divergent gate dissolves" claim is therefore unsound for exactly the shapes phase 2 exists to serve.

**Verdict: FLAWED (template-id invariance correct; the elide-to-`Shared(empty)` round-trip premise is false for non-canonical wrappers → C1).**

## D8
**Claim:** origins from keys at completion; use-sites from the template (per-slot, incl. #25 overrides); the search resolves own+cosigner slots simultaneously.

**SOUND, source-verified.** `compute_wallet_policy_id` DOES fold origins into the completion checksum: per per-`@N` record it writes `e.origin_path` (`identity.rs:193`) AND `e.use_site_path` (`:199`) into the preimage, so the id is origin-AND-use-site-sensitive → the id-search catches any origin or use-site mis-assignment (wrong → preimage differs → no match → refuse). `expand_per_at_n` (`canonicalize.rs:434-489`) sources origin per-`@N` (override beats `path_decl` baseline, `:436-444`) and use-site per-`@N` (override beats descriptor baseline, `:466-468`) — so use-site genuinely travels with the SLOT, not the key, and stays fixed while the search permutes (key+origin) across slots. At completion the origins are NON-empty (supplied from keys), so the `MissingExplicitOrigin` clause (`canonicalize.rs:450-455`: fires only when `origin_path.components.is_empty()`) does NOT fire — the completion descriptor is well-formed and id-computable even for degrade2. "Resolves which-slot-is-mine AND which-cosigner-is-which simultaneously" is coherent: all N keys (own + cosigner) are permuted across the N slots; the per-slot use-site/tree-position is the fixed frame. ✔

**Verdict: SOUND.**

## D9
**Claim:** origin cross-check = SOFT advisory, never reject; origin canonicality is NOT a funds-safety floor.

**SOUND, and now NECESSARY (not merely defensible).** degrade2 uses BIP-84 single-sig origins (`m/84'/0'/N'`) inside a multisig general policy. A rigid canonical-origin REJECT would kill this legitimate wallet: `canonical_origin` would offer `m/48'/0'/0'/2'` for a `wsh(multi)` but returns `None` for degrade2's miniscript body, and even where it returns `Some`, degrade2's BIP-84 origins are NOT the canonical BIP-48 — a canonical check would reject. The funds-safety gate is the address/id match (floor 5), which is origin-canonicality-AGNOSTIC: the address/scriptPubKey is computed from the actual supplied origin-derived child keys, and the id preimage hashes the actual origins (`identity.rs:193`). A WRONG origin → wrong derived key → wrong scriptPubKey/id → no match → refuse. So a non-canonical origin cannot leak a wrong wallet through the address/id gate: the gate is exact on the realized origin, and the advisory never overrides it. **No silent-wrong-wallet hole in the soft-advisory.** ✔ (One adjacency to C1: §4-floor-6's parenthetical "the template-id is origin-invariant, D7, so origins never affect card binding" is true, but the un-decodable-template defect (C1) means the card never gets engraved at all for these shapes — orthogonal to D9's funds-safety reasoning, which stands.)

**Verdict: SOUND.**

## D10
**Claim:** own keys may be multiple (one seed → many accounts); `--account` LIST primary + `--own-account-max K` subset-search; the subset×permutation space is `> N!`, cap-bounded; distinct-keys floor unaffected.

**Distinct-keys floor (§4-2) under multi-account own keys: SOUND.** degrade2's owner (fp `73c5da0a`) holds 4 slots at accounts 0,1,2,3 → four DISTINCT xpubs (`xpub6CatWdiZiodmU…`, `…odmY…`, `…odmb…`, `…odmd…` — all distinct strings; 11 distinct xpubs / 11 occurrences total, no reuse). Different account index ⇒ different child xpub ⇒ distinct 65-byte key ⇒ the pairwise-distinct floor is well-defined and does not false-fire on a multi-account owner. ✔

**The enlarged search space is real and breaks I2 — see Important I_new.** D10 correctly states the subset×permutation space is `> N!`, but does NOT propagate that into the §3.1-I2 prefix formula, which is still sized for `N!`. See the Interaction section.

**Subset-search finding a WRONG own-account subset:** for ADDRESS-search, impossible — a wrong own-account subset derives different child keys → different scriptPubKey → no match (collision-free, confirmed below). For ID-search, a wrong own-account subset is just another candidate in the enlarged space → subject to the I_new spurious-prefix risk, NOT a separate hole.

**Verdict: SOUND on distinct-keys; the `>N!` space is correctly identified but its consequence for the I2 prefix is not folded → I_new (Important).**

## Interaction: I2 prefix vs D10 search space (load-bearing)
The round-1/round-2 I2 close pinned the id-search prefix at `≥ ceil((log2(N!)+32)/8)` bytes (= 8 B at N=11, 9 B at N=13), sized so P(lone spurious | true assembly absent) ≈ `N!/2^(8·prefix)` ≤ ~2e-12, with a `+32`-bit margin. **That sizing assumes the candidate space is exactly `N!`.** D10's `--own-account-max K` subset-search enumerates a LARGER space: injective assignments of N slots drawn from a candidate-key universe of `(N − own_count) + K` keys (cosigners fixed + K own-account-derived candidates), i.e. `P(universe, N) = universe!/(universe−N)!`. With the **N!-sized 8-byte prefix** held fixed (degrade2, N=11, own_count=4):

| K (own-account-max) | universe | space `≈2^x` | P(lone spurious) @ 8-byte prefix |
|---|---|---|---|
| 4 (exact `--account 0,1,2,3`) | 11 | 2^25.3 | 2.2e-12 (safe) |
| 8 | 15 | 2^35.7 | 3.0e-9 |
| 16 | 23 | 2^45.6 | 2.9e-6 |
| 32 | 39 | 2^55.9 | 3.6e-3 (~1-in-275) |
| 64 | 71 | 2^66.5 | >1 (spurious match ~certain) |

The `+32`-bit margin is consumed by the space enlargement. The prefix MUST be sized to `ceil((log2(actual_space)+32)/8)` where `actual_space` is the realized subset×permutation count for the chosen `K` (and any explicit `--account` list collapses it back to `N!`), NOT the static `N!`. The doc's §3.1-I2 + §4-floor-5 use `N!`; D10 adds the enlarged space without patching the formula. **This is a real id-search silent-wrong-wallet path under `--own-account-max K`** (a lone spurious prefix-match on a wrong own-account subset is accepted as a unique match). → **Important I_new** (it's a formula-sizing gap, not a redesign; the strong-prefix mechanism already exists and just needs `actual_space` substituted; the surfaced-full-16-byte-id backstop and ambiguity-refuse remain).

**Address-search is UNAFFECTED — confirmed.** The predicate is full-256-bit-P2WSH-program equality; non-sorted shapes serialize children in stored slot order (round-1/round-2-verified `astelem.rs:155-188`; only `Sorted*` reorders via BIP-67), so distinct (subset, permutation) → distinct script bytes → distinct program → distinct address, with NO prefix-length dependence regardless of space size. The recommendation "address-search is the RECOMMENDED primary; id-search is runtime-weaker" is reinforced by D10: the gap is id-search-only.

Even after C1 is fixed, the wording should be scoped: `Shared(empty)` is admissible ONLY for canonical-wrapper shapes (`pkh/wpkh/tr-keypath/wsh(multi|sortedmulti)/sh(wsh(multi|sortedmulti))` — `canonical_origin == Some`); non-canonical wrappers MUST retain explicit per-`@N` origins on the wire. 4. **§8 should add** the I_new prefix-resizing item and the C1 non-canonical-origin-retention item to the SPEC-time decide list.

## Verdict
**RED — 1 Critical, 1 Important.**

- **C1 (Critical, feature-correctness / un-engravable card).** D7's "elide ALL origins to `Shared(empty)`" yields a keyless general-policy template that fails `md decode` (`validate_explicit_origin_required`, `decode.rs:75`/`validate.rs:221-246`; `MissingExplicitOrigin` because `canonical_origin(degrade2.tree)==None`). The flagship degrade2 and every non-canonical-wrapper general policy cannot round-trip → the engraved card is un-decodable. Empirically reproduced (md-cli 0.7.1: keyless+origin-elided general policy encodes but `md decode`/`md inspect` error `non-canonical wrapper requires explicit origin for @0`). Not a silent-funds-loss path (it's a hard decode failure, fails loud), but it breaks the central deliverable for the shapes phase 2 targets and is a false round-trip premise underpinning D7/§5-Slice-1/§8. **Fix:** elide origins to `Shared(empty)` ONLY when `canonical_origin(&tree).is_some()`; for non-canonical wrappers RETAIN the explicit per-`@N` origins (the existing `synthesize_unified:899-908` `Shared`/`Divergent` build already produces them) — the template-id stays origin-invariant either way (D7's invariance is real), so the card binding is unchanged; only the wire keeps the origins it needs to decode. Re-word §2a-D7, §3-Slice-1, §4-floor-6, §8.
- **I_new (Important, id-search silent-wrong-wallet under D10).** The §3.1-I2 / §4-floor-5 strong-prefix is sized for an `N!` space; D10's `--own-account-max K` subset-search enumerates `P((N−own)+K, N) > N!`, consuming the `+32` margin (K=16 → ~3e-6; K=32 → ~1-in-275; K=64 → spurious match near-certain). **Fix:** size the id-search prefix to `ceil((log2(actual_subset×permutation_space)+32)/8)` bytes (computed from the realized `K`/`--account` list at search start, where the cap estimator already enumerates the space), not the static `N!`; an explicit `--account` list collapses it back to `N!`. Address-search is unaffected (full scriptPubKey collision-free regardless of space size — re-confirmed) → reinforce "address-search primary; id-search opt-in, full-id recommended."

D8/D9 are SOUND and source-verified; the distinct-keys floor holds under multi-account own keys; no round-1/round-2 finding is reopened (I_new is a sizing delta on I2's already-correct mechanism). Architecture remains sound — both findings are bounded folds (scope the origin-elide to canonical wrappers; resize the id-prefix for the actual search space), not a redesign. Persist verbatim, fold, re-dispatch, converge 0C/0I before SPEC.
