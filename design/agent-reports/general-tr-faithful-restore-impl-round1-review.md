## Funds-safety adversarial pass — what I tried, and why it held

1. **Independent golden verification (the strongest leg).** Spun up an offline Bitcoin Core v25 instance and ran `getdescriptorinfo` + `deriveaddresses` on the two-leaf and multi_a-in-2leaf golden descriptors (external chain `/0/*` substituted for `/<0;1>/*`; Core v25 lacks multipath). Core independently reproduced **`bc1prgf4vyj…kms0ch` and `bc1pw49n2w…s88s86m` byte-exact** — the goldens are not self-referential toolkit output; the NUMS internal key, taptree construction, and address derivation agree with a foreign implementation. (The single-leaf `and_v/after` golden is not Core-v25-verifiable — tapscript miniscript support postdates v25 — so that golden rests on derive-once-then-pin + the parse→print guard.)
2. **Could a non-H-point `Single` slip through, or the H-point appear somewhere wrong?** Grepped md-codec 0.35.3 source (`descriptor-mnemonic` @ 7dd2ff0): `build_nums_internal_key` (to_miniscript.rs:183-190) is the **only** `DescriptorPublicKey::Single` producer, always the H-point, only at the tr internal-key position; wire keys ([chain_code‖pubkey]) always reconstruct as `XPub`. Probe: a raw x-only H-point as a *leaf* key is unreachable — `bundle` refuses it ("synthetic key … not found in key map", exit 2). The pass-through also cannot interact with multipath promotion: it returns the key unchanged, and Core's acceptance of the mixed fixed-internal-key/wildcard-leaf descriptor confirms the geometry.
3. **Gate bypass attempts.** Left-heavy and right-spine depth-2: both refuse exit 2 (chirality-independent, structural). Depth-2 + sortedmulti_a: refuses (sortedmulti slug fires first — both ModeViolation 2, both truthful). A hypothetical nested `Tr`-under-TapTree would slip the depth gate's `tag == TapTree` check but fails loudly in md-codec's `node_to_miniscript` (no `Tr` arm) → `bad()` exit 1 — loud, not silent-wrong; and the shape is not wire-constructible anyway. Keypath-only `tr(K0)`: bundles, restore refuses exit 2 via the is_nums:false arm. Malformed TapTree arity: wire-impossible, and the defensive arm refuses rather than treating it as a leaf.
4. **`extract_multisig_threshold` Some(k) misuse.** For the general-tr arm, `threshold_field` is forced None in `--json` (:1533); `EmitInputs.threshold` can only reach emitters in the general arm, and I probed EVERY remaining format at HEAD: specter exit 2 (`MissingField::WalletName` — restore has no `--wallet-name` flag, so this refusal is unsatisfiable-by-construction), electrum/sparrow/jade/coldcard exit 1 (template-required), bsms exit 2 (taproot refused), green exit 1 both legs (P2tr new refusal / P2trMulti existing is_multisig), bip388 exit 1 (multipath). descriptor/bitcoin-core emit the faithful descriptor, watch-only. **No format emits a wrong-but-plausible payload.**
5. **Shape breadth.** `tr(NUMS,pk(K0))` single-leaf, single-leaf `sortedmulti_a` (Template arm), and `tr(NUMS,{pk(K0),and_v(v:pk(K1),older(52560))})` all restore exit 0 with timelocks/structure preserved and the NUMS hex internal key — no silent collapse anywhere.
