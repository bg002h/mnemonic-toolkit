## Critical (must fix before plan)

- **[C1] SPEC §4/§5's dead-code mechanism is wrong — the byte-parity unit test does NOT keep ported-but-unconstructed variants/fns live; the enum needs `#[allow(dead_code)]`.** SPEC §4 (`SPEC...phase2_mk_md.md:77`): *"The `PrivateKeyMaterial` arm is ported verbatim though never constructed... exercised by the byte-parity unit test (§5), which keeps it live (no `dead_code` → no clippy `-D` failure)."* This is false in a **bin-only crate**. Empirical probe (`/tmp/deadcode_probe`, `cargo clippy --all-targets -- -D warnings`) produced: `error: variants Template and PrivateKeyMaterial are never constructed` and `error: function worst_class_on_stdout is never used` — even with the variant/fn exercised ONLY in `#[cfg(test)]`. Both mk-cli and md-cli are bin-only (no `src/lib.rs`: `mnemonic-key/crates/mk-cli/src/` and `descriptor-mnemonic/crates/md-cli/src/` contain only `main.rs`), so `pub` does not suppress dead-code, and `#[cfg(test)]` construction does not keep bin-target items live. Both CIs enforce `cargo clippy --workspace --all-targets -- -D warnings` (`mnemonic-key/.github/workflows/ci.yml:58`, `descriptor-mnemonic/.github/workflows/ci.yml:47`; md additionally sets global `RUSTFLAGS: "-D warnings"` at `:10`). **For mk-cli this is worse than the SPEC admits:** mk real code constructs ONLY `WatchOnly`, so BOTH `Template` AND `PrivateKeyMaterial` are never-constructed → both flagged. For md, only `PrivateKeyMaterial` is never-constructed. **The Phase-1 precedent the SPEC cites already did this correctly with `#[allow(dead_code)]`** — see C2. *Fix:* put `#[allow(dead_code)]` on the `OutputClass` enum (and on `worst_class_on_stdout` if it is ported at all — see C2) in both ports; drop the §4 claim that "the test keeps it live."

- **[C2] SPEC §4/§5 contradict the Phase-1 precedent they invoke: ms-cli did NOT port a byte-identical Ord-deriving enum or `worst_class_on_stdout`, and its byte-parity test asserts only the three literal strings.** SPEC §4 mandates *"the full 3-variant `OutputClass` enum + `emit_output_class_advisory` **byte-identical** to `secret_advisory.rs`"* plus *"`worst_class_on_stdout` for the multi-card subcommands repair/inspect"*; SPEC §5 says the unit test asserts *"the lattice `.max()` ordering"* and *"`worst_class_on_stdout(&[]) == None`"*. But the actual Phase-1 port at `mnemonic-secret/crates/ms-cli/src/advisory.rs` shows: (a) the enum is `#[allow(dead_code)] #[derive(Debug, Clone, Copy, PartialEq, Eq)]` with variant order `{ PrivateKeyMaterial, WatchOnly, Template }` (`:26-32`) — **NOT** byte-identical to the toolkit's `#[derive(... PartialOrd, Ord)] enum OutputClass { Template, WatchOnly, PrivateKeyMaterial }` (`mnemonic-toolkit/.../secret_advisory.rs:79-80`); it deliberately drops `PartialOrd/Ord` and reverses the order; (b) `worst_class_on_stdout` is **not ported at all**; (c) the byte-parity test (`ms-cli/tests/cli_output_class.rs::byte_parity_advisory_lines`) asserts only the three hard-coded literal lines, not any `.max()` lattice. The only thing that must be byte-identical is the **three literal strings** in `emit_output_class_advisory` (which ms-cli writes with `\u{2014}` escapes, `:40-41`). For mk/md, every emitting handler produces a **single** card-kind per invocation (verified: mk `repair`/`inspect` emit one `WatchOnly`; md `repair`/`inspect` emit one `Template`), so `worst_class_on_stdout` is unnecessary dead code exactly as for ms. *Fix:* rewrite §4/§5 to mirror the actual ms-cli precedent — port only `emit_output_class_advisory` + the (allow-dead-code) enum needed to call it; do NOT port `worst_class_on_stdout`; do NOT derive `Ord`; scope "byte-identical" to the three literal lines; model the unit test on `byte_parity_advisory_lines` (literal-string assertions), not on the toolkit-internal `output_class_lattice_and_lines`.
