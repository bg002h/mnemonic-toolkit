## 1. F5 fix ‚Äî split the nested subcommand name in the assembler
**The crux (recon):** the toolkit flattens as `format!("{parent}-{child}")` (`toolkit gui_schema.rs:1015-1017`) and BOTH parent and child names contain hyphens, so a naive hyphen-split corrupts `ms-shares`/`seed-xor`/`xpub-search-*` AND would break flat `import-wallet`/`export-wallet`. No structural marker survives the flatten. The ONLY reliable discriminator is the **closed set of 5 nested-parent prefixes**: `seed-xor`, `seedqr`, `slip39`, `ms-shares`, `xpub-search` (verified: no flat subcommand begins with any of these complete-token-plus-`-` prefixes).

Fix (recon option A ‚Äî GUI-only, minimal): in `assemble_argv_with_secret_mask` (`invocation.rs`), replace the single push at `:161-162` (`argv.push(subcommand.name.to_string()); mask.push(false);`) with a push of the nested token SEQUENCE via a helper:
```rust
const NESTED_PARENTS: &[&str] = &["seed-xor", "seedqr", "slip39", "ms-shares", "xpub-search"];
/// Reverse the toolkit's `{parent}-{child}` gui-schema flattening: a flattened
/// name is nested IFF it begins with "<parent>-" for one of the 5 known nested
/// parents (import-wallet/export-wallet etc. are FLAT and pass through unsplit).
fn subcommand_argv_tokens(name: &str) -> Vec<&str> {
    for p in NESTED_PARENTS {
        if let Some(child) = name.strip_prefix(*p).and_then(|r| r.strip_prefix('-')) {
            return vec![*p, child];   // M-2: `*p` (p is `&&str`); e.g. ["seed-xor", "combine"]
        }
    }
    vec![name]
}
```
Push each token with a `mask.push(false)` per token (both tokens non-secret ‚Äî mask parallelism preserved). This fixes Run AND Copy at once. **M-2:** also update the stale assembler doc-comment invariant "2. `argv[1]` = `subcommand.name`" (`invocation.rs:116`) ‚Äî it's now the nested token sequence.

The 12 nested forms + their correct argv: `seed-xor {split,combine}`, `seedqr {encode,decode}`, `slip39 {split,combine}`, `ms-shares {split,combine}`, `xpub-search {path-of-xpub, account-of-descriptor, address-of-xpub, passphrase-of-xpub}`.

## 2. F6 fix ‚Äî `""`-PREPEND inference sentinel on the two address-of-xpub dropdowns
The `(none)` sentinel mechanism (shipped v0.57.0 for restore/export-wallet templates): a `""` entry in a Dropdown's choice list stores as `Dropdown("")` ‚Üí displays `(none)` (`widget.rs:528,535` `display_or`) AND emits NOTHING (`emit_one` Dropdown arm guards `if !v.is_empty()`, `invocation.rs:415`). **CRUX ‚Äî PREPEND, not APPEND:** restore APPENDED `""` (keeping `opts[0]="bip44"` as a real virgin default ‚Äî `bip44` is sensible, `(none)` is opt-in). F6 needs the OPPOSITE ‚Äî for `address-of-xpub` `--address-type`/`--network` there is NO safe default (the toolkit MUST infer from the SLIP-0132 prefix), so `""` must be at **`opts[0]`** so a virgin form materializes `Dropdown("")` ‚Üí emits nothing ‚Üí the toolkit infers.

Fix: two GUI-only choice consts with `""` prepended, pointed at by the two `address-of-xpub` flags ONLY (`schema/mnemonic.rs:3243` `--address-type`, `:3256` `--network`; keep `default_value:None`):
```rust
const XPUB_SEARCH_ADDRESS_TYPES_INFER: &[&str] = &["", "p2pkh", "p2sh-p2wpkh", "p2wpkh", "p2tr"];
const NETWORKS_INFER:                  &[&str] = &["", "mainnet", "testnet", "signet", "regtest"];
```
Do NOT mutate the shared `NETWORKS` (15 `Dropdown(NETWORKS)` sites want the concrete default) or `XPUB_SEARCH_ADDRESS_TYPES`. Only `address-of-xpub`'s two dropdowns switch to `_INFER`.

### F6 PERSISTENCE MIGRATION (I-3 ‚Äî REQUIRED; the consts-only fix leaves the at-risk population bugged)
Autosave persists per-subcommand non-secret `values` incl. Dropdowns and rehydrates them (`persistence.rs:79-120,:342`). Any user who EVER opened `address-of-xpub` on ‚â§ v0.57.0 has `("--address-type", Dropdown("p2pkh"))` + `("--network", Dropdown("mainnet"))` PERSISTED ‚Äî the consts change alone leaves those emitting, so the funds false-negative **persists post-upgrade for exactly the affected users**. **Fold decision (recon-reviewed option a):** extend the load-time normalization (`normalize_loaded_form_values`, `persistence.rs:342-401` ‚Äî the hint-text-migration precedent, currently Text/Path-scoped) to DROP the `address-of-xpub` `--address-type` and `--network` persisted Dropdown values **when they equal the old materialized `opts[0]`** (`"p2pkh"` / `"mainnet"` respectively) ‚Üí reset to the inference sentinel (`Dropdown("")` / removed) so a rehydrated form emits nothing. Fail-safe: a user who DELIBERATELY chose `p2pkh`/`mainnet` is also reset, but inference is the safe default and the value is trivially re-selectable (vs. a silent wrong ownership verdict, which is not). Gate the migration on the exact persisted key + flag + old-value triple: **the on-disk subcommand key is the FLATTENED `sub.name` ‚Äî literally `"mnemonic:xpub-search-address-of-xpub"`** (`persistence.rs:59-60` `"<cli>:<sub.name>"`, `sub.name` at `schema/mnemonic.rs:4645`, UNCHANGED by the F5 assembler fix), the flags are `--address-type`/`--network`, the old values are `"p2pkh"`/`"mainnet"`. So it never touches other forms' persisted dropdowns. Add a migration unit test whose fixture uses THAT exact map key through `load()` (persisted stale values ‚Üí post-load carries no `p2pkh`/`mainnet` for those two flags; a persisted NON-default like `p2tr` is preserved; a fixture under any other key is a no-op).

**Scope (recon): SURGICAL to `address-of-xpub` this cycle** (the ONLY proven funds bug ‚Äî the toolkit infers there; the broader ~20 `Dropdown+None` flags mostly emit `opts[0]` == the toolkit's own default = benign).
