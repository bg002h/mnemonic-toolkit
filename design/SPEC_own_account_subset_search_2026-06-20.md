## 1. Summary

Today `complete_multisig_template` (the shared restore/verify-bundle core, `restore.rs:1421`) requires the supplied keys to EXACTLY fill the N slots (`pool.len() == n`, gates `:1687`/`:1696`) and enumerates `n!` orderings via `unrank_permutation` over `(0..n)` (`permutation_search.rs:494`). ## 3. The own-anchored count (the load-bearing math ‚Äî from the brainstorm D2, R0-GREEN)
`N` slots; `M` cosigner cards (own-only: exact, all used); `j = N ‚àí M` own-slot-count (own-only: determined); `K_own` own candidates. **Own-only enumerated count:**
> **`S_own = C(K_own, j) ¬∑ N!`**, `j = N ‚àí M`. (Choose `j` of `K_own` own: `C(K_own,j)`; assign the `j` own + `M` cosigners to `N` slots: `N!`.) Collapses to `N!` at `K_own = j` (byte-identical to v0.60.0 `perm_count_u128(n,n)`, `restore.rs:1722-1723`).

**Sorted-shape composition ‚Äî ENUMERATION-SIDE (open-point 5 + R0-r1 I-1 resolved):** for an order-INDEPENDENT shape (`sortedmulti`/`sortedmulti_a`, `is_order_independent_shape` at `synthesize.rs:335`), all `N!` orderings of a fixed key SET are the same wallet. **The mechanism is ENUMERATION-SIDE, NOT the v0.60.0 evaluator-filter** (R0-r1 I-1): v0.60.0 enumerates the full `n!` and the evaluator-closure `restore.rs:1800` (`if sorted_shape && assignment != identity ‚Üí false`) DISCARDS non-identity orderings match-side. For the SUBSET path that filter is WRONG to reuse verbatim ‚Äî "identity" is per-subset, so a verbatim `assignment==identity` skip would discard every non-first SUBSET. Instead, the own-anchored generator for a sorted shape **drops the `perm_rank` factor**: it emits exactly the `C(K_own, j)` distinct subsets, each in a single canonical (identity) slot order. Then **`S_own_sorted = C(K_own, j)`** is simultaneously the enumerated count, the cap-ETA scan count, AND the colliding-set size (the three COLLAPSE ‚Äî no prefix-vs-ETA divergence; the `realized_s` symbol is unambiguous). Distinct subsets ‚áí distinct sorted-pubkey multiset ‚áí distinct scriptPubKey (via the ¬ß5 distinct-keys floor) ‚Äî no two-subset address collision. (The v0.60.0 EXACT path ‚Äî `pool.len()==n`, no over-supply ‚Äî keeps its evaluator-filter mechanism byte-unchanged; only the subset path uses the enumeration-side generator.)

**Opt-in count** (`--search-cosigner-subset` ON): the cosigner count is uncertain ‚Äî the pool is `K_own` own + `M_sup` supplied cosigner candidates, and the search ranges over valid `(j, cosigner-subset)`. `S_opt ‚â§ P(K_own + M_sup, N)` (the uniform bound); the engine sizes `realized_s` to the ACTUAL enumerated count for the chosen enumeration (SPEC ¬ß4.3) and ¬ß6 hard-ceilings it. (The opt-in exact enumeration formula is pinned in ¬ß4.3.)

**`realized_s` = the ACTUAL enumerated count** (`S_own` / `S_own_sorted` / `S_opt`) ‚Äî NEVER `n!`, NEVER the looser `P(pool,N)` for own-only. Closed-form ‚áí computable up-front for `validate_prefix_strength` (`:1761`) + the cap ETA. **FLOOR ‚Äî enumerated ‚â° counted:** the generator MUST enumerate EXACTLY the counted set (bijection), verified by a brute-force-reference test (¬ß7). A scanned-but-uncounted placement = a collision the prefix wasn't sized for (the dangerous direction).

## 4. Engine changes (`permutation_search.rs`)
### 4.1 Subset-select enumeration
Add a pool-aware enumerator generalizing `unrank_permutation(rank, n)` (`:494`, which builds `elems=(0..n)`):
- **`unrank_kperm(rank, pool, n) -> Vec<usize>`** ‚Äî the `rank`-th injective k-permutation (lexicographic): place `n` of `pool` items into `n` ordered slots, returning the `pool`-indices. Count `P(pool, n)`. O(n¬∑pool) Lehmer-style unrank. Used by the OPT-IN/uniform path.
- **The OWN-ANCHORED generator** ‚Äî own-only is NOT a plain `unrank_kperm` over the whole pool (that includes cosigner-dropping placements `S_own` excludes). It is a composed rank over the `S_own = C(K_own,j)¬∑N!` space: `rank ‚Üí (combo_rank ‚àà [0,C(K_own,j)), perm_rank ‚àà [0,N!))`; `combo_rank` unranks to the `j`-subset of the `K_own` own indices (combinatorial-number-system unrank); `perm_rank` unranks (via the existing `unrank_permutation(perm_rank, N)`) to the ordering of the `N` selected keys (`j` chosen own + all `M` cosigners) into the `N` slots. **Bijective over `S_own` by construction** ‚Äî every assignment using exactly `j` own + all `M` cosigners, each once, nothing else. (Sorted shape: drop the `perm_rank` factor ‚Äî identity order only, space `C(K_own,j)`.)
- **`total_candidates` / cardinality:** extend (or add `total_candidates_subset(mode, S)`) so the search drives `S` candidates where `S = realized_s` (the ¬ß3 count), not `factorial(n)`. **All cardinality helpers REFUSE on `u128` overflow ‚Üí `None` ‚Üí `bad(...)`** (the #28 M1 lesson; `perm_count_u128` ‚Äî a private fn at **`restore.rs:1943`** [RELAUNCH mislabel-fix: it is NOT in `permutation_search.rs`] ‚Äî already does). This INCLUDES a NEW overflow-checked **combination helper `c_choose(K_own, j) -> Option<u128>`** (R0-r1 m-3 ‚Äî `C(K_own,j)` can overflow `u128` for `K_own` near the ¬ß6 ceiling, e.g. `C(256,128)`; the helper `checked_mul`s like `perm_count_u128` and returns `None` ‚Üí `bad`). `S_own = c_choose(K_own,j)? .checked_mul(factorial(N)?)?` ‚Äî overflow at any step refuses.

### 4.2 Sharding (R0-r2 confirmed structure-agnostic)
`search` (`:551`) shards the rank space `[0, S)` across `min(20, ncpu)` threads. The shard logic needs only `S` (cardinality) + a bijective `unrank` ‚Äî both provided. The own-anchored composed rank shards identically (contiguous rank ranges). No sharding redesign.

### 4.3 Opt-in exact enumeration
`--search-cosigner-subset` ON: the search ranges over `(own-subset, cosigner-subset, ordering)`. To keep `realized_s == enumerated`, the enumeration is fixed as: for the supplied `K_own` own + `M_sup` cosigner candidates filling `N` slots with `j` own + `(N‚àíj)` cosigner for each valid `j ‚àà [j_min, j_max]` ‚Äî count `S_opt = Œ£_j C(K_own,j)¬∑C(M_sup, N‚àíj)¬∑N!`. `realized_s` = that sum (closed-form, overflow-checked per ¬ß4.1). Disjoint by construction (each assignment has exactly ONE `j` = its own-slot-count) ‚áí no double-count. `j_min=1` (own ‚â•1 via `--from`), `j_max=min(K_own, N‚àí1)` (‚â•1 cosigner). **The opt-in GENERATOR (R0-r1 axis-3 note) is the STRATIFIED composition of ¬ß4.1's primitives:** partition `[0, S_opt)` into the j-strata (cumulative `C(K_own,j)¬∑C(M_sup,N‚àíj)¬∑N!`); within stratum-`j` compose `(own-combo-rank via CNS-unrank, cosigner-combo-rank via CNS-unrank, perm-rank via unrank_permutation(N))` ‚Äî bijective onto the stratum by the same argument as ¬ß4.1. Brute-force-reference-tested per ¬ß7-P3 (m-2). The ¬ß6 hard ceiling refuses if `S_opt` is too large. (Sorted shape: drop the `¬∑N!` per stratum ‚Äî identity order, `Œ£_j C(K_own,j)¬∑C(M_sup,N‚àíj)`.) `--own-slots <j>` pin flag is deferred to the plan (the inferred range is safe; open-point 3).

### 4.4 Early-exit policy (open-point 4 resolved)
- **prefix id-search:** FULL-SCAN (ambiguity certification ‚Äî ‚â•2 matches ‚Üí `Ambiguous` ‚Üí refuse). Own-first ORDERING does not shortcut this; the time-cap bounds it.
- **address-search:** collision-free (full scriptPubKey; the whole-pool distinct-keys floor `:1709` guarantees distinct subsets ‚áí distinct programs, ¬ßI-2) ‚áí a match is provably UNIQUE ‚áí **early-exit on first match is SAFE**.
- **CONTRACT (R0-r1 I-5 ‚Äî promoted from a plan open-item into a hard invariant).** The live engine `search` (`:551`) today FULL-SCANS BOTH modes (it short-circuits only at the 2nd match for ambiguity ‚Äî doc `:535-548`), and the engine is SHARED by the v0.60.0 EXACT path. Adding first-match early-exit is a real engine behavior change that MUST be gated: **address-search first-match early-exit is enabled IFF (over-supply path: `realized_s ‚âÝ n!`) AND `mode == Address`.** The v0.60.0 EXACT path (`pool.len()==n`) AND all id-search/prefix-id paths RETAIN the unchanged full-scan-with-2nd-match-ambiguity behavior, **byte-identical to v0.60.0**. The engine knob is an explicit per-call **`early_exit: bool`** (or a `SearchMode::Address { early_exit }` variant) ‚Äî named here so the plan author does not improvise the gating. **¬ß7 pins a v0.60.0 EXACT-path address-search outcome byte-for-byte before/after this change** (the regression guard). Prefix-id NEVER gets early-exit (it would miss a 2nd-match ambiguity ‚Üí silent-wrong-wallet).
- **full (non-prefix) wallet-id:** collision-safe ‚áí early-exit permitted but NOT required (keep full-scan for simplicity unless a perf need arises; not funds-safety-load-bearing ‚Äî a full id is unique).
- **Own-first ordering:** the enumerator yields own-anchored / low-account-index assignments first (the common case) so address-search hits the real wallet early. Ordering is a perf heuristic, NOT a correctness mechanism (the cross-link to ¬ß4.4 prefix-id full-scan is mandatory ‚Äî R0-r1 M-2).
