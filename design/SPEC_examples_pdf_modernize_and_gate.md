## 0. Summary

`docs/Examples.pdf` — the CLI tutorial — is generated by `.examples-build/gen.sh`, hard-pinned to
`mnemonic 0.55.3` (`gen.sh:22`), with **zero CI gating** (no workflow in `.github/workflows/` references

This cycle: **(1)** modernize the pin to the fixed, reproducible `mnemonic 0.75.0` tier and regenerate;
**(2)** rewrite every prose line coupled to output that changed (chiefly the now-false "Each card is printed
twice…" narration, `gen.sh:209-211`); **(3)** make gen.sh's output a *pure function of (repo, mnemonic binary)* —
eliminating environment-dependent branches and path/HOME leakage that would break a CI diff; **(4)** add
`.github/workflows/examples.yml` (gate A): build `mnemonic` from source → re-run gen.sh → **diff the fresh
`Examples.md` against a newly-committed golden `.examples-build/Examples.md`** → fail-closed on drift; build the
PDF; attach it to the release on an `examples-v*` tag. Appendix B's `mnemonic-depth2` reconstruction becomes a
clearly-labelled **static committed capture** that gen.sh emits verbatim (no execution → nothing for CI to
rebuild → the exemption is structural, not a diff-carve-out).

**NO-BUMP, toolkit-only, docs/CI-only** — zero `crates/**` source changes, no flag changes, no cross-repo
locksteps (`schema_mirror` / manual flag-coverage do not fire).

## 2. D1 — Pin modernization + determinism/portability hardening

### D1.1 Pin sites (mechanical)

Bump `0.55.3` → **`0.75.0`** (a FIXED reproducible tier — the current `crates/mnemonic-toolkit/Cargo.toml`
version — never a floating "current") at:

1. `gen.sh:22` — the FATAL equality check (the enforcement point).
2. `gen.sh:3` — header comment.
3. `gen.sh:87` — YAML `subtitle:`.
4. `gen.sh:88` — YAML `date:` → the regen date (static text; NEVER `$(date)` — determinism).
5. `gen.sh:104` — "`mnemonic` **v0.55.3** on Linux" intro prose.
6. `gen.sh:696` — Appendix A "Generated with `mnemonic` v0.55.3 on Linux".
7. `.examples-build/.gitignore:5` — the "Run with mnemonic 0.55.3" comment (recon missed this site; also reword
   its "Everything else here is a build artifact" framing since `Examples.md` becomes tracked, §6.2).
8. Appendix-B framing sites `gen.sh:709,716-717,724` — restructured under D4, not merely bumped.

Historical-attribution prose is **kept as history**, disambiguated where it could read as the doc's binary
version: `gen.sh:531` "the v0.55.3 non-NUMS feature" → "the non-NUMS internal-key feature (shipped in v0.55.3)";
`gen.sh:666` "which v0.49.1 routes *around* the codec" → unchanged (clearly historical).

**Consequence (accepted, by design):** with gate A live, **every future toolkit release that bumps the crate
version turns `examples.yml` red** until the same change re-pins `gen.sh:22` + sites above and regenerates the
golden (the freshly-built binary reports the new version → FATAL at `gen.sh:22`; independently, `install.sh`'s
self-pin advances → the captured `--list`/`--dry-run` blocks drift). This converts the Examples regen from a
memory-enforced release-ritual item into a **gate-enforced** one — the loud-rot discipline the FOLLOWUP asks for.
The per-release cost is mechanical (~pin sites + `bash gen.sh > Examples.md` + pandoc + commit). Listed for the
release-ritual checklist in §8.

### D1.2 Binary sourcing

- gen.sh resolves `mnemonic` via `PATH` after `export PATH="$HOME/.cargo/bin:$PATH"` (`gen.sh:18`). Add an
  override hook, prepended AHEAD of `~/.cargo/bin` so CI's source-built binary always wins over any stale
  cargo-installed one:
  `export PATH="${EXAMPLES_BIN_DIR:+$EXAMPLES_BIN_DIR:}$HOME/.cargo/bin:$PATH"`.
  CI sets `EXAMPLES_BIN_DIR="$GITHUB_WORKSPACE/target/debug"`. Displayed commands stay bare `mnemonic` —
  display-faithful, execution-pinned.
- CI builds **debug** (`cargo build --bin mnemonic`) — precedent: `manual.yml:92-96` ("Debug binary is sufficient
  for transcript-replay byte-comparison"). The `gen.sh:22` version check then validates the built binary.

### D1.3 Environment stability (new — required for a byte-diff gate; the recon under-flagged this)

The committed baseline **leaks author-machine paths into captured output**, which would diff on any CI runner:

- `Examples.md:98,109` — `$ sh '/scratch/code/shibboleth/mnemonic-toolkit/scripts/install.sh' --list` (the
  hardcoded `REPO=/scratch/...` at `gen.sh:15`, expanded at display time by the double-quoted `run` calls
  `gen.sh:179-180`).
- `Examples.md:111,116-136` — `install.sh --no-gui --dry-run` output embeds `/home/bcg/.cargo/bin` and
  `/home/bcg/.local/share/man/man1` (install.sh derives them from `${CARGO_INSTALL_ROOT:-$HOME/.cargo/bin}` and
  `${XDG_DATA_HOME:-$HOME/.local/share}`, `scripts/install.sh:75,322,336`).

Fixes (P1):

1. `gen.sh:15` → `REPO="${REPO:-$(cd "$(dirname "$0")/.." && pwd)}"` (script-relative; works from any checkout).
2. `gen.sh:179-180` → single-quote the two install.sh `run` calls so the *display* shows the literal
   `sh "$REPO/scripts/install.sh" --list` (unexpanded — `run` prints `$1` verbatim and `eval`s it, so execution
   still expands `$REPO`); add a half-line of prose: "`$REPO` = your clone root".
3. After PATH resolution (order matters — PATH is computed from the real `$HOME` first), pin the output-visible
   environment: `unset XDG_DATA_HOME CARGO_INSTALL_ROOT; export HOME=/home/user; export LC_ALL=C LANG=C TZ=UTC`.
   `--dry-run` then deterministically prints `/home/user/...` on every machine. Residual risk that some invoked
   tool needs the real `$HOME` is retired by the P1 determinism gate (§9): regen twice under two different real
   HOMEs → byte-identical.
4. Strict preflight (`gen.sh`, always on): FATAL up-front with a clear message if `jq`, `python3`, or (per D1.4)
   `bitcoind`/`bitcoin-cli` are missing — today a missing `jq` would silently capture `command not found` *into
   the document body* via `run`'s `eval … 2>&1` (`gen.sh:78`).

TTY note: `run` captures via `$( … 2>&1)` — never a TTY — identically local and CI, so `is_terminal`-conditioned
behavior (e.g. the `--no-auto-repair` help text's verify-bundle TTY gate) cannot diverge.

### D1.4 Eliminate environment-dependent output branches

For a whole-document diff gate, gen.sh output must not depend on what happens to be installed:

- **§6.6 bitcoind cross-check (`gen.sh:656-679`)**: currently `if command -v bitcoind … else show-fallback`. The
  committed baseline captured the **live** branch (`Examples.md:1474` — the real `deriveaddresses` output; the
  address is deterministic for the fixed descriptor). **Decision: keep the live capture.** CI installs the pinned
  **Bitcoin Core v27.0** exactly as `.github/workflows/bitcoind-differential.yml:60-100` already does
  (sha256-verified tarball + actions/cache, offline `-chain=main`) — an in-repo, already-maintained pattern.
  gen.sh drops the silent `else` branch: missing bitcoind = preflight FATAL (D1.3.4), never a silently different
  document. (Alternative, if R0 judges Core-in-CI over-scoped for gate A: flip §6.6 to the `show`-fallback as the
  golden — but that *downgrades* the committed doc by deleting a real external-oracle capture, so it would need a
  user sign-off; see STOP ledger S2.)
- **Appendix B (`gen.sh:724-774`)**: the `if command -v mnemonic-depth2` branch is removed entirely — see D4.
  This also removes the author-machine dependence (the recon author *had* `mnemonic-depth2` installed; CI never
  will — today's gen.sh would emit two different documents).

### D1.5 Determinism statement

With D1.1–D1.4 applied, gen.sh output is a pure function of (repo tree, `mnemonic` binary): all keys derive
**live** from the three hard-coded public BIP-39 vectors (`gen.sh:27-29`) via `mnemonic convert`; the 4 tracked

## 5. D4 — Appendix B: the gate exemption (ratified) — mechanism

**Ratified constraint:** mainline 0.75.0 still refuses depth-≥2 (recon-verified byte-identical refusal); the
never-merged `experimental/taproot-depth-ge2` build is NOT CI-reproducible (unreleased rust-miniscript #953 pin);
the gate must never try to rebuild it.

**Mechanism — "B-static":** delete the `if command -v mnemonic-depth2` runtime branch (`gen.sh:724-774`,
including the else-fallback at `:764-773`) and emit Appendix B **unconditionally** as:

1. Framing prose (rewritten per D3) + the existing `show` block of build steps (`gen.sh:719-722`) — unchanged in
   spirit: how a reader would build the twin themselves.
2. **Live** (executed, gated) commands that need only the shipped binary — all recon-verified stable:
   `run 'cat taproot-4leaf.desc'` (`gen.sh:735`), the `bundle --json | jq` md1-chunk extraction + `cat`
   (`gen.sh:742-743`), and the mainline **refusal** `run 'mnemonic restore …'` (`gen.sh:749`). Keeping the
   refusal LIVE is a feature: the gate then *proves on every CI run* that mainline still refuses depth-≥2 — if a
   future miniscript bump silently lifts the cap, examples.yml goes red and forces a doc decision.
3. One **static committed capture**: the `mnemonic-depth2 restore` reconstruction (today's live output,
   `Examples.md:1597-1617`), frozen as a literal heredoc in gen.sh with an explicit in-document label, e.g.:
   > *(STATIC CAPTURE — recorded 2026-06-15 from the experimental `mnemonic-depth2` build at v0.55.3. This block
   > is NOT regenerated by `gen.sh` and is not reproducible with a released binary or in CI; build the branch as
   > shown above to reproduce it. Everything else in this document is live-captured and CI-gated.)*
   The two `--version` runs (`gen.sh:729-730`) fold into this static block or are dropped (they exist only to
   support the now-obsolete "same version string" framing).

**Why this shape:** the exemption becomes *structural* — gen.sh never executes `mnemonic-depth2`, so its output no
longer depends on the author's machine, and the CI differ needs **zero skip/carve-out logic**: the static block
is literal heredoc text, byte-covered by the whole-file diff like any other prose (it *cannot* drift except by an
intentional edit). The ratified "show-only fallback" intent (never rebuild depth2 in CI) is satisfied while
keeping the appendix's demonstrative payload. Fallback alternative if R0 objects to shipping a frozen 0.55.3-era
transcript: drop item 3 and keep only the `:764-773`-style omission note — a content downgrade, R0's call
(open question Q3).

## 6. D5 — Gate A: `.github/workflows/examples.yml`

### 6.1 What the gate is

One job, modeled on `manual.yml` (binary build: `manual.yml:92-96`; release-attach: `manual.yml:147-168`) and
`bitcoind-differential.yml:60-100` (pinned Core install):

```
Triggers:
  push:  branches [master, main]
         paths: .examples-build/**, docs/Examples.pdf, scripts/install.sh,
                crates/**, Cargo.lock, .github/workflows/examples.yml
         tags:  examples-v*        # paths never filter tag pushes (manual.yml:3-6 comment)
  pull_request: same paths
Steps:
  1. checkout (fetch-depth: 0)
  2. apt: pandoc texlive-xetex texlive-latex-recommended texlive-latex-extra
          texlive-fonts-recommended texlive-fonts-extra fonts-dejavu jq
          (monofont "DejaVu Sans Mono", gen.sh:94; jq explicit though runner-preinstalled)
  3. pinned Bitcoin Core v27.0: cache + sha256-verified tarball, per
     bitcoind-differential.yml:60-100; prepend its bin/ to PATH   (D1.4)
  4. cargo build --bin mnemonic                                    (debug; manual.yml precedent)
  5. regen:  EXAMPLES_BIN_DIR="$GITHUB_WORKSPACE/target/debug" \
             bash .examples-build/gen.sh > .examples-build/Examples.md
     (explicit `|| exit 1`: a gen.sh FATAL — pin mismatch, missing preflight dep — fails the job)
  6. GATE:   git diff --exit-code -- .examples-build/Examples.md
     (fail-closed; the drift shows verbatim in the CI log)
  7. build PDF: the exact pandoc line from gen.sh:9-11, cwd .examples-build; verify file exists
  8. upload workflow artifacts: Examples.pdf + Examples.md
  9. on examples-v* tag: ensure release exists + `gh release upload … Examples.pdf --clobber`
     (verbatim manual.yml:147-168 pattern)
```

**Trigger scoping rationale — deliberately wider than the sibling books.** `manual.yml`/`quickstart.yml` scope to
`docs/<book>/**`, which makes their verify gates *lagging* for CLI-behavior changes (they only re-run when the
book itself is touched). Including `crates/**` + `Cargo.lock` + `scripts/install.sh` here makes gate A a
**leading** indicator: a PR that changes any captured output (or bumps the version, or advances the install.sh
self-pin) turns red *in that PR*, forcing the lockstep regen. Cost: one debug cargo build + gen.sh (~seconds of
CLI runs) + one pandoc render on crates-touching PRs — small next to the existing `rust.yml` build the same PR
already pays. (The sibling books' narrow scoping is out of scope here; noted as a possible follow-up.)

## 7. Honesty — what gate A catches, and what it does not

- **Catches (the FOLLOWUP's rot class):** any change in captured CLI output vs the golden — display-format
  changes (the twice→once class), new/removed subcommands in `--help`, install.sh pin advances, version bumps
  (via the `gen.sh:22` FATAL), refusal-message drift, and any hand-edit to the golden that regen doesn't
  reproduce. The three-era patchwork we found **could not have survived this gate**.
- **Does NOT catch:**
  - **(a) The narration blind spot.** The gate diffs regenerated-output vs golden — and gen.sh regenerates prose
    *heredocs* verbatim, so prose is only gated for "golden matches gen.sh", not for "prose tells the truth about
    the adjacent output block". A future edit that changes an output block (with a faithful regen) but forgets
    the coupled narration — or edits narration to say something false about unchanged output — passes gate A as
    long as gen.sh and golden agree. Exactly this class produced `gen.sh:209` ("printed twice"). Gate A shrinks
    the window (output can't drift silently; prose drift now requires an *active* false edit passing review
    rather than passive rot) but does not close it. The full answer is the deferred **gate B**: per-command
