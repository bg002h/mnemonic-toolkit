### MECHANISM — close the paste-drift hole (two DISTINCT mechanisms — do not conflate)

`verify-examples` gates the `.out` on disk; the chapter fence is a hand-paste with no sync. The fix has two parts with DIFFERENT jobs:

**M1. Build-time include filter — eliminates prose↔golden divergence; it is NOT error-detection.** Add a pandoc Lua filter replacing a sentinel fence ` ```{.text include="transcripts/<stem>.out"} ` with the file body at build time. This MASKS drift by construction (rendered prose always equals the current `.out`); it surfaces nothing on its own. "Paste-drift becomes structurally impossible" holds ONLY for whole-body includes. **Excerpted/annotated fences require either (a) a line-range / section include or (b) a prose-is-substring-of-golden subset gate. DEFAULT to (b) [FOLD: Minor #7]:** mechanism (a) line-range include is NOT a native pandoc capability — pandoc's fenced-code include (include-code-files / include-files) is whole-file only; (a) needs a bespoke Lua filter parameterizing start/end lines or named-region markers in the `.out`, which materially expands M1 scope. Use (b) substring/subset-gate as the DEFAULT for excerpted fences; treat (a) as opt-in only where a contiguous clean line-range exists. **Canonical excerpt case: `22-first-bundle.md`** (verified live — its prose fence drops the merged `warning: secret material on argv …` line that `.out` line 1 carries; pair-mode `2>&1` merges that stderr warning into the `.out`). A per-block triage (P0) classifies each output fence whole-include vs excerpt; an excerpt is NOT counted "converted/gated" until its mechanism (a or b) exists — else it lands in the false-completion allow-list the L4 case study warns against.

**M2. verify-examples runner unification — the SOLE drift/regression detector [FOLD: Critical-tier #1 — runner fork].** FOUR runners, not three (Inventory §J, all live-verified):
- `manual` = CANONICAL (recursive `find` + per-cmd `mktemp -d` + triple `.err` + MK_BIN).
- `quickstart` already symlinks manual's (`tests/verify-examples.sh → ../../manual/tests/verify-examples.sh`, confirmed).
- `technical-manual` = non-recursive, no-tmpdir, pair-only (HAS MK_BIN) → upgrade to canonical before it gains subdirs or `.err`.
- **manual-gui = a SEPARATE, MK-BLIND fork** (`docs/manual-gui/tests/verify-examples.sh` — case block lines 18-25 has NO `MK_BIN=*` arm, `grep MK_BIN`→0 hits; `Makefile:230-235` passes no MK_BIN). Any manual-gui transcript invoking `mk` (mk1 = xpub card; 23 binary-derived files) would pass `$MK_BIN` through literally and FAIL/mis-substitute. **P4 MUST: (1)** replace it with a symlink to the canonical runner (quickstart's pattern), **(2)** INSERT `MK_BIN="$(MK_BIN)"` into `docs/manual-gui/Makefile:230-235` after the `MS_BIN` line, before `TRANSCRIPTS=` (re-grep live line numbers at write time — they shift on insert), **(3)** add `MK_BIN=` to `manual-gui.yml`'s `make` invocation, **(4)** verify no manual-gui `.cmd` relies on old pair-only / no-tmpdir semantics.

**Error-detection is M2-only.** A book with includes but no CI replay against REAL pinned bins gets ZERO error-detection. Today quickstart/tech-manual/manual-gui CI run `make lint` with `*_BIN=true`/`MK_BIN=mk` stubs (verified `quickstart.yml:77`, `technical-manual.yml:83-86`, `manual-gui.yml:106`) → until P1/P2/P4 add real bins AND switch CI to `verify-examples`, those books get zero error-detection even with includes. **Every book with includes MUST also have a real-pinned-bin `verify-examples` step in CI.**
